# Glossary: Project-Coined Terms (part 1)

*Last updated: 2026-07-07. Part of the [Manatee glossary](../glossary.md) — see the index there for all terms and for how entries are structured.*

Manatee-specific vocabulary, each mapped to the closest standard concept in the literature.

### 0B (zero-bytes / zero-allocation)
Project shorthand marking an API call that allocates zero bytes of heap memory after warmup — it runs entirely in pre-allocated storage.

C# is a garbage-collected language: objects created "on the heap" are reclaimed later by a collector that can pause the program at an unpredictable moment. In a game ticking dozens of times per second — and especially on Re-Volt's Stationeers worker thread — those pauses read as stutter, so requirement R8 demands that steady-state operation (all tier-0 through tier-2 calls: driving sources, adjusting values, reading results) allocate nothing at all once initial buffers exist. `docs/api.md` annotates each such member with `0B`, alongside sibling markers `shape` (allocates, but only at structural-commit time, never in a tick) and `cold` (diagnostic path, may allocate). The guarantee is enforced mechanically: BenchmarkDotNet's MemoryDiagnoser gates it in CI with no exemptions, and a runtime `SteadyStateGuard`/AllocationSentinel tripwire (api.md §8) can check it in situ. The EE analogy: a circuit designed to draw zero quiescent current — nothing is consumed except when real work (a structural edit) happens, and the datasheet marks which pins that promise applies to.

**Project-coined notation**, closest standard concepts: allocation-free code / "no GC pressure" / zero-allocation hot path.

**Where it appears:** `docs/api.md` (notation section, per-member `0B` annotations, §8 SteadyStateGuard, §21 enforcement layering); `docs/solver.md` Threading and Allocation Contract; `docs/design.md` R8; `docs/testing-strategy.md` zero-alloc gate and flake-isolation notes.

### 12 V cottage problem
The project's name for the pain a player personally experiences when running a 12 V supply across cottage-scale distances: the wires eat the voltage before it reaches the load.

In Vintage Story, 12 V DC is the deliberate starter tier — easy and safe — but real resistive line loss is simulated, and at 12 V even modest cable runs drop enough voltage that lamps dim and devices fall below their nameplate tolerance. The lost fraction scales inversely with the square of the supply voltage (the same power delivered at higher voltage means less current, and loss goes as current squared times wire resistance), so the frustration is the setup for curriculum Lesson 14, which teaches why real grids transmit at high voltage and step down with transformers — and includes the ground-return (SWER) coda showing single-wire return works at 240 V but fails at 12 V. In systems terms it is an intentionally under-provisioned default configuration: the player hits the bottleneck first, then is handed the scaling lesson that explains the fix.

**Project-coined term**, closest standard concept: resistive line loss / IR voltage drop in low-voltage distribution.

**Where it appears:** `docs/curriculum.md` Lesson Arc III, Lesson 14; `docs/design.md` Voltage and frequency standards (12 V tier rationale) and the Grounding model.

### 2D harness / 2D schematic harness
The standalone 2D schematic editor/simulator built on manatee-core, serving three roles at once: the developer tool and primary integration test bed for the core, the runtime engine inside the Vintage Story "gaming tablet," and the lesson-authoring tool.

It is a complete circuit-drawing client — place components on a grid, wire them, watch live simulation — that runs with no game engine in the loop, so it exercises the full API surface (netlist construction, devices, the reduction layer, save/load persistence, probes) exactly as the game mods will, but in an environment where everything is scriptable and headless. Scripted harness scenarios (build, edit, save/load, fault) run in CI. One codebase (`Manatee.Schematic`) fills all three roles by being built as three *pure* layers — a document model (the schematic as data), an interaction state machine (editing behavior driven by abstract input events), and a view layer that emits a display list (a flat sequence of draw commands as data) — see `docs/harness.md`. Because none of the layers touch a UI framework, the whole thing can be tested headlessly (synthetic pointer events in, document mutations and draw commands out) with no display attached; a thin Avalonia desktop shell wraps it for interactive use. For an EE the closest analogy is a bench test fixture: a jig with sockets, supplies, and probes that exercises the device under test through its real connectors before it is installed in the end product. It is delivery-order item 1 together with manatee-core itself, and doubles as the lesson-authoring tool for the curriculum corpus.

**Project-coined term**, closest standard concepts: test harness (software-testing scaffolding that runs code under test) combined with a schematic capture editor (EDA); "2D" distinguishes this schematic-plane client from the 3D in-game integrations.

**Where it appears:** `docs/harness.md` (the whole doc; architecture); `docs/testing-strategy.md` Game-Layer Testing; `docs/design.md` (The Three Clients, Repository Layout, Delivery Order — it is delivery step 1 alongside manatee-core); code under `tablet/` (`Manatee.Schematic` plus the Avalonia dev harness).

**References:** the layering echoes Model–View separation as in Gamma, Helm, Johnson & Vlissides, *Design Patterns* (1994); "test harness" is standard vocabulary, e.g. Meszaros, *xUnit Test Patterns* (2007).

### 2f-aliasing guard
The nickname for the **coupling dissipativity** invariant test, which guards against the 2f-aliasing / phantom-torque bug in the mechanical↔electrical coupling.

The test: build a closed loop — motor → shaft → alternator → back to the motor — and assert that from *any* initial condition its stored energy monotonically decays. Every element in that loop is lossy, so a correctly coupled system can only run down, like a flywheel spinning against friction. The value of the construction is fault isolation: both the mechanical and electrical integrators are individually dissipative, so if the loop ever gains energy (or fails to decay), the only possible culprit is the coupling between them — specifically the counter-torque averaging window. This is the invariant-testing style used throughout the project: rather than checking specific numbers against an oracle, assert a physical property (here, passivity) that no correct implementation can violate, turning an entire class of bugs into a single always-on tripwire.

**Project-coined term**, closest standard concepts: a *dissipativity/passivity* check on a co-simulation coupling, implemented as a property-based invariant test.

**Where it appears:** `docs/testing-strategy.md` (Invariant Checks, "Coupling dissipativity"); the mechanism it guards is in `docs/vintage-story.md` (Mechanical Network Coupling).

### 5 Hz flicker
The visible pulsing of lamp brightness when lamps run on the ~5 Hz AC that waterwheel and windmill alternators naturally produce — a deliberate, load-bearing teaching phenomenon, not a bug.

At mains frequency (50/60 Hz) a lamp's brightness variation is too fast for the eye; at 5 Hz it is plainly visible (the luminance actually pulses at twice the electrical frequency — see 2f-aliasing — so ~10 Hz). The game keeps it because it teaches, in order: why the real world converged on 50 Hz, that the easy fix is DC lighting, and that the ambitious fix is raising frequency via high-pole-count alternators (shaft speed is capped by mechanical overheat, so pole count is the honest knob) — which in turn pays off in dramatically smaller transformers. Lesson III.11, the oscilloscope lesson, is where the flicker gets explained. One caveat is normative: 3–30 Hz rhythmic flashing is the photosensitive-epilepsy trigger band, so the client forces a first-boot choice between faithful lamp rendering and a low-passed (smoothed) rendering; the choice affects display only — the simulation and the oscilloscope waveform are identical either way, the way a filter on a chart changes the plot but not the data.

**Where it appears:** `docs/curriculum.md` (Lesson Arc III, lesson 11); `docs/design.md` (Frequency, incl. the flicker-accessibility decision; also the oscilloscope justification).

### Accept-list
- The enumerated, closed set of Falstad element types and parameters that Manatee's importer fully honors electrically; everything outside it is either logged-and-ignored or rejected per the accept / reject contract.

The list (falstad-format.md §7) names each accepted dump type and, per type, exactly which parameters carry meaning — e.g. wires, ground, R/C/L (with initial conditions), DC and sine sources only (waveforms 2–7 rejected), switches, diodes/zeners with a few named models, transformers, probes, and comment/annotation lines. It is closed in a strong sense: even on an accepted element line, trailing tokens the importer does not recognize cause a loud rejection, whereas upstream circuitjs silently ignores extras — that strictness exists because silently-tolerated extra tokens are how file-format dialects drift apart. Programmers will recognize this as an allowlist (default-deny); for an EE, it is like an approved-parts list on a certified design — a component not on the list doesn't get soldered in no matter how plausible it looks. Manatee's own lesson files are additionally restricted to this subset so they load unmodified in falstad.com for authoring.

**Standard term:** allowlist / whitelist (default-deny input policy); here specialized to the importer's supported-element subset.
- **Where it appears:** `docs/falstad-format.md` §7 (the **Accept** bucket and the corpus rule); referenced by `docs/curriculum.md` authoring rules and `docs/testing-strategy.md` (lesson corpus).

### Accept / reject contract
- The Manatee Falstad importer's error posture: every line type in an imported circuit file falls into exactly one of three explicit buckets, and nothing is ever silently dropped.

The three buckets partition all Falstad/circuitjs dump types: **accept** — elements whose electrical semantics we fully honor (the accept-list); **ignore with a logged notice** — presentation-only lines with no electrical meaning (scope layouts, hints, sliders), which are skipped but counted and reported so a lesson author sees what was dropped; and **reject loudly** — everything else (unsupported devices, malformed numbers, NaN/Inf, non-positive R/C/L), failing with the line number, the offending token, and a one-line reason. This is deliberately stricter than upstream Falstad and Electrical Age parsers, which silently skip or guess. For a programmer this is just input validation with a total, exhaustive policy; for an EE, think of it as incoming inspection where every part on the shipment manifest is either used, logged as cosmetic packaging, or bounced back with a defect report — never quietly binned. The motivation is that the lesson corpus is CI ground truth (design.md R16/R20), so silent data loss would corrupt the tests themselves.

**Project-coined term**, closest standard concepts: input-validation policy / parser error contract; the strict-rejection stance is the inverse of Postel's robustness principle, adopted deliberately to prevent dialect drift.
- **Where it appears:** `docs/falstad-format.md` §7 ("Recommendation: Manatee importer subset and error posture").

### Across-tick clamp
- Enforcing a device limit (e.g. a legacy generator's advertised-power current limit) by adjusting the device's stamped values *between* game ticks, using last tick's solved results — never by switching the device's mode in the middle of a single solve.

A hard limit like "this generator supplies at most X amps" is a nonlinearity: strictly modeled, the solver would have to detect the violation mid-solve, change the device's equations, and re-solve, possibly iterating. Manatee's legacy-device adaptor instead reads the previous tick's solution, clamps the offending quantity, restamps the device accordingly, and lets the next tick's linear solve run uninterrupted — the limit converges over a few ticks rather than within one. Programmers can read it as an eventually-consistent feedback loop: correction is applied one iteration late, trading instant exactness for a solver that stays linear and predictable per tick. EEs will recognize the underlying idea from fixed-timestep companion modeling: the nonlinearity is linearized about the previous timestep's operating point, one Newton-like update per game tick, with a tick of delay in limit enforcement as the accepted cost (fine at game timescales). The same across-tick pattern is used for the adaptor's constant-power loads (G = P/V_prev²).

**Project-coined term**, closest standard concepts: source limiting via companion-model linearization about the previous timestep; contrast with in-solve mode switching / iterated Newton–Raphson limiting as done inside SPICE's per-timestep NR loop.
- **Where it appears:** `docs/stationeers.md` — The Legacy-Device Adaptor (settled 2026-07-05); background in `docs/solver.md` failure handling.
- **References:** Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520 (companion models and Newton iteration); Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods* (linearized companion models).

### actuals
- The power/current each adapted Stationeers device *actually received or supplied* according to the solve, as opposed to what it asked for or advertised — written back into the vanilla game's own accounting after every solve.

The vanilla Stationeers power API is a four-call contract (available/needed, provide/draw, in joules per half-second tick). Pre-solve, the adaptor reads devices' *wants* (needed power, advertised available power) and stamps the circuit accordingly; post-solve, it distributes the actuals by calling the same vanilla provide/draw methods with what the MNA solution really delivered — which under brownout or overload can be less than requested. This keeps vanilla bookkeeping (battery charge, device on/off state) consistent at all times, which is what makes the mod cleanly removable: pull Manatee/Re-Volt out and vanilla mechanics resume from correct state. For programmers, it is a write-back cache-coherence step: the solver is the source of truth for one tick, and its results are flushed into the legacy data structures so the two systems never diverge. For an EE, it is the difference between a load's nameplate rating and its measured consumption at the actual bus voltage — the game's ledgers are settled with the measurement, not the nameplate.
- Ordinary English used with a specific project meaning (wanted/advertised vs. delivered); no standard EE or CS term is being replaced.
- **Where it appears:** `docs/stationeers.md` — Integration Seams (`ApplyState_New` distribute phase) and The Legacy-Device Adaptor.

### AdaptedLoad
- The stock built-in device model in manatee-core representing an unconverted Stationeers appliance: a constant-power load (design requirement R18) with brownout hysteresis, staggered rejoin, recloser lockout, and an anti-free-energy ledger.

`AdaptedLoad` lives in `Manatee.Core.Devices` and is constructed with its electrical and protection parameters (advertised watts, conductance clamps `gMin`/`gMax`, brownout low/high voltage thresholds, lockout count, rejoin stagger timing), then enlisted with `DeviceHost.Add(...)` alongside its terminal nodes and identity keys. `DeviceHost` ticks it each game tick; internally it re-linearizes its power draw against the previous tick's voltage and runs its brownout/rejoin/lockout state machine. One sharp edge, documented in the tutorial: built-in devices cache their component and terminal-node handles privately and expose no re-pin surface, so `AdaptedLoad` is only safe on **static islands** — networks whose topology never rebuilds. Dynamic-topology integrations wrap the same behavior in a key-re-resolving adaptor instead. For an EE: it is a self-protecting constant-power load with undervoltage dropout and a recloser, packaged as one library part; the "handles" caveat is a software wiring concern — after the network is rebuilt, the part's saved terminal references point at the old build.
- Project-coined class name; the electrical model it implements is a constant-power (constant-P) load with undervoltage load-shedding. See the companion entry *AdaptedLoad / constant-power (G=P/V²)* for the model itself.
- **Where it appears:** `docs/integration-tutorial.md` §4 (worked construction example), `docs/api.md` §18 (built-in models and the `DeviceHost` static-island restriction), `docs/design.md` R18.

### AdaptedLoad / constant-power (G=P/V²)
- The mathematical model behind `AdaptedLoad`: an unconverted Stationeers device is treated as a constant-power load, made solvable each tick by replacing it with a plain resistor whose conductance is G = P\_wanted / V\_prev² (clamped to [gMin, gMax]).

A constant-power element (I = P/V) is nonlinear — its current depends on the very voltage being solved for. Rather than iterating inside a tick, the adaptor linearizes **across game ticks**: it uses last tick's solved voltage V\_prev to pick this tick's conductance, which converges to the true operating point over a few ticks in normal operation. For programmers, this is a fixed-point iteration spread over the tick loop, with the solver seeing only a linear circuit each step; once converged, the ε-no-op `Adjust` makes the per-tick update free. The lag has two failure modes, each with a designed counter-measure: (1) voltage collapse — below a threshold the device **browns out** (drops to near-zero load) with hysteresis (drop at V\_low, rejoin at V\_high), per-device staggered rejoin delays, and recloser-style lockout after repeated brownouts in a short window (manual reset), so an overloaded net settles or collapses legibly instead of strobing at tick rate; and (2) the energy pump — a load toggled at tick rate could exploit the one-tick lag to extract unearned joules, so each adapted device carries an across-tick **energy-debt ledger** (deliverable this tick = advertised − accumulated debt), whose settlement is exact under the fixed timestep the codebase drives. Note the deliberate difference from SPICE practice: SPICE linearizes nonlinear elements *within* a timestep via Newton iterations; our adaptor linearizes *between* timesteps and accepts the lag, buying it back with the ledger and hysteresis.
- **Standard term:** constant-power (constant-P) load, per-step secant/companion-model linearization; the protection behaviors are standard power-engineering concepts (undervoltage load shedding with hysteresis, recloser lockout).
- **Where it appears:** `docs/api.md` §18 (built-in models), `docs/design.md` R18, `docs/stationeers.md` (The Legacy-Device Adaptor), `docs/experiments/2026-07-05-adversarial-playtest.md` (the energy-pump finding that motivated the ledger).
- **References:** Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520 (companion-model linearization of nonlinear elements); Vlach & Singhal, *Computer Methods for Circuit Analysis and Design* (linearization and iteration schemes); constant-power "PQ" load models are standard in power-systems load-flow literature.

### AdaptedSource
- A built-in device model representing a power *supply* whose game-side spec is "advertised watts": a voltage source with an internal series resistance, plus a current clamp that keeps it from delivering more than its advertised power.

It is the supply-side sibling of `AdaptedLoad`: where the adaptor turns an unconverted vanilla *consumer* into a circuit element, `AdaptedSource` does the same for a vanilla *generator*. Electrically it is an EMF behind an internal resistance (a Thévenin source); the twist is the power clamp, which is enforced *across ticks* rather than inside the solve — each tick it reads the load resistance implied by the previous solution and sets its limit so it targets the exact EMF, converging in one step when the load is resistive. For programmers: it is a feedback controller whose only memory is the last frame's output, so it needs no persistent state of its own. Indeed it is declared **stateless** (`StateSize = 0`, registers no state unit), which means snapshot/restore has nothing to save for it — unlike `AdaptedLoad`, which carries a brownout latch and an energy-debt ledger.

**Project-coined term**, closest standard concept: a Thévenin-equivalent source (ideal voltage source + series internal resistance) with a constant-power output limit, linearized across timesteps rather than iterated within one.
- **Where it appears:** `docs/api.md` §18 (built-in models, `Manatee.Core.Devices`); listed alongside `AdaptedLoad`, `Battery`, `Alternator`.
- **References:** Thévenin sources and internal resistance: Horowitz & Hill, *The Art of Electronics*, ch. 1.

### adaptor
- The Stationeers-side shim that makes *unconverted vanilla devices* — ones still speaking the game's simple watts-in/watts-out power API — behave as real circuit elements inside the MNA solver.

Vanilla Stationeers devices only say "I want P watts"; a real solver needs a conductance. The adaptor (requirement R18) bridges the two by treating each such device as a **constant-power load linearized across ticks**: it computes G = P/V_prev² from last tick's voltage and stamps that as a plain resistor this tick, with the value clamped to the legal conductance range. Because that linearization lags reality by one tick, two protections ride along: **brownout hysteresis** (drop out at V_low, rejoin at V_high, with staggered rejoin delays and a recloser-style lockout after repeated brownouts) so an overloaded network settles or collapses legibly instead of strobing at tick rate, and an across-tick **energy ledger** (deliverable this tick = advertised − accumulated debt) so an oscillating load cannot pump the one-tick lag for free energy. For programmers: it is a compatibility facade over a legacy API, with a rate limiter and an accounting invariant. For EEs: the linearization is the standard fixed-point treatment of a constant-power (P–V) load, but iterated across game ticks instead of within one Newton solve. Native device conversions replace the adaptor per-device over time; vanilla state stays maintained so removing the mod reverts cleanly.

**Project-coined term** (in the software sense of the Adapter pattern), closest standard EE concept: a constant-power load model under fixed-point/successive-approximation linearization, plus undervoltage load-shedding with hysteresis (recloser behavior).
- **Where it appears:** `docs/design.md` R18; `docs/stationeers.md` (legacy-device adaptor); `docs/testing-strategy.md` (Property and Fuzz Tests); the built-in `AdaptedLoad`/`AdaptedSource` models in `docs/api.md` §18.
- **References:** Adapter pattern: Gamma, Helm, Johnson & Vlissides, *Design Patterns* (1994). Constant-power loads and negative incremental resistance: Horowitz & Hill, *The Art of Electronics*.

### adaptor-stability property test
- A randomized CI test asserting that the tick-lagged feedback loops in the system — adapted constant-power loads, and later all island-coupling boundaries — always settle rather than oscillate forever.

Everything the adaptor and the island couplers do is a feedback loop with one step of delay: this tick's stamped value is computed from last tick's solution. Delayed feedback can ring, so stability is *enforced by test*, not assumed. The original form throws randomized sets of constant-power loads at a falling supply and requires that the network either settles or browns out within k ticks — never oscillating unboundedly, never NaN. Per `docs/solver.md` (Islands), the same harness was extended to **boundary couplings** (decoupling transformers, DC converter two-ports, which exchange amplitude+phase per substep with explicit relaxation): randomized load steps across a boundary must settle, never ring unboundedly. For programmers: a property-based test (seeded random inputs, universal assertion) over a distributed system's convergence. For EEs: an automated check that the discretized loop is damped across a randomized envelope of operating points.

**Project-coined name** for a standard technique: property-based (generative) testing applied to numerical stability of a lagged fixed-point exchange.
- **Where it appears:** `docs/testing-strategy.md` (Property and Fuzz Tests: "Adaptor stability (Stationeers)"); `docs/solver.md` Islands ("the adaptor-stability property test extends to boundary couplings").
- **References:** Property-based testing originates with Claessen & Hughes, "QuickCheck: A Lightweight Tool for Random Testing of Haskell Programs" (ICFP 2000).

### additive restore / RestoreResult
- The rule (binding, `docs/api.md` §14) that loading saved state into an island only overwrites the state units the save blob actually contains, and never resets anything else; `RestoreResult` is the struct reporting what happened.

Every piece of evolving state (a capacitor's voltage, a fuse's partial melt, a battery's charge) is a *state unit* addressed by a client-stable `StateKey`. `Restore(blob)` matches blob entries to live units by that key: matches are overwritten, everything else is **left untouched** — units are born at cold defaults when built, and a Restore call never actively cold-starts anything. This additivity is what makes restore compose when topology drifted between save and load, since keys are netlist-global but blobs are per-island: if two saved islands **merged**, offer each blob in turn and each restores its own keys without clobbering the other's; if one saved island **split**, offer the same blob to every resulting island and each takes its matching keys. `RestoreResult` reports `Matched`, `Untouched`, and `OrphansInBlob` (blob entries with no live unit — expected in the split case, a world-edited-between-save-and-load signal otherwise; drain the keys via `DrainOrphans`). Crucially, *coverage* — "did any unit truly cold-start?" — is checked **in aggregate** after all blobs are offered (sum of `Matched` vs the island's `StateUnitCount`), never per call. Decision log §24.19 records why: the previous cold-start-the-unmatched semantics made multi-blob restore into a merged island last-blob-wins (silent state loss) and gave the split case no path at all. For EEs: think of it as a patch applied by part number, not a wholesale replacement of the assembly.

**Project-coined term**, closest standard concepts: an upsert/patch (merge) semantics for deserialization, as opposed to replace semantics; the whole-netlist `SaveCanonical` is the Memento-pattern counterpart.
- **Where it appears:** `docs/api.md` §14 (snapshot/restore) and §24.19 (decision log); API surface `IslandHandle.Restore(ReadOnlySpan<byte>) → RestoreResult`, `IslandHandle.StateUnitCount`, `RestoreResult.DrainOrphans` (an `IslandHandle` is obtained via `net.Islands.Of(n)`).

### AddSegment / RemoveSegment
- The two reduction-layer operations for placing and deleting one cable segment: the vocabulary a game integration uses to describe wiring, instead of touching the solver's netlist directly.

`ConductorGraph.AddSegment(segmentKey, junctionA, junctionB, spec, partition)` declares one piece of cable between two junctions, with a `ConductorSpec` (resistance, plus optional current/thermal limits — a 6 A fuse is just a segment with a `LimitSpec`), under client-chosen 128-bit keys that survive everything the layer does internally. The reduction layer then *compacts*: long series chains of segments collapse into a few equivalent resistors before the solver ever sees them, which is how ~10k raw Stationeers cables become a graph in the low hundreds. Outside a `BeginBulkBuild` scope each call recompacts eagerly and diffs against the realized netlist so unchanged nodes emit no edits (handles survive); inside a bulk scope segments stage and one compaction pass runs at Dispose. Under `ClientPartitioned` mode the `partition` argument is mandatory (the default value throws). `RemoveSegment(segmentKey)` deletes a segment and dirties its island for a rebuild, coalesced to the next `Solve` — so N cuts in one tick still cost one rebuild. **Melting a fuse is a `RemoveSegment`**: the solver never mutates the circuit on a limit (R7), it reports a limit event, `Attribute` maps it back to the culprit segment, and the client removes it. For EEs: these are schematic-capture edit operations whose netlist extraction (with series reduction) is automatic and incremental.

**Project API names**; closest standard concepts: incremental netlist editing plus series-equivalent circuit reduction.
- **Where it appears:** `docs/integration-tutorial.md` §3 (intake) and §8 (fuse melt); `docs/api.md` §19 (`ConductorGraph` intake contract); `docs/compaction.md` (the reduction rules underneath).
- **References:** Series/parallel network reduction is textbook material, e.g. Vlach & Singhal, *Computer Methods for Circuit Analysis and Design*.

### Adjust
- One of Manatee's four mutation verbs: change an existing component's electrical value (ohms, farads, henries, switch open/closed, diode parameters, transformer turns ratio) without changing the circuit's wiring.

`Adjust` is Tier 2 in the change-cost model: because only *values* change, not which components connect where, the matrix's sparsity pattern is unchanged — the solver reuses its cached symbolic factorization and pays only a numeric refactorization (recomputing the numbers in an already-planned decomposition). For programmers: it is an in-place field update that invalidates a computed cache but not the schema; contrast `Edit` (Tier 3, topology change, replan everything) and `Drive` (Tier 1, source values only, no refactor at all). One important wrinkle: an `Adjust` whose new value is within `AdjustEpsilon` of the old (a ratio-based test, default 1e-4) is an ε-no-op — it stays on the cached LU factorization entirely, so converged feedback loops (like a device adaptor recomputing G = P/V² each tick) stop paying Tier 2 without the client having to check. `CostOfAdjust` answers "what would this cost" without doing it; `TickStats.AdjustNoOps` counts the skips.

**Project-coined term**, closest standard concept: element value update / parameter modification with reuse of the symbolic factorization (only numeric refactorization required).
- **Where it appears:** `docs/api.md` §4 (verbs and cost tiers), §9 (AdjustEpsilon pinned definition), §18 (device layer); `docs/integration-tutorial.md` §1, §4.
- **References:** Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods* — symbolic vs. numeric phases of sparse LU factorization; Vlach & Singhal, *Computer Methods for Circuit Analysis and Design*.

### AdjustNoOps
- A per-tick counter in `TickStats` recording how many `Adjust` calls fell inside the ε-gate and cost nothing.

Each solver tick publishes cheap diagnostic counters (`TickStats`); `AdjustNoOps` counts the `Adjust` calls that landed within `AdjustEpsilon` of the already-stamped value and therefore degraded to free tier-0 writes on the cached factorization. Its practical meaning is "your converged devices, made visible": in steady state a healthy integration shows `AdjustNoOps` roughly equal to its adaptive-device count while `Refactorizations` stays at zero. For an EE, it is the instrumentation equivalent of watching how many feedback loops have settled onto their operating point; for a programmer, it is a cache-hit counter for conductance updates. If `Refactorizations > 0` on a quiet tick, some device's math is not converging — this counter tells you which side of the ε boundary the fleet is on.

**Where it appears:** `docs/api.md` §9 (`TickStats.AdjustNoOps`), `docs/integration-tutorial.md` §4 and §9 (the performance contract).

### afilter leftovers
- Three line types (`%`, `?`, `B`) in Falstad circuit files that are dead remnants of an old audio-filter feature; our importer skips them but logs and counts what it dropped.

The Falstad/circuitjs1 text format accumulated line types over decades, and `%`, `?`, and `B` survive from a since-removed audio-filter mode — upstream's own loader ignores them before element dispatch, and one options-header flag bit (32, "afilter linear scale") is likewise vestigial. They carry no electrical meaning, so Manatee's importer classifies them as "ignore with a logged notice": accepted silently enough that real-world exports (which contain them constantly) still import, but counted and reported so a lesson author sees exactly what was dropped. Think of them as deprecated fields in a wire format that the parser must tolerate but should never silently swallow — deliberately stricter than upstream, whose error posture is silently lossy.

**Where it appears:** `docs/falstad-format.md` §1 (line-type dispatch table), §2 (options-header flags), and §7 (the importer's accept/ignore/reject contract).

**References:** the circuitjs1 source (`CircuitLoader.java`), which the format spec cites line-by-line.

### AllocationSentinel / tripwire
A debug-mode guard object that snapshots the thread's allocated-bytes counter when armed and asserts the count has not moved when it is released.

Concretely it is a C# `ref struct` — a type the compiler confines to the stack, so it can never itself be heap-allocated or smuggled into a background task — wrapped in a `using` block: `Arm()` captures `GC.GetAllocatedBytesForCurrentThread()`, and `Dispose()` (the automatic end-of-scope call) asserts the delta is zero. For an EE: it is a test clamp placed around one section of the circuit for the duration of a measurement, removed automatically when the measurement ends. It is armed automatically inside `SteadyStateGuard`, or standalone. It is explicitly best-effort: at first arm it runs a capability probe (allocate a small object, check the counter moved); on runtimes where the counter is inert (Unity/Mono) it disarms process-wide, sets `IsReliable == false`, and logs once. The binding zero-allocation enforcement is the CI MemoryDiagnoser gate, not this sentinel.

**Project-coined term**, closest standard concept: an allocation-assertion scope guard (RAII-style guard object, cf. the Dispose pattern), akin to `Assert.That(..., Is.Not.AllocatingGCMemory())` in Unity's test framework.

**Where it appears:** `docs/api.md` §8 (`AllocationSentinel`, `SteadyStateGuard`) and the caveats list §24.10; `docs/integration-tutorial.md` sharp edge 14.

### Ambient-temperature-adjusted
- Describes limit thresholds (how much current a cable or fuse tolerates) that are shifted by the local environment's temperature, rather than being fixed constants.

In Stationeers the environment spans −150 °C on Europa to +800 °C on Vulcan, and Manatee makes those temperatures move the *real* thresholds in the compaction layer's limit envelope: a cable near melting point ambient has almost no thermal headroom, while a cryogenic one can carry more current before overheating. For programmers: the adjustment is metadata-only — an environment change dirties the envelope and triggers a recompute of threshold numbers, never a change to the circuit matrix, so it is guaranteed cheap (a Tier-0 operation in the change-cost tiers). For EEs: this is exactly the ambient-correction (derating) factor applied to conductor ampacity tables in practice — e.g. the correction factors attached to NEC ampacity tables — implemented by mechanism instead of per-planet special cases.

**Standard term:** temperature derating / ambient correction of ampacity ratings.

**Where it appears:** `docs/stationeers.md` Graph Construction; `docs/compaction.md` Responsibilities #4 (limit attribution); `docs/api.md` limit-envelope sections.

### `+` as delimiter
A quirk of the Falstad text format: the `+` character acts as a token separator (like a space), so it can never appear inside a number.

The upstream circuitjs loader tokenizes each line with Java's `StringTokenizer` using the delimiter set `" +\t\n\r\f"` (`CircuitLoader.java:142`). This is a fossil of circuits being pasted out of URLs, where a space is encoded as `+`; treating `+` as whitespace made pasted URLs "just work." The consequence is a parsing landmine: a value written `1e+5` splits into two tokens (`1e` and `5`) and breaks the line, whereas `1e5` or `1E5` parses fine. circuitjs never trips over this because Java's `Double.toString` never emits a `+` in exponents, and Manatee's importer/exporter is bound by the same rule: never write `+` in an exponent. For an EE, the analogy is a connector pinout constraint inherited from an obsolete adapter — arbitrary, but violating it breaks interoperability, so it is honored forever.

**Standard term:** none — this is a format quirk (whitespace/delimiter handling), documented here so nobody "fixes" a lesson file into unparseability.

**Where it appears:** `docs/falstad-format.md` §1 (tokenization), §5.4, §7 (writer rule: never emit `+` in exponents; parse with InvariantCulture).

### Attribution / Attribute
The act of mapping a limit event raised by the solver on a *collapsed equivalent* component back to the *source geometry* that is actually at fault — which voxel melts, which fuse pops — via the reduction layer's `Attribute` call.

The solver only knows about the reduced circuit: when current through an equivalent resistor exceeds a limit, it emits a `LimitEvent` naming that equivalent component, and per requirement R7 it **never mutates the circuit itself** — reacting is the client's job. The reduction layer's `ConductorGraph.Attribute(in LimitEvent e, out AttributionResult a)` answers the reverse-lookup: given the event, it returns the culprit `SegmentKey` (plus the limit kind and margin), using current density at the narrowest cross-section as the tie-breaking rule, evaluated per limit type; for i²t thermal events the event's `PairIndex` field says which (rating, melt) pair of the envelope tripped, which pins the segment. The call is zero-allocation ("0B") so it is safe on the hot per-tick path. The programming analogy: it is symbolication — the solver's event is a raw address in optimized (compacted) code, and `Attribute` is the source map that translates it back to the original file and line. The client then pops the fuse or starts the fire at that segment on the next tick.

**Project-coined term**, closest standard concept: back-annotation — mapping results computed on a reduced/transformed model back onto the original design description.

**Where it appears:** api.md §12 (`LimitEvent`, `PairIndex`) and §19 (`ConductorGraph.Attribute`, `AttributionResult`), design.md R7, compaction.md item 4, integration-tutorial.md (fuse-blowing loop).

### Attribution (weakest voxel)
The compaction layer's standing knowledge of *which physical piece* of a collapsed cable run is its limiting element — the specific voxel or segment that would melt, pop, or overheat first — used to drive tooltips and breaker-trip reports.

When many voxels or cable segments are collapsed into one equivalent resistor (see series-chain collapse), the individual pieces' ratings are folded into a per-limit-type **limit envelope**: the minimum rating over the constituents, tracked separately for instantaneous ampacity, i²t thermal mass, and melting threshold — each of which can point at a *different* segment in a mixed-material chain (this is exactly how a lead segment in a copper run behaves as a fuse with zero special-casing). Because envelopes are maintained continuously — including environment-adjusted thresholds when ambient temperature changes — the answer to "what's the weak point here?" is available *before* anything fails: R15's teaching tooltips lean on it ("14 A through an 8 A-rated section"), and breakers can report the culprit at the moment of trip. The programming analogy: it is a maintained index (like a min-heap keyed per limit type) over the collapsed chain, kept incrementally up to date rather than recomputed on demand.

**Project-coined term**, closest standard concepts: worst-case rating / derating analysis of a series path; the "weakest link" is the element with the smallest margin per failure mode.

**Where it appears:** design.md R7 and R15 (tooltips "attribution already knows the weakest voxel"), compaction.md item 4 (limit attribution / limit envelope), api.md §12 (thermal envelopes as Pareto sets).

### Automation off-ramp
- A project design rule: once the player has genuinely learned a manual skill, the game offers an optional craftable device that automates it — a convenience they can build, never a requirement.

The rule (the "mastery corollary", settled 2026-07-06 in design.md) exists because a mechanic that is a thrill the first five times becomes a chore forever after. Examples: the pre-wired elevator controller block (after mastering relay-logic elevator control), DC lighting as the escape from low-frequency AC flicker, and a sync-check breaker that refuses to close out-of-phase while the player still matches generator speed by ear. The off-ramp preserves the skill's teaching value — you must earn it first — while amortizing the tedium; the docs call this "the difference between a curriculum and a treadmill." For programmers, the shape is familiar from good tooling: you learn to do it by hand, then you script it. For an EE, it is the difference between hand-synchronizing generators with lamps and installing a synchro-check relay.

**Project-coined term**, closest standard concepts: progression/mastery design in game design; quality-of-life automation.

**Where it appears:** `docs/design.md` (Mastery corollary), the elevator-control section of design.md, and `docs/experiments/2026-07-05-adversarial-playtest.md` (where the principle was first identified).

### Averaging window
- The exact time interval over which the alternator's counter-torque is averaged before being reported to Vintage Story's mechanical (shaft) network: all electrical substeps since the previous mechanical read.

This is normative, not an implementation detail (settled 2026-07-06). Single-phase AC power pulsates at twice the electrical frequency — about 10 Hz at our 5 Hz — which lands right at the mechanical network's ~100 ms torque re-sum cadence. If the counter-torque were *sampled* at an instant instead of averaged over exactly that interval, the ripple would alias into a phase-dependent DC torque offset: phantom (free) torque, i.e. a perpetual-motion exploit for motor-spins-alternator builds. The programming framing: it is a classic sampling-rate-mismatch bug between two loosely coupled simulations, fixed by integrating over the exact inter-read interval rather than point-sampling; `GetResistance()` must be a pure read of that cached mean, valid at any time on the server main thread. The standing invariant test is that a closed motor→shaft→alternator→motor loop decays monotonically from any initial condition.

**Project-coined usage**, closest standard concepts: anti-aliasing / averaging interval; in co-simulation literature, a conservative coupling quantity averaged over the macro-step.

**Disambiguation:** despite the pairing in some project discussion, "merge window" in the docs names an *unrelated* interval — the ticks between an island **merge** commit and the merged island's first published solve, during which node reads on both sides return last-good pre-merge values (`docs/integration-tutorial.md`, api.md §17). Only the torque-averaging sense belongs to the mechanical coupling rule described here.

**Where it appears:** `docs/vintage-story.md` §2 (Mechanical Network Coupling — "The averaging window is normative"), `docs/testing-strategy.md` (the decay invariant), `docs/experiments/2026-07-05-adversarial-playtest.md` (the 2f-aliasing finding); merge-window (island) sense in `docs/integration-tutorial.md`.

### Back-coupling / loose coupling / co-simulation
The two-way but deliberately loosely-synchronized link between Vintage Story's mechanical (shaft) simulation and Manatee's electrical simulation: each side periodically reads a slowly-updating, averaged summary of the other, rather than the two being solved together.

The mechanical network re-sums torque only every ~100 ms, while the electrical island substeps much faster. So shaft speed enters the electrical solve as a piecewise-constant parameter per tick, and the alternator reports its counter-torque back as the *mean over all electrical substeps since the previous mechanical read* (the normative averaging window — an instantaneous sample would alias the 2×-line-frequency power pulsation into a phantom DC torque). For programmers: this is two independent state machines exchanging rate-limited, smoothed telemetry instead of sharing state — like two services polling each other's cached metrics rather than making synchronous RPCs. For the EE: it works because both physical systems have inertia (rotational mass on one side, storage elements and slow time constants on the other), so neither reacts faster than the exchange cadence. "Co-simulation" is the standard umbrella term for coupling two solvers this way; "back-coupling" is our name for the reverse path (electrical load → mechanical counter-torque).

**Standard term:** co-simulation with weak (loose) coupling — a single-pass, non-iterated (Jacobi-style) data exchange, with the reverse path time-averaged rather than iterated to convergence. (Not to be confused with waveform relaxation, which iterates the coupled solve over a time window; the only alpha-relaxation mechanism in this project is the transformer/power-transfer CouplerSpec, a different object entirely.)

**Where it appears:** `docs/vintage-story.md` §2 (Cadence consequence, averaging window), `docs/design.md` (Mechanical co-simulation bullet).

**References:** Gomes, Thule, Broman, Larsen & Vangheluwe (2018), "Co-simulation: A Survey", ACM Computing Surveys 51(3).

### baseKey.Derive(ordinal) / StateKey.From(key)
Two deterministic key-construction helpers: `Derive` mints stable sub-keys for the several components inside one multi-component device, and `From` reuses a component's `ExternalKey` bits as its `StateKey` when the device is a single bare component.

A "device" as the game sees it (a battery, say) is often several solver components (an EMF source, an internal resistor, a state-of-charge integrator). Each component needs its own `ExternalKey` (topological identity), so the device takes one base key and calls `baseKey.Derive(ordinal)` for component 0, 1, 2… The crucial property is *determinism*: the same base key and ordinal always yield the same derived key, in every session. That is what makes restore-by-key work — a reloaded world rebuilds its netlist, re-derives the same keys, and every saved blob and cached reference finds its component again. For the EE reader, it is the same idea as a hierarchical reference designator: the battery is `B3`, its internals are `B3.1`, `B3.2`, assigned by rule rather than by whatever order they happened to be created in, so the schematic and the parts list always agree. `StateKey.From(key)` covers the degenerate case: a device that is exactly one component has no reason to invent a second identity for its saved state, so its `StateKey` is just its `ExternalKey`'s 128 bits reinterpreted (this is also the default for coupler state). What matters in both cases is uniqueness per netlist (debug-asserted) and determinism across sessions.

**Project-coined API**, closest standard concept: deterministic hierarchical key derivation (compare hierarchical instance names in SPICE subcircuits, or derived keys in content-addressed stores).

**Where it appears:** api.md (ExternalKey §, `DeviceHost` §18, snapshot/restore §); integration-tutorial.md §3 (ExternalKey vs StateKey) and §4 (DeviceHost.Add building components under derived keys).

### BeginBulkBuild / BulkBuild
- The load-time batching scope on `ConductorGraph`: many `AddSegment` calls are staged, and one compaction pass runs when the scope closes, instead of recompacting after every segment.

`BeginBulkBuild(expectedSegments)` returns a disposable `BulkBuild` object used in a C# `using` block; segments added inside it only *stage* into buffers that grow geometrically (doubling) from the presize hint, and the expensive work (series-chain compaction, island construction, plus partition/merge validity checks) happens exactly once at `Dispose` — the moment the `using` block exits, at load time, never deferred to the first solve. Islands sit in `Building` status throughout. Outside a bulk scope, the two mutators differ in cost: each individual `AddSegment` recompacts eagerly — correct but currently O(graph) per call — while `RemoveSegment` merely dirties the affected island, with the rebuild coalesced to the next `Solve` (`docs/api.md` §19). For an EE, the analogy is netlist entry: you would not re-run the reducer (a load-flow study) after typing each line of a schematic — you enter the whole netlist and reduce once. This is the intended path for chunk loading / world load, where thousands of cable segments arrive at once; the canonical use is Stationeers save load: ~10k segments in one scope, compacting to ~143 nodes. It also coalesces the event journal (one `IslandCreated` summary per island rather than per-segment events). The heap allocation caused by staging-buffer growth is charged to BulkBuild Dispose (load time), never to the first `Solve`.

**Project-coined term**, closest standard concepts: batched/deferred rebuild (same pattern as `BeginBulkUpdate`, but at the `ConductorGraph`/reduction layer rather than the raw netlist, and specialized for the load path); batch insertion / deferred index rebuild (databases); RAII or `IDisposable` scope guard (Stroustrup; C# `using` statement).
- **Where it appears:** `docs/api.md` §19 (`ConductorGraph.BeginBulkBuild` signature and semantics), §6 (staging growth), §15 (journal summary events), §22.a (Re-Volt load walkthrough), event-journal notes in `docs/api.md` §events; `docs/integration-tutorial.md` §3 ("Intake: ConductorGraph, BulkBuild, and the two keys"); `docs/compaction.md`.

### Bounded lag
The guarantee that when two coupled islands exchange values across a power-transfer boundary, each side's view of the other is at most one substep old — the staleness is bounded and small by construction.

Separate islands are solved as separate matrices, so a boundary coupler cannot give both sides simultaneously exact values; one side necessarily works from the other's most recent result. Manatee bounds that staleness: boundary-coupled islands are scheduled as one lockstep unit and exchange amplitude+phase every *substep*, so the lag is exactly one substep — 1/20 of an AC cycle at the standard 20 samples per cycle. (Exchanging only once per game tick instead would mean a quarter-cycle of lag at 5 Hz — far too coarse.) The residual error from this lag is further damped by the coupler's relaxation α and cannot create energy: the coupler clamps transfer to what the source island actually delivered, and its energy ledger dumps any accounting surplus as heat. For a programmer this is the familiar shape of an eventual-consistency bound turned hard: replicas (islands) sync on a fixed schedule, and the protocol guarantees a maximum staleness of one sync interval. For an EE: it is a fixed, known transport delay of one sample in the coupling path — small enough, at 20 samples/cycle, to behave like a slight phase error rather than an instability.

**Project-coined term**, closest standard concept: the bounded communication-step delay of co-simulation / waveform-relaxation schemes (staleness of exchanged boundary waveforms limited to one exchange interval).

**Where it appears:** `docs/solver.md` Islands (coupling-device exchange scheme, settled 2026-07-05); `docs/api.md` §7 and §11 (per-substep exchange, lockstep scheduling units).

### breaker (as coupling device)
- The settled design decision (2026-07-05, the "relay-vs-breaker duality") that a breaker is *not* a switch element inside a circuit matrix but a device at the islands layer that merges or separates whole islands.
- Physically a breaker and a relay contact do the same thing — make or break a connection — but Manatee gives them different API citizenship chosen per device type. A **relay contact** is a netlist switch: a tier-2 parameter change that stays inside its island's matrix, fast enough for elevator-style control logic that toggles every tick. A **breaker** is a coupling device: closed = the two islands are one merged matrix; open = separate matrices; and the *opening* transition is a full tier-3 island rebuild (split detection is deliberately not attempted — the rebuild is "the honest cost"). This asymmetry is acceptable precisely because breakers are safety devices that change state rarely, while relays are logic devices that change constantly. For an EE: it is a modeling-granularity choice, like deciding which switches justify re-deriving the network equations versus which are handled as a conductance change. For a programmer: it is choosing between a cheap in-place update and an expensive re-partition, keyed on expected change frequency — a classic read-path/write-path trade.
- **Project-coined framing**; closest standard concepts: an ideal switch element (the relay side) versus network partitioning / tearing at the system level (the breaker side).
- **Where it appears:** `docs/solver.md` Islands ("Relay-vs-breaker duality"); `docs/design.md` Doc-review round (2026-07-05) and the Stationeers client section; `docs/stationeers.md` "Islands and Coupling Devices".

### breaker / recloser lockout / sync-check breaker
- The gameplay-facing faces of the breaker device: a fault reporter, the player's lockout-tagout tool, and (sync-check variant) a guard against connecting two out-of-phase AC systems.
- Three roles from the design docs. **Fault locating (R15):** a breaker reports the voltage and current it saw at the moment of trip; since a short's fault current is limited mostly by cable resistance between breaker and fault, V/I approximates the resistance — and hence the distance — to the short. One division turns a mystery walk down the cable run into a taught skill. **Lockout (hazards):** in Vintage Story, chiseling into a *live* conductor shocks you; the protection mechanism is not a special case in the code but the real-world procedure — turn off the breaker first. This is lockout-tagout as an emergent skill: the sim adds no griefing-protection logic, only legible physics. **Sync-check (mastery off-ramp):** manually synchronizing an alternator to a live AC grid by ear is a taught thrill; the sync-check breaker is the craftable convenience that refuses to close while the two sides are out of phase, so the skill never becomes a chore. For programmers: the sync-check is a precondition guard on the merge operation; the trip report is structured error context attached to the event that split the islands.
- **Standard terms:** all three are real power-engineering devices/procedures — circuit breaker, lockout-tagout (LOTO, an OSHA-codified procedure), and synchronism-check ("sync-check", ANSI/IEEE device number 25) relays. "Recloser lockout" is likewise a real utility concept (a recloser giving up after repeated trips). Our usage matches the real ones; the project decision is that they are gameplay content, not solver special cases.
- **Where it appears:** `docs/design.md` R7/R15, Hazards ("Cable voxels are never protected from the chisel"), and the Mastery corollary; `docs/vintage-story.md` §1 (chisel interaction).

### brownout clamp / brownout dropout
- The legacy-device adaptor's rule that below a voltage threshold an adapted device is dropped from the circuit entirely and the vanilla game is told it received nothing.

In stationeers.md this is called the **brownout clamp**: it caps how bad the constant-power feedback spiral can get by removing the load once voltage falls under the threshold, while the death spiral *above* the threshold remains real, shipped physics. design.md R18 extends the bare clamp into full **brownout dropout with hysteresis** — drop at `V_low`, rejoin only at `V_high`, staggered rejoin, recloser-style lockout — after an adversarial playtest showed the plain single-threshold version is a tick-rate flapper. "Clamp" here is not the EE sense of clamping a voltage to a rail; it is a limiter on behavior — think of a circuit breaker or a process supervisor killing a runaway job, not a Zener diode. The "vanilla told it received nothing" half keeps the game's own bookkeeping consistent so the mod stays cleanly removable.

**Project-coined term**, closest standard concept: **undervoltage load shedding** (grid scale) / **undervoltage lockout, UVLO** (device scale).
- **Where it appears:** stationeers.md "The Legacy-Device Adaptor"; design.md R18; realized as `AdaptedLoad` in the API.

### Building
- An island status meaning "this circuit is being constructed or rebuilt on a background thread — don't trust it yet, keep using the last known-good answer."

Every island (independent connected circuit) carries a status from the state machine `Empty → Building → Ready → Dirty → Faulted`. While an island is `Building`, its heavy setup work (symbolic analysis, matrix factorization) runs on a background task; meanwhile reads (`Solution.Voltage` etc.) return the last *published* solve, `Step` is a no-op, and `Solution.IsLive(island)` returns `false` so the client knows the numbers are stale. The actual flip `Building → Ready` — like all global mutations — commits on the sim thread in a topology phase, never concurrently with a solve. For an EE: think of it as a bus section tagged out for work while the panel meters keep showing the pre-outage readings, clearly flagged as frozen. The concrete payoff is Stationeers' save load: 10k cable segments build in the background while the game runs Re-Volt's simple scalar fallback until each island flips Ready.

**Project-coined term**, closest standard concepts: a state in a lifecycle state machine; double-buffered / copy-on-write handoff between a worker thread and a consumer.

**Where it appears:** `IslandStatus` enum and state machine, `docs/api.md` §11; off-thread tier-3 compute/commit split, §21; Re-Volt load walkthrough, §22.a; `docs/stationeers.md` (background rebuild handoff).

### cable-segment granularity
- Recording the wiring graph one physical cable piece at a time, rather than one lump per merged network.

Stationeers itself only cares that a cable run is connected, so its network builder throws away per-cable adjacency once it has merged everything into a `CableNetwork`. Manatee needs the fine detail — each segment becomes a resistor edge whose resistance comes from cable type and length, and fuses live at specific positions in the run — so Re-Volt's `NetworkExport` recovers adjacency from each cable's `OpenEnds` *before* the builder discards it, and hands the core `CableInfo { RefId, Connections }` per segment. For a programmer: it is the raw, unaggregated event stream kept because a downstream consumer needs it, even though the producer only wanted the aggregate. For an EE: it is the full distributed-resistance model of the run, which the compaction layer then legitimately collapses (series chains of ~10k segments reduce to low-hundreds of nodes) while keeping enough bookkeeping to attribute a limit trip back to the exact culprit segment.

**Project-coined term**, closest standard concept: modeling a wiring run as a per-conductor-section lumped-element network before node reduction.
- **Where it appears:** `docs/stationeers.md` Graph Construction; the intake side of `ConductorGraph.AddSegment` in `docs/integration-tutorial.md` §3.

### CableNetwork / CableNetworks
- Stationeers' own grouping of connected cable runs, which the game hands to manatee-core as ready-made partitions instead of the solver discovering them itself.

The Stationeers engine already merges every contiguous run of cable into one `CableNetwork` object, so when Manatee integrates via Re-Volt it does not need to rediscover connectivity from scratch: the netlist is built in `ClientPartitioned` mode, each node tagged with its CableNetwork's id as a `PartitionKey`. Solver islands are then computed by union-find *over* these networks: two CableNetworks whose ids never merge in the game can still belong to one solver island when a closed coupling device (breaker, bus-tie) bridges them galvanically, while a boundary coupler (a decoupling transformer/converter, `CouplerId`) keeps them as separate islands that form one lockstep scheduling unit, exchanging power per substep — couplers are the only sanctioned cross-partition join in this mode. (Distinct from the `AddIdealTransformer` primitive, which stamps both ports into one matrix and merges the two sides into a single island.) In programming terms, CableNetworks are a pre-computed connected-components cache supplied by the host game; Manatee treats them as trusted input and layers its own coarser grouping (islands) on top. For an EE: a CableNetwork is roughly one physical wiring harness; an island is the actual electrical circuit, which may span several harnesses joined by closed switchgear.
- **Where it appears:** `docs/stationeers.md` (Integration Seams, Islands and Coupling Devices), `docs/solver.md` (Islands: "pre-partitioned" input), `docs/design.md` (Stationeers section), `docs/api.md` (`PartitionKey`, `SetPartition`).

### cadence
- The fixed, regular interval at which a client game asks the solver to advance the simulation.

Each integration has one cadence, set by the host game's own loop rather than by Manatee: Stationeers is a self-clocked 500 ms tick (`DefaultTickSpeedMs = 500` — a stopwatch-paced loop, not an OS timer or per-frame call), and Manatee solves DC plus storage dynamics with dt = 0.5 s and no subcycling there; Vintage Story's mechanical network updates at roughly 100 ms. Cadence matters numerically because a fixed dt keeps Backward Euler companion conductances constant, which is what lets steady-state ticks stay in tier 1 against a cached LU; it also matters for correctness — `docs/vintage-story.md` notes that sampling a counter-torque at the wrong cadence would alias, the same phenomenon as under-sampling a signal. Programmers can read it as the period of a fixed-timestep game loop; EEs as the system's sampling interval.
- Standard usage (common in real-time and game-engine parlance for "tick rate/interval"); the EE-adjacent concept is the sampling period.
- **Where it appears:** `docs/stationeers.md` (Performance, Threading), `docs/solver.md` (mechanical-input cadence, coupler exchange), `docs/vintage-story.md` (aliasing note).

### canon / canon-pending / binding / decision log
- The project's status vocabulary for how much authority a document or record carries, and which document wins when two of our design docs disagree.

A doc (or a specific section) marked **binding** is the contract: code and other docs must conform to it. **Canon** means "the settled, authoritative account"; **canon-pending** means it is expected to become canon but still awaits the owner's sign-off. `docs/api.md` is headed "Status: CANON pending owner review" — it is the as-built API surface produced by the autonomous overnight build, treated as the binding contract for all clients, but the human owner has not yet ratified it. Contested design calls within it are recorded in its **Decision log** (api.md §24), a numbered list of decisions with rationale, so a reviewer can audit or reverse any single call without archaeology; unresolved residue lives separately in Open items (§23). At the other end of the authority scale, experiment reports under `docs/experiments/` are explicitly "decision inputs, not canon" — evidence, never contract. The concrete precedence instance: `docs/stationeers.md` carries a revision-pending banner declaring itself "do not treat as canon on the contested points" — after the 2026-07-06 core build, `api.md` superseded parts of it (the per-network 3-phase seam vs. the one-global-tick-body model, the incremental-hooks claim, the galvanic-bridge assertion), so **api.md §22.a / §7 / §23 are binding** wherever the two disagree, until stationeers.md is rewritten and signed off by Sukasa. Programmers can read this as precedence rules for conflicting configs; an EE can map it onto engineering change management and drawing revision control — canon is the released drawing, canon-pending is a drawing awaiting the approver's stamp, the decision log is the ECN/ECO record explaining each change, and the stamped revision governs while superseded sheets are marked as such rather than deleted.

**Standard term:** closest concepts are "normative vs. informative" text in standards documents (e.g. RFC/ISO usage), Architecture Decision Records (ADRs), and drawing revision control.

**Where it appears:** `docs/stationeers.md` revision-pending note (top of file); `docs/api.md` header, §22.a/§7/§23 (the referenced binding sections), §24 (Decision log), §23 (Open items); `CLAUDE.md` (api.md as "the binding as-built surface", experiments as "decision inputs, not canon").

### capstone

### Catenary / catenary span
The sagging-cable curve used to render a bare wire that hangs between two support posts in the Vintage Story mod, instead of drawing the wire block-by-block.

A catenary is the shape a uniform flexible cable naturally assumes when suspended from two points under gravity (a cosh curve — visually similar to a parabola). In Manatee it is purely a *rendering* choice, not an electrical one: bare wire need not be supported at every voxel, so a span between two posts is drawn as a single static catenary mesh, while short runs are drawn as ordinary voxel geometry — the renderer picks the representation by the number of wire voxels between supports. The plan borrows the segment-transform math and two-endpoint activation model from Vintage Story's rope/cloth system but deliberately skips its live mass-spring physics (no wobble, no rip, no per-frame cost). Electrified fences fall out of this: a bare-wire span is the fence. For a programmer: think of it as a level-of-detail switch — same underlying circuit edge, two interchangeable visual encodings chosen by span length. For an EE: this is the same curve as a real overhead transmission line span, and "span" is used in the same sense.

The term is standard (mathematics/civil engineering); "catenary" is also the standard railway/power-line term for suspended overhead conductors.

**Where it appears:** `docs/vintage-story.md` (Wire Rendering — cloth-system precedent, `ClothSystem.CreateRope`, gotchas on render range and region boundaries); `docs/design.md` (Device notes — electrified fences, renderer-selection rule).

### `cct=` / `ctz=` URL forms
The two query-string parameters falstad.com uses to embed an entire circuit inside a shareable link — documented for completeness; Manatee's importer does not consume URLs.

The Falstad circuit simulator can serialize a circuit as a compact text format (one line per element), and its share links carry that same text in the URL: `?cct=` holds it essentially raw (URL-escaped, with `%24` standing in for `$`, the line separator), while `?ctz=` holds it LZW-compressed and base64-encoded to keep long circuits within practical URL length. For a programmer this is just two transport encodings of one payload — plaintext vs. compressed — like a `.txt` next to its `.gz`. For an EE: the payload is the netlist itself, so a Falstad link *is* the circuit, not a pointer to one stored on a server. Manatee cares because these URLs are part of the installed base of classic-text-format circuits (as opposed to the newer XML export), but the importer's accept/reject contract is defined over the decoded text, not over URLs.

These are Falstad/circuitjs1's own parameter names (see `CirSim.java:171-178`, `ExportAsUrlDialog.java:101`), not a Manatee coinage.

**Where it appears:** `docs/falstad-format.md` §1 (URL forms note) and the format-history discussion of where classic-text circuits survive.

**References:** Paul Falstad / Iain Sharp, *circuitjs1* source code (the format's only authoritative specification).

### change-cost tier
A project-coined classification of every netlist edit by how much work the solver must redo: (1) value-only changes that reuse everything, (2) conductance changes that redo the numeric factorization, (3) topology changes that rebuild the island from scratch.

The solver keeps a factored matrix (an LU decomposition) cached between solves, the way a database keeps a compiled query plan. Tier 1 — changing a source value, or the per-step history terms of a capacitor/inductor companion model — only alters the right-hand side of the equation system, so the cached factorization is reused and the solve is a cheap forward/back substitution; this is the every-tick hot path, and a linear island in steady operation lives entirely here. Tier 2 — a switch toggle, variable resistor, fuse blow, or one Newton iteration — changes matrix *values* but not its sparsity *pattern*, so the symbolic analysis (the pattern-dependent plan) is reused and only the numeric factorization is redone. Tier 3 — adding or removing a component or node — changes the pattern itself and forces a full symbolic + numeric rebuild of the island, so it is batched. Requirement R4 makes the tiers explicit in the API precisely so that a client author who is not a numerical analyst can still reason about the cost of every call; both games' integration seams are designed to map onto them.

**Project-coined term**, closest standard concept: matrix reuse levels — RHS update / numeric refactorization with reused symbolic analysis (KLU-style refactor) / full re-analysis.

**Where it appears:** `docs/design.md` R4; `docs/solver.md` "Change-Cost Tiers" and "The Netlist API" (every mutator's doc-comment is tagged with its tier); `docs/integration-tutorial.md` §1; `docs/testing-strategy.md` (equivalence tests and the standing per-tier benchmark suites).

**References:** Davis, T. A. & Palamadai Natarajan, E. (2010), "Algorithm 907: KLU, a direct sparse solver for circuit simulation problems", ACM TOMS — the symbolic/numeric refactorization split tier 2 relies on; Vlach, J. & Singhal, K., *Computer Methods for Circuit Analysis and Design* — LU factorization and forward/back substitution in circuit simulators.

### change-cost tier (T0/T1/T2/T3)
The API document's explicit four-level notation for what each call costs the solver: T0 touches no matrix at all, T1 changes only right-hand-side values, T2 changes conductances (numeric refactorization), T3 changes topology (island rebuild).

Every member in `docs/api.md` carries a `[Tn]` tag, so the cost profile of any client integration can be audited by grep rather than by understanding the numerics — the point (per R4) is that the performance model is *legible*, not folklore. T0 is metadata (e.g. `SetAmbient`, an envelope recompute that never touches the matrix; the `Meta` facade is the one tier-0 surface). T1–T3 are the solver's change-cost tiers, and the API binds them into its verb grammar: `Drive` is T1, `Adjust` is T2, structural edits inside a `StructuralEdit` scope are T3, and `Reconfigure` (coupler membership) is "T3-lite" but reports as `Tier.Topology` — decision log §24.11 explicitly forbids a fifth tier value. The performance contract is also stated in tier terms: tiers 0–2 allocate zero heap bytes after warmup.

**Project-coined notation**, closest standard concept: matrix-update cost classes — RHS-only update vs. numeric refactorization vs. symbolic re-analysis; T0 has no standard analogue because it is by definition outside the solver.

**Where it appears:** `docs/api.md` §"Notation" (`[Tn]` tags throughout), the four-verbs invariant, decision log §24.11; `docs/design.md` R4 and R8.

**References:** Davis, T. A., *Direct Methods for Sparse Linear Systems*, SIAM, 2006 — the symbolic-analysis/numeric-factorization separation underlying the T1/T2/T3 distinction.

### change-cost tier / tier-1/2/3
The three-level cost classification of netlist edits as used at the Stationeers/Re-Volt integration seam: tier-1 value changes (back-substitution on a cached factorization), tier-2 conductance changes (numeric refactor, symbolic reused), tier-3 topology changes (full island rebuild).

In `docs/stationeers.md` the tiers are the design language for slotting Manatee into Re-Volt's replacement of the vanilla power tick: `Initialize_New` (dirty-driven, fires on topology change) is the tier-3 moment where the island graph is rebuilt or patched; `CalculateState_New` (gathering device demand/supply) is tier-1/2, where the adaptor updates stamps from what devices want; the solve itself replaces the vanilla scalar `_powerRatio` computation. The mapping is the practical payoff of R4: a game modder can look at which vanilla phase they are in and know from the tier which solver cost they are paying, the same way a web developer distinguishes a cache hit, a recomputation, and a schema migration. Same taxonomy as the solver's change-cost tiers; this entry records the seam-mapping usage.

**Project-coined term**, closest standard concept: matrix update classification — RHS update / refactorization / re-analysis.

**Where it appears:** `docs/stationeers.md` "Integration Seams"; `docs/design.md` R4.

### Chisel material / chisel material blocks
- A block type that Vintage Story's chisel tool is allowed to carve into microblock voxels — in Manatee, cable materials (copper wire, heating wire, insulation) are deliberately made chisel materials so cable voxels ride the game's normal voxel pipeline.

In Vintage Story, a chiseled block stores a list of "materials" (block ids) plus a list of small cuboids saying which sub-voxels are made of which material. Our reading of the engine source (`BlockEntityChisel.AddMaterial`, `BEChisel.cs:457`) shows any block id can be appended as a material; the whitelist (`ItemChisel.IsValidChiselingMaterial`) only governs which blocks a player may *start* chiseling, and the block attribute `"canChisel": true` forces acceptance. By registering our cable blocks as legal chisel materials, cable voxels become ordinary microblock voxels that the vanilla engine meshes, culls, selects, and collides for free — like implementing a feature by conforming to an existing plugin interface instead of forking the renderer. For an EE: the cable's *geometry* is handled entirely by stock game machinery; only its *electrical semantics* live in our code. The catch (documented alongside): material blocks get no runtime hooks — no tick, no interaction — so all electrical behavior must live on the block entity or an attached behavior.
- **Where it appears:** `docs/vintage-story.md` sec 1 ("Any block can be a chisel material", with engine file:line evidence); `docs/design.md` R17 and the cooking-slab example.

### churn-tick re-pin
- The binding rule that on any game tick which changed circuit topology, an integration must re-resolve its device handles a *second* time — after `Solve` runs, but before it reads results back — because the rebuild inside `Solve` just reissued every handle.

Manatee identifies devices two ways: a cheap **handle** (like a raw array index — fast, but only valid for the current internal layout) and a durable **ExternalKey** (like a database primary key — survives anything). When topology changes (a cable is cut, networks merge), the affected island is rebuilt *inside* `Solve`, and every component/node handle in that island is reissued. So the normal early-tick re-pin phase (triggered by `DrainChanges`) is not enough on such a tick: the handles it resolved were invalidated by the very `Solve` that followed. The rule: `if (topologyChanged || changesDrained) Repin(...)` immediately after `Solve`, before any readback dereferences handles. Missing it is not a crash — stale reads become counted no-ops — but `TickStats.StaleHandleReads != 0` is defined as a re-pin bug, and the walkthrough's post-tick assert fires. For an EE: it is like re-clipping your meter probes after physically rewiring the board — the old attachment points no longer correspond to the nodes you meant. The executable fake (`FakeRevoltClientTests`) proves both directions: with the re-pin the scripted run holds `StaleHandleReads == 0`; without it, a cut tick trips the counter.

**Project-coined term**, closest standard concepts: iterator/pointer invalidation after container reallocation; cache invalidation on rebuild.

**Where it appears:** `docs/api.md` §22.a step 4 (the Re-Volt walkthrough tick loop), backed by the adaptor re-pin contract in §18 and the error-model table in §20; exercised by `examples/RevoltWalkthrough/` and `FakeRevoltClientTests`.

### Circuit (per-island Circuit)
- The solver-layer object that holds one island's assembled MNA matrix, its factorization, and the stamp storage that fills it — one `Circuit` instance per electrically connected island.
- In Manatee's layering, `netlist` tracks components and connectivity for the whole world, but actual number-crunching happens per island: each island gets its own `Circuit`, which owns the node-to-matrix-row mapping, the coordinate-format stamp slots, the gmin shunts and reference-node handling, and the LU factorization it reuses between solves. For a programmer: it is the compiled, mutable working set behind an island — like a prepared statement plus its cached query plan, rebuilt only when topology changes (`Analyze`) and re-executed cheaply otherwise (`Set*`, `FactorizeIfDirty`, `Solve`, all heap-free after build). For an EE: it is simply "the matrix" for one connected subnetwork; disconnected subnetworks never share a matrix, so a small change in one island never re-factorizes another.
- **Standard term:** there is no single standard name; the closest concepts are the per-subcircuit (island) MNA system matrix and its LU factors. Note the name collides with the everyday EE word "circuit" — in Manatee code, capital-C `Circuit` always means this internal solver object, not the user-visible schematic.
- **Where it appears:** `docs/solver.md` (Layering, Islands), `core/Manatee.Core/Solver/Circuit.cs` (internal class; not part of the public API — clients only see `netlist`).

### classic text format
- The original line-oriented, plain-text Falstad circuit file dialect — one element per line, space-separated fields — as opposed to the XML dialect that current circuitjs1 exports; this is the dialect Manatee's importer targets.

### client
- A consumer of manatee-core: one of the three game-facing integrations — the Vintage Story voxel world, the Vintage Story tablet (2D schematic simulator), or Stationeers via the Re-Volt mod.
- The core is game-agnostic; each client supplies conductor geometry and component placements in its own natural terms (voxels, schematic wires, cable networks) and receives observations back (node voltages, branch currents, limit events) mapped onto that geometry. Clients talk only to the `netlist` and `devices` layers, never directly to the solver, and nothing in manatee-core names a game. For an EE: the client is the "physical plant" description plus the instrument panel — it says where the copper is and reads the meters; the core does the math in between. For a programmer: this is the ordinary library-consumer sense of "client" (as in client/server or API client), with the compaction layer acting as the adapter between each client's geometry and the solver's netlist.
- **Where it appears:** `docs/design.md` (The Three Clients), `docs/compaction.md` (intro, Client Intake Contracts), `docs/solver.md` (Layering).

### client intake contract
- The defined agreement for how each client feeds its geometry into the reduction (compaction) layer — what shape of input each game hands over, and what the layer does with it.
- Each of the three clients has its own contract: the Vintage Story voxel world delivers sparse voxel storage that is greedy-meshed into prisms and grouped into regions (sparky's proven pipeline, with three block representations feeding one intake); Stationeers delivers `NetworkExport` adjacency data (cable segments, ports, fuses); and the tablet/2D harness delivers schematic wires that are already near-minimal, so they pass through essentially untouched (a wire is an ideal node, per Falstad convention). For a programmer: these are input-adapter specifications — one internal pipeline, three format-specific front doors. For an EE: think of it as the standardized terminal block where each installation's as-built wiring diagram gets handed to the analysis department, in whatever drawing convention that installation uses.
- **Standard term:** input adapter / ingestion contract (the pattern is essentially Gamma et al.'s Adapter, applied to data ingestion).
- **Where it appears:** `docs/compaction.md` (Client Intake Contracts section); per-client detail in `docs/vintage-story.md` (R17) and `docs/stationeers.md`.

### ClientPartitioned / PrePartitioned
- The pair of option values (one on `Netlist`, one on `ConductorGraph`) that tell manatee-core the *client* — the game code using the library — decides which partition (isolated sub-network) every node belongs to, and the core must enforce those boundaries.

In Stationeers, cable networks are separate by game rule: two CableNetworks never merge except through an explicit coupler device. `PartitioningMode.ClientPartitioned` (in `NetlistOptions`, set by the `NetlistOptions.Stationeers(dt)` bundle) turns that game rule into an enforced API contract: every node carries a client-assigned `PartitionKey` (the CableNetwork id), the `partition` argument becomes mandatory on `ConductorGraph.AddSegment`, and an `Edit` that would bridge two partitions with an ordinary component throws `PartitionMergeException` and rolls the whole batch back atomically. `GraphOptions.PrePartitioned` is the matching setting on the `ConductorGraph` reduction layer. For an EE: it is like an interlock that physically prevents you from landing a jumper between two galvanically isolated buses — the only sanctioned bridge is a coupler (think isolation transformer / DC-DC link). For a programmer: it converts "networks never merge" from a discipline the adaptor must maintain into a type-and-runtime-checked invariant — violations fail loudly at the edit, not as silent corruption later. The opposite modes, `SelfPartitioned` (Netlist) / the self-partitioned graph option, let the core discover connectivity itself, as Vintage Story uses.

**Project-coined terms**, closest standard concepts: enforced graph partitioning; design-by-contract precondition on netlist edits.

**Where it appears:** `docs/api.md` §s on `NetlistOptions.Partitioning`, `PartitionKey`, and commit rejection; `docs/integration-tutorial.md` §2–§3 and sharp-edge #12.

### Coalescing (matrix work)
- Our rule that no matter how many structural changes hit an island in one tick, the expensive matrix reconstruction happens at most once, at the next `Solve`.

Structural edits (removals, galvanic-coupler opens, merges superseded by rebuilds) mark an island `Dirty` immediately, but the symbolic+numeric rebuild of its equation matrix is deferred and folded together: a deconstruction burst of twenty removals still costs exactly one rebuild (`TickStats.IslandRebuilds` counts it). For programmers this is classic batching/debouncing of an expensive recompute; for an EE, think of it as re-deriving the nodal equations once after all wiring changes are done, rather than after every snipped wire. The binding caveat (api.md §17 rule 3, §15): coalescing discards **matrix work only, never events** — every `IslandId` that ceases to exist still emits exactly one terminal event (`IslandsMerged`, `IslandRebuilt`, or `IslandRemoved`) in both the journal and `DrainChanges`, so observers never see history edited, only work skipped.

**Project-coined term**, closest standard concepts: batching / debouncing of recomputation; lazy re-evaluation.
- **Where it appears:** `docs/api.md` §17.2–17.3 (the Reconfigure-to-Solve story), §15 (event-stream completeness invariant), §6 `RemoveSegment`.

### Cold / cold-path
- Project notation marking API members that are diagnostic or setup-time paths, never executed during a simulation tick, and therefore allowed to allocate heap memory.

The API annotates every member with an allocation budget: `0B` (zero allocation after warmup, safe every tick), `shape` (allocates only at structural-commit "shape" time), and `cold` (diagnostic path, never in a tick — may allocate freely). `Edit`/`Commit` are the archetypal **shape** members (`Commit` is annotated `shape (may alloc)`), and island rebuilds are shape work too — amortized structural cost triggered lazily at the next `Solve`, not steady per-tick numeric cost. Genuinely **cold** members are the diagnostic/setup paths: `SaveCanonical`/`SaveNormalized`, `SpiceDeck.Emit`, `Diff`, `ReproBundle`, and `WaveformTap.Attach` (a call-once subscribe). The distinction exists because per-tick allocation causes garbage-collector pauses that would stutter a game; it is enforced by BenchmarkDotNet's MemoryDiagnoser in CI. An EE analogy: cold paths are like a bench-calibration mode — you may take the instrument offline and do slow, thorough work there, but never while the circuit is live. Note this is the opposite polarity of the common CPU-cache sense of "cold" (rarely-executed code); our sense matches that intuition but is specifically an *allocation license*, not just a frequency observation.

**Project-coined notation**, closest standard concepts: cold path / slow path (vs. hot path) in performance engineering.
- **Where it appears:** `docs/api.md` notation preamble (§ front matter), §13 (`WaveformTap.Attach`), §21 allocation-buckets table ("cold, never in a tick" row).

### CollectDirty
- The API call that fills a caller-supplied buffer with one handle per island (or coupled-island group) that needs solving this tick — the worklist for parallel dispatch.

`IslandTable.CollectDirty(Span<IslandHandle> into)` writes one `IslandHandle` per **scheduling unit**: independent islands appear individually, while islands linked by boundary couplers (transformers/converters, which must be solved together) deduplicate to their unit lead, so the caller never double-schedules half of a coupled pair. It is annotated `0B` — zero heap allocation — because the caller owns the buffer; the core just fills it. This is the first half of the normative per-tick fan-out idiom (collect once, then dispatch cached closure-free work items across threads). For an EE: the solver keeps a dirty flag per electrically independent sub-circuit, and this call reads out the list of sub-circuits whose equations must be re-solved, one entry per group that shares a matrix-level coupling.

**Project-coined API member**, closest standard concepts: a dirty-set / worklist query, as in dirty-flag invalidation and worklist algorithms.
- **Where it appears:** `docs/api.md` §11 (`IslandTable`), §22.b (parallel scheduling idiom); `docs/integration-tutorial.md` tick-loop example.

### `#` comment lines
Lines beginning with `#` in a Falstad-format circuit file, used to carry human-readable teaching narrative alongside the circuit description.

The upstream Falstad/circuitjs1 text format has **no comment syntax at all** — every line is expected to describe an element or a setting. Electrical Age (the Minecraft mod Manatee succeeds) extended the format with `#` comment lines and put its lesson prose in them, and Manatee adopts that convention for its own lesson corpus: the electrical content and the teaching narrative live in one file. Compatibility is graceful but noisy: when circuitjs encounters a `#` line it logs an exception to the browser console and skips the line, so lesson files still load unmodified in falstad.com for authoring and debugging. Think of it like `//` comments in source code — ignored by the machine, essential to the human — except that the original "compiler" never defined them, so it merely tolerates rather than understands them.

**Project-coined term** (inherited from Electrical Age), closest standard concept: a comment line — nonexistent in the upstream Falstad format.

**Where it appears:** `docs/falstad-format.md` §1, §6 (EA dialect), §7 (importer accept-list and corpus rule); the R20 lesson corpus (`lessons/`).

### compaction / reduction layer / ConductorGraph
- Manatee's name for the shared layer that turns raw conductor geometry (thousands of voxels or cable segments) into the smallest electrically-equivalent circuit before it is handed to the solver, while remembering enough to reverse every observation back to physical geometry. "Compaction" and "reduction layer" are interchangeable names for the concept and design (`docs/compaction.md` is titled "Reduction Layer ('compaction')"); `ConductorGraph` (`core/Manatee.Core/Reduction`) is the class that implements it.

  The layer sits between the game clients and the netlist and has five responsibilities: **region building** (grouping equipotential geometry into single circuit nodes via union-find), **series-chain collapse** (runs of resistor edges with no taps become one equivalent resistor — ~10k Stationeers cable segments reduce to low-hundreds of nodes), **probe interpolation** (voltages at eliminated interior points are recovered by resistance ratio, so instruments can still read "inside" a collapsed run), **limit attribution** (each collapsed element carries a limit envelope — per-limit-type minima over its constituents — so when the solver reports an overcurrent on the equivalent resistor, this layer answers *which* voxel or segment melts; the lead-fuse-in-a-copper-run case works with zero special-casing), and **island bookkeeping** (connectivity tracking feeding the solver's independent sub-circuits). It implements R10 (series collapse with probe interpolation) and R11 (incremental topology maintenance: additions and merges are handled incrementally, any removal rebuilds the affected island, and a periodic full-rebuild "resync backstop" with drift detection reconciles the incremental state — an SRE-style reconciliation loop). Its correctness is carried by equivalence tests rather than trusted: the same circuit solved compacted and uncompacted must produce matching probe readings and limit events. For a programmer: it is a lossy-compression-with-back-maps layer, like a database materialized view (or a debugger's source map over optimized code) that stays queryable at the original granularity. For an EE: it is automated series/parallel network reduction with voltage-divider interpolation to recover eliminated node voltages.
- **Project-coined term**, closest standard concept: network reduction / graph reduction (in SPICE-adjacent literature: node collapsing, series–parallel reduction; more generally node elimination, cf. Kron reduction). The word "compaction" is borrowed from sparky (the earlier Vintage Story prototype), whose incremental voxel compaction is the design ancestor. Not to be confused with GC compaction (memory management), which also appears in our docs.
- **Where it appears:** `docs/compaction.md` (whole document); `docs/design.md` R10–R11 ("Shared reduction layer ('compaction')") and System Overview; `docs/testing-strategy.md` (compacted-vs-raw equivalence tests); `docs/integration-tutorial.md` §2–3; implemented as `core/Manatee.Core/Reduction/ConductorGraph*`, contract in `docs/api.md` §19.
- **References:** Vlach & Singhal, *Computer Methods for Circuit Analysis and Design* (Van Nostrand Reinhold), for network reduction; L. Pillage, R. Rohrer & C. Visweswariah, *Electronic Circuit and System Simulation Methods* (network reduction and node elimination); Cormen, Leiserson, Rivest & Stein, *Introduction to Algorithms*, for union-find (disjoint-set) structures.

### ComponentRef / ComponentKind
- `ComponentRef` is an untyped, kind-tagged reference to any component — the runtime counterpart of the compile-time typed IDs — and `ComponentKind` is the enum tag saying which kind of component it points at.

The typed handles (`ResistorId`, `CapacitorId`, ...) make misuse a compile error, but some machinery must be generic over *all* components at runtime: limit-event attribution ("which component tripped?"), fault diagnostics, `Meta` calls like `SetLimits` and `SetDebugName`, and key-based lookup (`TryResolve`). `ComponentRef` serves those paths: a small value struct `(ComponentKind Kind, int Slot, uint Gen, ushort Net)` obtained from any typed ID via `IComponentId.AsRef()`. It is the same generational handle data plus an explicit kind tag — a tagged union of the ID types, analogous to how a lab's asset database stores "category + serial number" for any instrument, whereas each bench fixture only accepts its own instrument type. `ComponentKind` enumerates the eight primitive kinds (Resistor, VSource, ISource, Capacitor, Inductor, Switch, Diode, IdealTransformer). Erasing the type is one-way by design: you can always widen a typed ID to a `ComponentRef`, but narrowing back is the caller's problem.

**Project-coined term**, closest standard concepts: type erasure / tagged union (discriminated union); SPICE has no analogue because its element identity is just a name string.

**Where it appears:** `docs/api.md` §3 (definition, `AsRef()`), §4 (`Meta.SetLimits`, `SetThermalEnvelope`, `SetDebugName`, `TryResolve`), §12 (`LimitEvent.Source`), §11 (`FaultDiagnostic.Worst`, `DescribeFault`).

### Conductor geometry
- The raw spatial description of wiring that a game client hands to Manatee — voxels in Vintage Story, cable segments in Stationeers, schematic wires in the 2D harness — before any of it is turned into circuit nodes.

Clients describe their world "at natural granularity": every individual voxel or cable piece, with its material, cross-section, and limits, exactly as it exists in the game world. The reduction layer (ConductorGraph) then compacts this into a minimal netlist — equipotential geometry unions into single nodes, series runs collapse into single resistors — while keeping the maps needed to route measurements and limit events back to specific pieces of geometry. For an EE: conductor geometry is the physical layout drawing; the netlist is the schematic extracted from it. For a programmer: it is the client-facing input format, deliberately verbose and world-shaped, that a translation layer normalizes into the solver's compact internal representation — the same relationship as source code to compiled IR.

- **Where it appears:** `docs/compaction.md` (intro and Responsibility #1, region building), `docs/design.md` R10–R11; fed in via `ConductorGraph.AddSegment`/`BeginBulkBuild` (`docs/integration-tutorial.md` §3, api.md §19).

### ConductorGraph / SegmentKey / JunctionKey / ConductorSpec
- The reduction layer's intake surface: the client describes its cable geometry as segments between junctions, and `ConductorGraph` compacts that geometry into a minimal netlist while remembering how to map everything back.

- `ConductorGraph` is a class constructed over a public `Netlist`. The client feeds it wire geometry edge by edge: `AddSegment(segmentKey, junctionA, junctionB, spec)` says "a piece of conductor with these physical properties runs between these two junction points." `SegmentKey` and `JunctionKey` are 128-bit client-chosen identifiers (two `ulong`s, pinned in `Reduction/Keys.cs`) — the game names its own wires and connection points, like primary keys in a database, so identities survive rebuilds and reloads without the solver assigning anything. `ConductorSpec` carries each segment's physics (resistance per unit length, length, protection limits). Internally, segments become resistor edges and junctions become circuit nodes; long series chains with no taps collapse into single equivalent resistors (an EE would call this series combination; a programmer, lossless compression with an index). The map runs both ways: `PortNode(junctionKey)` returns the live node where a device attaches after compaction, `AddProbe` reads voltage inside a collapsed run by interpolation, and `Attribute` maps a limit event (e.g. an overcurrent trip) on a collapsed element back to the specific original segment that melts. The graph is deliberately never saved: geometry is the client's truth, re-driven into a fresh graph at load, and all derived identities are computed from the client's keys so handles re-resolve identically.
- **Project-coined term**, closest standard concepts: netlist extraction / circuit reduction front-end; the collapse itself is standard series-resistance combination (elementary circuit theory) with an attribution index kept alongside.
- **Where it appears:** `docs/api.md` §19 (the intake contract), `docs/compaction.md` (the reduction semantics), `docs/integration-tutorial.md` §3 (worked usage), `Manatee.Core.Reduction` namespace (`core/Manatee.Core/Reduction/`).

### ConductorSpec
- The small value struct describing one cable segment's physical properties, passed to `ConductorGraph.AddSegment`.

- It has three fields: `OhmsPerLength` (how resistive the cable material is, per unit of length), `Length` (how much of it this segment is), and `Limits` (an embedded `LimitSpec`: ampacity and thermal-melt parameters, i.e. how much current the segment tolerates before protection events fire). The segment's electrical resistance is simply `OhmsPerLength × Length`, so a game defines one spec per cable *type* and reuses it — in the tutorial, a fuse is just a segment whose spec carries a 6 A limit while the wire around it carries none. For programmers: it is a plain data record, the physics equivalent of a config object. For EEs: it is the wire-gauge table row — resistivity per metre plus current rating — expressed as a struct.
- **Where it appears:** `docs/api.md` §19 (struct definition), `docs/integration-tutorial.md` §3 (usage in `AddSegment`), `Manatee.Core.Reduction`.

### CostOfAdjust / CostOfReconfigure
- Query methods on `Netlist` that answer "how expensive would this change be?" without actually making the change.
- `CostOfAdjust(ResistorId, ohms)` returns `Metadata` (tier 0, free) if the new value falls inside the ε-no-op gate, else `Conductance` (tier 2, a matrix update). `CostOfReconfigure(CouplerId, state)` returns `Rhs` (tier 1) if the coupler is already in that state, else `Topology` (tier 3, an island merge or split — the expensive rebuild path). Because they only classify, they mutate nothing and allocate nothing, so a game's scheduler can call them mid-tick to decide whether an operation fits the remaining frame budget or should be coalesced into a background task — Re-Volt's off-band scheduler uses them together with `EditReceipt.EstimatedRebuildDim` for exactly this. For a programmer this is a dry-run/estimate API, like `EXPLAIN` before running a database query. For an EE it is like reading a component's datasheet power figure before committing it to the design: you learn the price without paying it.
- **Project-coined term**, closest standard concept: a cost-estimation or dry-run query (cf. query planners' `EXPLAIN`); no equivalent exists in SPICE-family simulators, which do not expose incremental-update costs to callers.
- **Where it appears:** `docs/api.md` §4 (signatures on `Netlist`) and §9 (scheduler usage); `docs/integration-tutorial.md` §9 ("Ask before paying") and sharp-edges appendix item 13.

### counter-torque averaging window
- The span of time over which the alternator averages its electrical load before reporting counter-torque back to the mechanical network: all electrical substeps since the previous mechanical read.
- The two simulations run at different rates — the electrical solver substeps fast, while VS's mechanical network re-sums torque only every ~100 ms. Single-phase AC power pulsates at *twice* the electrical frequency (~10 Hz at our 5 Hz starter frequency), which lands right at the mechanical cadence. If the alternator reported an instantaneous torque sample, the sampling would alias: depending on where in the power pulsation each read happens to land, the mechanical side would see a phase-dependent DC torque offset — phantom drag or, worse, free torque. So the window is normative (settled 2026-07-06): `GetResistance()` returns the mean over every electrical substep since the last mechanical read, as a pure read of a cached mean. For programmers: this is the classic aliasing problem of sampling a fast periodic signal at a slow rate, fixed by integrating over the sample interval instead of point-sampling. If the *coupling dissipativity* invariant test fails (a closed motor→shaft→alternator→motor loop must monotonically decay), this window is the prime suspect, since both integrators are individually dissipative.
- **Project-coined term**, closest standard concept: anti-aliasing by interval averaging (a boxcar/moving-average filter over the co-simulation exchange interval).
- **Where it appears:** `docs/vintage-story.md` §2 ("The averaging window is normative"); `docs/testing-strategy.md` Invariant Checks (coupling dissipativity); `docs/design.md` Simulation Model.
- **References:** Oppenheim & Schafer, *Discrete-Time Signal Processing*, for aliasing and the sampling theorem.

### counter-torque / GetResistance
- The mechanical drag an alternator pushes back onto the shaft that spins it, reported to Vintage Story's mechanical network through the engine's `GetResistance()` callback.
- Physically, generating electrical power is never free: the more current an alternator delivers, the harder it becomes to turn (Lenz's law, informally "generator loading"). In our co-simulation this is the feedback edge from the electrical solver back to the game's mechanical network: VS's contract is `IMechanicalPowerNode.GetTorque(tick, speed, out resistance)`, where consumers return a positive `resistance`, and the alternator overrides `GetResistance()` to compute that value from its current electrical load. It is re-read on every mechanical network update, so the back-coupling is live rather than a one-time parameter. For programmers: think of it as a callback the mechanical engine polls each update, and the value it returns must be the **mean over all electrical substeps since the previous mechanical read** — a cached running average, pure to read at any time on the server main thread — never an instantaneous sample (see *counter-torque averaging window* for why). For the EE: this is the standard electromagnetic braking torque of a loaded generator, exposed to the game engine through its "resistance" API.
- **Standard term:** generator counter-torque / electromagnetic braking torque (Lenz's law); `GetResistance` is Vintage Story's engine API name, not ours.
- **Where it appears:** `docs/vintage-story.md` §2 (Alternator, Mechanical Network Coupling, and the normative averaging-window ruling); `docs/design.md` (Simulation Model, mechanical co-simulation bullet).
- **References:** Fitzgerald, Kingsley & Umans, *Electric Machinery*, for generator loading and counter-torque; Vintage Story engine source `vssurvivalmod/Systems/MechanicalPower/` (the binding API contract, cited with file:line in vintage-story.md).

### coupler
- A device that bridges two islands — a breaker, a decoupling transformer, or a power converter — and the *only* legal way separate islands or client partitions connect.
- Islands are the connected components the solver works in (the unit of solving, failure, and handle invalidation, per design requirement R5); a coupler is a deliberate, managed doorway between two of them. Three kinds exist: a `Breaker` is galvanic — when Closed the two sides merge into one matrix (with a 1 mΩ closed-switch branch so through-power is measurable), when Open they are separate islands, and the opening transition is a full island rebuild; a `DecouplingTransformer` and a `ConverterTwoPort` are boundary couplers — the sides *always* stay separate islands and exchange power per substep through a relaxation-damped handshake with an energy ledger. For programmers: a coupler is the sanctioned interface between two otherwise-encapsulated modules, and the API enforces the encapsulation — under `ClientPartitioned` wiring, any ordinary edit that would bridge two partitions throws `PartitionMergeException`. For the EE: it is a two-port element placed at a subcircuit boundary; the solver refuses any other inter-subcircuit connection. Coupler handles (`CouplerId`) are document-stable: they survive every island rebuild and merge, unlike component and node handles.
- **Project-coined term**, closest standard concept: a two-port coupling device at a subcircuit/partition boundary (breaker, transformer, or converter).
- **Where it appears:** `docs/integration-tutorial.md` §1 (islands), §5–§6 (tick loop, handle rules); `docs/design.md` R5; `docs/api.md` (`CouplerSpec`, `AddCoupler`).

### coupler / coupling device
- The API-surface term for a device joining two islands: a galvanic `Breaker`, a `DecouplingTransformer`, or a `ConverterTwoPort` — under `ClientPartitioned` wiring, the only sanctioned cross-partition join.
- The three `CouplerSpec` factories differ in how the two sides relate. `Breaker()` is galvanic and bidirectional: Closed means both sides share one matrix (an incremental, cheap merge) with a 1 mΩ series branch whose signed flow feeds `Exchange(c).PowerA2B`; Open means separate islands, and *opening* is an island rebuild (tier 3, the honest cost — breakers are safety devices that change rarely). `DecouplingTransformer(...)` and `ConverterTwoPort(...)` are boundary couplers: the sides remain distinct islands forever and exchange amplitude+phase (or regulated power) once per substep, damped by an explicit relaxation α, with a per-boundary `EnergyLedger` and physical conservation mechanisms (energy-debt droop; surplus converted to `HeatDumpedJ`, never stored as work). Islands joined by boundary couplers form one scheduling unit and substep in lockstep, so the exchange is deterministic. For programmers: a breaker is a lock-free shared data structure (one matrix), a boundary coupler is message-passing between two solvers with damping to keep the handshake stable. The coupler's handle is document-stable — it outlives every rebuild of the islands it straddles.
- **Project-coined term**, closest standard concept: two-port boundary element / inter-subcircuit link in a partitioned (relaxation-based) co-simulation.
- **Where it appears:** `docs/api.md` §7 (`CouplerSpec`, `CouplerPorts`, `ExchangeView`, `EnergyLedger`), §11 (scheduling units), §16 (handle stability); `docs/solver.md` Islands.
- **References:** relaxation-based partitioned simulation is the territory of waveform-relaxation methods — Lelarasmee, Ruehli & Sangiovanni-Vincentelli (1982), "The Waveform Relaxation Method for Time-Domain Analysis of Large Scale Integrated Circuits", IEEE Trans. CAD.

### CouplerId / AddCoupler / RemoveCoupler
- The handle type and lifecycle calls for a coupler: `AddCoupler(spec, ports, key, stateKey)` returns a `CouplerId`, and `RemoveCoupler(id)` retires it.
- `CouplerId` is a small value struct (slot + generation + netlist tag) that acts like a ticket for a specific coupler. Unlike component and node handles, which die whenever their island is rebuilt, a `CouplerId` is **document-stable**: it survives every island rebuild and merge, and even a drift/journal-overflow resync never touches it — it is invalidated only by `RemoveCoupler` (or a whole-netlist reload via `FromCanonical`, after which it is re-resolved by `ExternalKey`) — so the tutorial's rule is "cache your breaker handles forever", and a breaker `Reconfigure` through a cached id is always safe at the top of the tick loop. For the EE: the handle names the device on the schematic, not its position in whatever matrix the solver currently holds. One rigidity to know: the four ports are fixed at `AddCoupler`. Re-routing a coupler (e.g. a network merge moved one of its sides) is `RemoveCoupler` + `AddCoupler` with the *same* `ExternalKey`/`StateKey`, legal in one atomic batch because removals apply before additions within a commit — the key then resolves to the new handle. A re-added galvanic breaker defaults **Closed** (re-apply Open yourself before the next Solve if it was tripped); a boundary coupler's `CouplerRuntime` scalars re-attach via same-`StateKey` snapshot/restore of the A-side island.
- **Standard term:** none — this is manatee-core API surface; the handle pattern is the classic generational index (slot + generation counter) used to detect stale references.
- **Where it appears:** `docs/api.md` (handle structs, `AddCoupler`/`RemoveCoupler`, §16–§17); `docs/integration-tutorial.md` §5–§6 and sharp-edges appendix item 17.

### CouplerRuntime
- The bundle of persistent scalar state that a boundary coupler (transformer or converter two-port) carries inside the solver core between ticks.

A boundary coupler is not memoryless: it holds a handful of evolving numbers — the converter's DC-link capacitor voltage, and the debt-droop, energy-ledger, and relaxation integrators that keep the two-island exchange stable and conservative. `CouplerRuntime` is the name for those scalars as a snapshot unit. In the snapshot format they are keyed by the coupler's client-passed `StateKey` and emitted by the coupler's A-side island (the documented anchor), so save → restore is bit-for-bit (api.md §14, Law 4). Programmers can think of it as the private fields of a stateful object that must be serialized so a reload does not cold-start it; an EE can think of it as the stored charge and control-loop integrator states of a real converter, which determine its next instant of behavior. It matters operationally when re-routing a coupler: `RemoveCoupler` + `AddCoupler` with the same `StateKey` in one batch, snapshotting the A-side island before the edit and restoring after — same-`StateKey` matching re-attaches the runtime so a settled converter resumes without a transient. (Galvanic breakers, by contrast, carry no `CouplerRuntime`; their Open/Closed state is the client's to re-apply.)
- **Project-coined term**, closest standard concept: the internal state variables (capacitor voltage plus controller integrator states) of a power-converter model.
- **Where it appears:** `docs/api.md` §14 (snapshot payload, state-unit kind 7), `docs/integration-tutorial.md` §7 and sharp-edge #17.

### coupling conservation
- The invariant that a device coupling two islands never delivers more energy to the receiving island than the source island actually produced — no free energy at boundaries.

Islands joined by a power-transfer boundary (transformer, converter) live in separate matrices and exchange values with a small lag, so a naive implementation could double-count energy during that lag. Each boundary therefore carries a running energy ledger (∫P_out − ∫P_in − modeled losses), and the invariant test asserts the ledger stays within tolerance over any window — sharpened (2026-07-05) from per-exchange clamping to windowed accounting, with property tests driving deliberately hostile oscillating loads (square waves near the relaxation constant 1/α, tick-rate toggles). Any bookkeeping surplus is dumped into the coupler's heat output rather than becoming stored work. For programmers: it is a conservation check on a distributed transaction — the two sides settle asynchronously, but audited totals must reconcile over every window. Note the audit is a windowed physical energy balance from public readbacks, not the ledger's own `Residual` (which is ~0 by construction and cannot signal a violation).
- **Project-coined term**, closest standard concept: energy conservation / passivity of a co-simulation coupling interface.
- **Where it appears:** `docs/testing-strategy.md` Invariant Checks; `docs/solver.md` Islands (energy ledger); `docs/design.md` Energy accounting rule; `docs/api.md` §7 (conservation audits).

### coupling dissipativity
- The invariant (added 2026-07-06) that a closed motor → shaft → alternator → motor loop must lose energy monotonically from any starting condition; a failure is blamed on the mechanical↔electrical coupling code, not the components.

The loop contains only lossy elements integrated by dissipative methods, so physically and numerically its total energy can only decay — like a feedback control loop built entirely from stable, damping stages: if the whole thing rings up, the fault must be in the glue between stages. Here the glue is the mech↔elec exchange, specifically the counter-torque averaging window (vintage-story.md, Mechanical Network Coupling): the electrical side substeps many times per game tick while the mechanical side ticks once, and a *sampled* rather than *averaged* counter-torque aliases the AC waveform's 2f power ripple into a spurious energy pump — hence the doc's tag "the 2f-aliasing guard." The test seeds arbitrary initial conditions and asserts monotone decay; the related dissipativity gate for island boundaries runs an exhaustive stability grid at gain-capable turns ratios, which is what motivated clamping transformer relaxation α to ≤ 0.7 (api.md §7).
- **Project-coined term**, closest standard concepts: passivity/dissipativity of a coupled numerical scheme; aliasing of the double-frequency (2f) instantaneous-power ripple under undersampling.
- **Where it appears:** `docs/testing-strategy.md` Invariant Checks; `docs/api.md` §7 (dissipativity gate, α clamp); `docs/vintage-story.md` Mechanical Network Coupling (normative averaging window).

### ctx.Previous
- The property on `DeviceTickContext` through which a device, during its per-tick update, reads the last *published* solution — the node voltages and branch currents from the previous solve.

  This is how feedback works without letting devices see half-computed state: a device like the constant-power adaptor reads last tick's voltage from `ctx.Previous`, computes its next conductance (`g = P/V²`), and stamps it via `Adjust`. The programming framing is double-buffering — devices always read a consistent, completed snapshot, never the frame being drawn. The EE framing is a one-tick-delayed feedback path around the solver. The determinism guarantee is deliberately narrow: `Previous` equals "previous tick's values" only for the device's *own* island, because a device's drive phase is scheduled before its island's step; under a parallel schedule a read of a *foreign* island may see that island's older or newer publication. During merges and Building/Dirty states, `Previous` holds last-good published values (with `IsLive == false`); a Faulted island instead reads as de-energized (zeros) while faulted, with the pre-fault vector held only internally for warm-start and re-exposed as last-good once a repairing change flips the island back to Dirty. Failed solves never publish.
- **Project-coined term** (an API member); closest standard concept: the previous-timestep solution vector used to evaluate nonlinear device models, as in SPICE's iteration on the prior operating point.
- **Where it appears:** `docs/integration-tutorial.md` §4 and §6, `docs/api.md` §18 (`DeviceTickContext`) and §21 (determinism scope).

### de-energized
- The read-side behavior where every probe of an island returns 0 V / 0 A — either because the island is genuinely dead (no sources) or because it is currently in the `Faulted` state.

When an island's solve fails (singular matrix after gmin, contradictory sources, Newton fallback ladder exhausted), the island flips to `Faulted` and the API's answer to "what's the voltage here?" becomes zeros, with `IsLive == false`. This is deliberately *status-scoped and enforced at the read path* (api.md §17.4, decision log #27): the solver still holds the pre-fault solution vector internally as a warm-start for retry, but callers can't see it while the fault stands — analogous to a cache that keeps a stale entry for recovery purposes but refuses to serve it. The moment a tier-2/3 change flips the island back to `Dirty`, reads revert to that last-good published solution. For an EE, this is the software equivalent of lockout-tagout: the panel reads dead until the fault is cleared, even though the wiring hasn't changed. The game layer decides presentation (Vintage Story: "something smells wrong"; the tablet: the actual `FaultDiagnostic`, since it's educational). The tutorial warns that a ~0 V read is therefore ambiguous — check `IsLive`/status before concluding a circuit is off.

**Project usage note:** "de-energized" is ordinary electrical vocabulary (a conductor carrying no voltage), but Manatee uses it specifically for this *read contract*, including the fault-masking case where internal state is nonzero.
- **Where it appears:** `docs/solver.md` Failure Handling; `docs/integration-tutorial.md` §6 and §8 (and the sharp-edges list); `docs/api.md` §11 island state machine, §17.4, §20 failure table, decision log #27; `Solution.IsLive`.

### DebugLevel / DebugName
- Manatee's two knobs for diagnosability: `NetlistOptions.Debug` (an enum: `Off | Asserts | AllocationSentinel`) selects how aggressively the solver self-checks, and `Meta.SetDebugName` attaches a human-readable label to a component in debug builds.

`DebugLevel` is fixed at netlist construction and escalates the strength of internal tripwires: `Off` runs lean; `Asserts` enables debug assertions such as "no NaN/non-finite value anywhere in the solution vector" and "no structural rebuild inside the steady-state region"; `AllocationSentinel` additionally arms a per-tick allocation tripwire that captures `GC.GetAllocatedBytesForCurrentThread()` and asserts the delta is zero — proof that the hot loop honored its zero-heap-allocation contract (where the runtime supports the counter reliably; Unity/Mono may not). For an EE, these are the software analog of instrumenting a prototype with extra test points and a current clamp: they change nothing about behavior, only how loudly deviations are caught. `SetDebugName` is the complementary human-facing half — it tags a component handle with a name ("kitchen lamp") so fault diagnostics and logs speak in the client's vocabulary; it exists only in debug builds and is allowed to allocate, so it is explicitly barred from the steady-state tick.

**Project-coined term** (API surface names), closest standard concepts: assertion levels / debug build configuration, plus a memory-allocation tripwire; `SetDebugName` parallels the debug-name facilities in graphics APIs (e.g. Vulkan/D3D object labels).
- **Where it appears:** `docs/api.md` §4 (`Meta.SetDebugName`, cold-path table §22), §5 (`NetlistOptions.Debug`), §8 (`AllocationSentinel` mechanics and reliability caveat).

### Decoupling boundary
- The seam between two simulation islands joined by a decoupling coupler (e.g. a decoupling transformer), across which only summary values — amplitude and phase — are exchanged, not the full circuit equations.

Because the two sides live in separate matrices and communicate through a narrow per-substep message (amplitude + phase, damped by relaxation), the boundary behaves like an API between two services rather than shared memory: each side sees only the other's reported interface values, with bounded lag. The educational consequence, and the reason curriculum.md calls it out, is that this exchange "launders" reactive power — the subtle phase relationships between voltage and current that produce power-factor and impedance effects are flattened into the summary values, so they cannot be faithfully observed *across* the boundary. Curriculum authoring rule: any lesson teaching reactive behavior (power factor, impedance) must keep its whole circuit in one island, using idealized (same-matrix) transformers.

**Project-coined term**, closest standard concept: a subcircuit/partition boundary in a relaxation-based (partitioned) circuit simulation — the interface between subsystems in co-simulation.

- **Where it appears:** `curriculum.md` Authoring Rules; the underlying mechanism is `design.md` Simulation Model (island coupling) and `solver.md` Islands.
- **References:** Lelarasmee, Ruehli & Sangiovanni-Vincentelli (1982), "The Waveform Relaxation Method for Time-Domain Analysis of Large Scale Integrated Circuits", IEEE Trans. CAD — the classic treatment of partitioning a circuit and iterating on interface waveforms, of which this is a deliberately lossy, game-rate cousin.

### Decoupling transformer
- One of Manatee's two transformer device classes: the utility-scale (pole-to-network) transformer, modeled not as circuit elements inside one matrix but as a *boundary between two islands* that exchanges amplitude and phase each substep.

Manatee splits transformers by scale (an Electrical Age precedent, settled 2026-07-05). *Idealized* transformers (small, local, and all tablet lessons) are real two-ports inside a single matrix — turns ratio enforced via auxiliary MNA rows, fully faithful to reactive behavior. *Decoupling* transformers instead keep the two windings in separate islands (separate matrices, separately factorized, separately threadable) and couple them by exchanging summary values (amplitude + phase) per substep, damped by explicit relaxation (`relaxationAlpha`, default 0.5) for stability; transfer is clamped to what the source island actually delivered, so no free energy. For a programmer: it converts a shared-memory problem into message passing, at the cost of one substep of lag and loss of exact phase fidelity. Gameplay steers long-distance power transfer toward decoupling types — precisely where per-island thread isolation pays. In the API it is a coupler (`CouplerSpec.DecouplingTransformer`, addressed by `CouplerId`), not a netlist device.

**Project-coined term.** In EE literature "isolation transformer" means galvanic isolation for safety, which is *not* what we mean; the closest standard concept is a partition/co-simulation interface between relaxation-coupled subcircuits that happens to be physically a transformer. An outside EE should read "decoupling" as *simulation* decoupling, not electrical decoupling capacitors.

- **Where it appears:** `design.md` Simulation Model (island coupling); `solver.md` Component Set and Islands; `api.md` §7 (couplers) and §18; `testing-strategy.md` (boundary equivalence tests).
- **References:** Lelarasmee, Ruehli & Sangiovanni-Vincentelli (1982), "The Waveform Relaxation Method for Time-Domain Analysis of Large Scale Integrated Circuits", IEEE Trans. CAD, for the relaxation-coupled-partitions idea; Ho, Ruehli & Brennan (1975), "The Modified Nodal Approach to Network Analysis", IEEE Trans. Circuits and Systems, for the auxiliary-row two-port used by the *idealized* sibling.

### DeferredStructuralOps
- A per-tick diagnostic counter in `TickStats` recording how many structural (tier-3) operations were attempted inside the steady-state guard in a release build and therefore deferred to the guard's `Dispose`.

The steady-state guard (`EnterSteadyState`) marks the region of a tick where the world topology must not change — like a lockout tag on a panel while someone is working on it. If code nevertheless issues a `Reconfigure` (e.g. a breaker open/close) inside that region, debug builds assert immediately; release builds queue the op in a netlist-owned pre-allocated ring and execute it when the guard closes — same tick, before `Solve` — then count it here. Deferred ops are never dropped, and the same-tick bound guarantees a breaker the game believes is open never leaves an island energized across a tick. A nonzero value is itself a bug signal: it means client code violated the "nobody rebuilds the world inside the device loop" contract, even though the solver recovered gracefully. (`Edit()` is the exception — it throws in all build modes, since a live structural-edit scope cannot be coherently deferred.)

- **Where it appears:** `api.md` §8 (guard semantics, deferred-op storage) and §9 (`TickStats`); `integration-tutorial.md` §5, step 3 of the tick loop.

### Delayed-burn behavior
- Re-Volt's existing gameplay behavior in Stationeers where an overloaded fuse or cable burns out after a period of sustained overload rather than the instant a limit is crossed.

This matches how real fuses work: destruction depends on accumulated heating, not instantaneous current, so a brief surge is survivable while a sustained moderate overload is not. Manatee reproduces this *by mechanism rather than by special case*: fuses and breaker trip curves ride manatee-core's ordinary limit events, including the i²t thermal accumulator — a per-device running integral of current-squared over time (a leaky-bucket rate limiter, in programming terms) that trips when it exceeds the device's rating. Ratings are ambient-temperature-adjusted through the compaction layer's limit envelope, so Europa's −150 °C and Vulcan's +800 °C move the real thresholds instead of being hard-coded exceptions. The design point is that legacy behavior falls out of the physics model, so the integration carries no bespoke burn-out logic to keep in sync.

The underlying quantity is the standard fuse-engineering **I²t (Joule integral / melting integral)**; "delayed-burn behavior" is just our name for the observed Re-Volt behavior being preserved.

- **Where it appears:** `stationeers.md` Graph Construction (fuses/trip curves paragraph); mechanism in `compaction.md` (limit envelope) and the solver's limit events.
- **References:** Horowitz & Hill, *The Art of Electronics*, on fuse ratings and time–current behavior; fuse I²t characteristics are standardized in IEC 60269 (low-voltage fuses).

### Desktop Shell
- `Manatee.Harness`: the desktop application (built on Avalonia, a cross-platform C# UI toolkit) that hosts the schematic editor's drawing canvas plus authoring conveniences — deliberately kept *outside* the tested tablet stack.

The 2D harness is layered so that everything that matters — the circuit document, the interaction state machine, the display-list view — is pure, engine-free code covered by headless tests. The Desktop Shell is the thin outermost wrapper: an Avalonia window hosting the Skia canvas, a lesson-text pane, a component palette, and probe readouts, used for authoring lessons and poking at circuits on a PC. It is swappable by design — nothing in `Manatee.Schematic` may reference Avalonia — the way a lab instrument's front panel can be replaced without touching the measurement electronics inside. Shell-level end-to-end tests (via Avalonia.Headless) are possible but expected to stay rare; the layers below carry the coverage.

**Project-coined term**, closest standard concept: the "shell"/host layer in a Hexagonal (Ports-and-Adapters) or "functional core, imperative shell" architecture.

**Where it appears:** `docs/harness.md` sec. "The Desktop Shell"; the `Manatee.Harness` project in the solution.

### Device / DeviceHost / TerminalSpec
- The three pieces of manatee-core's devices layer: `Device` is the base class a multi-part electrical model derives from, `TerminalSpec` declares its external connection points, and `DeviceHost` is the built-in driver that builds and ticks devices.

A `Device` (e.g. `Battery` = EMF source + internal resistance + state-of-charge integrator) packages several solver primitives behind one object. It has exactly two lifecycle moments, separated by cost tier: `Build` runs once, at tier-3 (structural) time, minting each internal component's key deterministically as `baseKey.Derive(ordinal)` — like assigning fixed pin numbers on a part, so the same device rebuilt later gets the same identities and save/restore-by-key works. `Tick` runs every timestep and receives only a `DeviceTickContext`, whose API is capped at tiers 1–2 *by type*: the context exposes only `Drive`/`Adjust` overloads, so a device author physically cannot write a structural change into the hot loop — it does not compile. Think of it as a hardware interlock expressed in the type system. `TerminalSpec` states how many terminals the device has and which are `ReturnTerminals` — the ones the wiring policy automatically binds to the partition's reference rail, so single-terminal game devices need no hand-written grounding. `DeviceHost` is the convenience driver: one `Add` call opens a single edit, builds, and registers state; one `Tick` call ticks every enlisted device in order. Crucially, it does **not** implement the re-pin contract, so it is only safe on islands whose topology never changes (see the DeviceHost entry).

**Project-coined term**, closest standard concepts: a SPICE *subcircuit* (`.SUBCKT`) for Device/TerminalSpec — a named block with a fixed terminal list expanding into primitives — and a behavioral-model evaluation loop for DeviceHost. The tier-capped `Tick` has no direct SPICE analogue; it is closest to updating companion-model parameters between timesteps without altering the matrix structure.

**Where it appears:** `docs/api.md` §18 (Devices-layer contract: the full `Device`, `TerminalSpec`, `DeviceTickContext`, and `DeviceHost` surface), §5 (wiring policy and ReturnTerminals), tutorial §4.

**References:** Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520 (subcircuits, companion models); Gamma et al., *Design Patterns* (1994) — Composite and Template Method are the nearest patterns for Device, Facade for DeviceHost.

### Device port identity / recorded ports
- An agreed addition to Re-Volt's network export that records *which terminal* of a device each cable connection lands on, so Manatee can attach the device to the electrically correct node.

Sukasa's `NetworkExport` format gives Manatee the cable-segment adjacency plus device and fuse lists. For a single-terminal vanilla device that is enough, but a multi-terminal device (a transformer, a breaker straddling two networks) is electrically meaningless without knowing which connection is which port — the same way a schematic symbol is useless if you don't know which pin is which. "Device port identity" is that per-connection annotation in the export; "devices attach at their recorded ports" is the graph-construction rule that consumes it: when segments become resistor edges and junctions become nodes, each device terminal is wired to the node its recorded port says, rather than guessed from proximity. For a programmer, it is a foreign key from a connection record to a specific field of the device, replacing an ambiguous "connected to device X" edge; for an EE, it is simply pin assignment carried through the netlist export.

**Project-coined phrasing**, closest standard concept: pin/port mapping in a netlist (the terminal-order list of a SPICE `.SUBCKT` instantiation, or pin numbers in EDA netlist formats).

**Where it appears:** `docs/stationeers.md` (Graph Construction: the agreed `NetworkExport` additions and the "devices attach at their recorded ports" pipeline rule).

### Device wants / demand-supply
- The power amounts vanilla Stationeers devices report each tick — how much a load *wants* to draw and how much a generator says it can *supply* — which the legacy-device adaptor converts into circuit stamps.

Vanilla Stationeers has no circuit solver: each game tick, its power system asks every device for numbers through a four-call API (available/needed, provide/draw; joules per half-second). Re-Volt intercepts that tick, and during the `CalculateState` phase Manatee's adaptor gathers these "wants" and translates them into solver terms: a load wanting P watts is stamped as a conductance G = P/V_prev² (constant-power behavior linearized across game ticks — see the legacy-device adaptor); a generator advertising available power is stamped as a voltage source with an internal resistance and a current limit derived from what it advertised. After the MNA solve, the adaptor reports *actuals* — what the circuit really delivered — back through the same vanilla calls, so vanilla bookkeeping stays intact and the mod remains removable. Programming framing: wants are a request/quota API, and the solver is the arbiter that turns requests into grants under physical law. EE framing: the wants are per-device power setpoints that the adaptor converts into companion-model element values, updated once per tick from the previous operating point.

**Project phrasing** (borrowed from the game's own API vocabulary); closest standard concepts: constant-power (P-type) load models and their linearization about the previous operating point, as in power-flow analysis.

**Where it appears:** `docs/stationeers.md` (Integration Seams: `CalculateState_New` "gather demand/supply → adaptor updates stamps from device wants"; The Legacy-Device Adaptor); the built-in `AdaptedLoad`/`AdaptedSource` models in `docs/api.md` §18.

### DeviceHost
- manatee-core's built-in, single-threaded driver that builds stock devices onto a netlist and ticks them each timestep — safe only on islands whose wiring never changes.

`DeviceHost(Netlist)` is a pure public-API consumer: its `Add<T>(device, terminals, baseKey, stateKey)` opens exactly one `Edit`, calls the device's `Build` (which mints component keys deterministically as `baseKey.Derive(ordinal)`), registers the device's save-state unit anchored on terminal 0 when it has one, and enlists the device; `Tick(dt)` then ticks every enlisted device in `Add` order through a tier-≤2 `DeviceTickContext`. The sharp edge is documented deliberately: devices cache their component *and terminal-node* handles privately at `Build`, and `DeviceHost` implements no re-pin — it cannot re-resolve caller-owned terminal nodes from the device's own key. So after a co-island rebuild it keeps ticking through stale handles: by the stale-handle sentinel this is a counted no-op (`StaleHandleReads > 0`), never a crash, but the device silently stops affecting the circuit. Hence the rule: `DeviceHost` is complete only for **static-topology** islands (the tutorial's "quiet network"); anything on a churning cable graph must instead be a key-re-resolving adaptor that owns its own re-pin loop (the `RevoltAdaptor`/`TutorialLoad` pattern). EE analogy: a bench fixture with soldered leads — perfect while the board never changes, wrong the moment someone reworks the traces under it, at which point you need probes you can re-land (keys) rather than joints.

**Project-coined term**; closest standard concept: a serial device-evaluation/scheduler loop in a simulator, roughly the per-timestep model-evaluation pass of a SPICE engine restricted to parameter updates.

**Where it appears:** `docs/api.md` §18 (contract, including the explicit "does NOT implement the re-pin contract" caveat) and §22 walkthroughs; `docs/integration-tutorial.md` §2 (construction) and §4 (built-in devices on static islands vs adaptors on churning islands).

### devices layer
- The layer of manatee-core that holds behavioral device models — a device is a small bundle of solver primitives (resistors, sources, etc.) plus an optional per-tick update — sitting above the netlist and below the game clients.

In the manatee-core layering (`solver` → `netlist` → `devices` → clients), a device is defined by its terminals, a `Build` method that creates its internal components on the netlist, a matching tear-down, and an optional `Tick()` for behavioral parameter updates (battery discharge/SoC curves, alternator rotor/EMF dynamics, constant-power adaptor loads). (Converter two-ports, by contrast, are realized as couplers in the as-built API, not as Tick-driven devices.) For an EE: a device is a subcircuit with behavior attached. For a programmer: it is a facade over a handful of typed netlist handles, with the key guarantee that device authors never touch matrices — everything they can do goes through the typed netlist verbs, and their per-tick powers are further capped by `DeviceTickContext`. Built-in models include `Battery`, `Alternator`, `AdaptedLoad`, `AdaptedSource`, and `IdealizedTransformer`.

**Standard term:** closest concepts are SPICE *subcircuits* (`.subckt`) and *behavioral models*; our devices additionally carry imperative per-tick update code, which SPICE subcircuits do not.
- **Where it appears:** `docs/solver.md` (Layering, The Netlist API, Component Set); `docs/api.md` §18 (`Manatee.Core.Devices`: `Device`, `DeviceHost`, built-in models).
- **References:** ngspice manual (subcircuits, `.subckt`); Vlach & Singhal, *Computer Methods for Circuit Analysis and Design* (macromodeling).

### DeviceTickContext
- The capability object handed to a device's per-tick update (`Device.Tick`), exposing the previous solution, the timestep, and only the cheap tier-1/2 mutation verbs — so a device physically cannot make an expensive structural change from the hot loop.

It is a C# `readonly ref struct`: a zero-size, stack-only value that cannot be stored or escape the tick (the compiler enforces this). It exposes `Previous` (last published solution — deterministic for the device's own island, scheduling-dependent for foreign islands under a parallel schedule), `Dt`, and the full tier-1/2 verb set: `Drive` overloads (change a source value — right-hand-side only) and `Adjust` overloads for every adjustable primitive (resistor, switch, capacitor, inductor, diode, transformer turns ratio). What is *absent* is the point: there is no add/remove verb, so writing a topology change inside `Tick` is a compile error, not a runtime check. For an EE: it is like an instrument panel that offers only knobs, no soldering iron — you can retune the circuit each tick but not rewire it. For a programmer: it is the capability-security pattern applied to the change-cost tier system, "cannot exceed its tier by type."

**Project-coined term**, closest standard concepts: the capability-object pattern plus a context/parameter object — a narrowed facade in the sense of the GoF Facade pattern, with the narrowing itself in the spirit of Robert C. Martin's Interface Segregation Principle; there is no SPICE analog because SPICE models are declarative.
- **Where it appears:** `docs/api.md` §18 (definition) and §4 (`Netlist.TickContext(dt)`); `docs/integration-tutorial.md` §4.
- **References:** Gamma, Helm, Johnson & Vlissides, *Design Patterns* (Facade pattern); Robert C. Martin, *Agile Software Development: Principles, Patterns, and Practices* (Interface Segregation Principle).

### DeviceTickContext (ctx)
- `ctx` is the conventional variable name, used throughout the integration tutorial, for the `DeviceTickContext` passed into a device's per-tick body.

The tutorial's rule is "what Tick may do is capped by type": inside the drive phase of the tick loop, a device receives `ctx` and may call only `Drive` and `Adjust` overloads plus read `ctx.Previous` — a structural change written inside the device loop does not compile. The canonical usage is the constant-power load's linearization: read last tick's voltage from `ctx.Previous`, compute a conductance from it, and `ctx.Adjust(...)` the resistor — where repeated adjusts within `AdjustEpsilon` degrade to free no-ops once the circuit converges. For an EE: `ctx` is the once-per-timestep window in which a behavioral model may retune its element values based on the previous operating point. For a programmer: it is the same object as `DeviceTickContext` (see that entry); this entry exists because the tutorial consistently names it `ctx` (`var ctx = _net.TickContext(Dt)`).
- **Where it appears:** `docs/integration-tutorial.md` §4 (capped Tick, R18 adaptor body) and §5 (tick loop, drive phase under `EnterSteadyState`); backed by `docs/api.md` §18.

### diagnostic (event)
- The machine-readable failure detail the solver attaches when an island fails to solve, naming the specific offending nodes and components so the game client can show a real explanation instead of a generic error.

When an island's matrix is singular even after gmin (e.g. contradictory ideal voltage sources in parallel), or Newton iteration exhausts its fallback ladder, the island enters the `Faulted` state and a diagnostic is reported as an event. In the as-built API this is the `FaultDiagnostic` struct (fault kind, worst component, worst node, counts — richer detail drained via `DescribeFault` on the cold path). The design contract: no NaN ever leaves the API, the previous solution is held internally for warm-restart, and the Faulted island reads as de-energized while the client decides presentation — Vintage Story says "something smells wrong," while the educational tablet renders the actual diagnosis. For a programmer: it is a structured error object rather than a boolean failure flag, analogous to a compiler error with file-and-line versus "build failed." For an EE: it is the solver telling you *which* branch made the system indeterminate, like a short-finder rather than a blown fuse.

**Standard term:** closest analog is a SPICE error report on singular-matrix or convergence failure (e.g. "singular matrix" naming a node); ours is a typed, per-island event consumed programmatically rather than printed text.
- **Where it appears:** `docs/solver.md` (Failure Handling); `docs/api.md` §17 failure table and `FaultDiagnostic` / `DescribeFault`.
- **References:** ngspice manual (convergence and singular-matrix error reporting); Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520 (origin of the convergence-failure handling tradition).

### Diff/Resync
- The as-built name for the two-part mechanism that keeps Manatee's incremental compaction honest: a *recompaction diff* that applies only what changed, plus a *resync backstop* that rebuilds from scratch and compares.

When conductor geometry changes (a voxel placed, a cable cut), `ConductorGraph` stages the change against shadow geometry, recompacts the affected area, and diffs the result against the currently realized netlist — so the solver receives a minimal set of edits rather than a whole new circuit. Because incremental maintenance against a live game mutation stream will inevitably have edge-case bugs, the resync backstop keeps the cheap from-scratch build available at all times: a validation mode rebuilds in the background, diffs against the incrementally-maintained state, and logs any discrepancy. For an EE, this is like continuously cross-checking an adjusted running total against a full re-measurement; for a programmer, it is incremental computation guarded by a rebuild-and-compare invariant check (run continuously in CI and as a debug config in-game), turning silent state corruption into bug reports.
- **Project-coined term**, closest standard concepts: incremental recomputation with a shadow copy, validated by a rebuild-and-compare ("golden rebuild") consistency check.
- **Where it appears:** `docs/compaction.md` (status header and "Incremental Maintenance" / "Resync backstop"), `docs/testing-strategy.md` (incremental equivalence), implemented in `core/Manatee.Core/Reduction/ConductorGraph*` with `docs/api.md` §19 as the as-built contract.

### dissipation / heat is universal
- Manatee's design rule that *every* device dumping power as loss must actually emit that heat into the game world, making overheating an emergent design constraint rather than a scripted event.

In most games, only designated "heater" items produce warmth and only designated "overheatable" items can burn out. Manatee instead treats heat as an invariant of the energy ledger: any device that dissipates power publishes that dissipation to a mod-owned, position-indexed **heat registry**, and downstream systems (room warmth, food perish rates, player comfort, overheat damage) read from it. A space heater is then just "the honest version" of every other device — a resistor whose heat output is the product. For an EE this is simply enforced conservation of energy; for a programmer it is a single source of truth (the registry) with all consumers reading the same accounting, so nothing can dissipate power without a thermal consequence. In Stationeers the dumped heat lands in the game's real atmospheric thermal simulation.
- **Project-coined term** (design rule), closest standard concept: thermal design / power-dissipation budgeting, driven by conservation of energy.
- **Where it appears:** `docs/design.md` (device notes, "Heat is universal"; energy-accounting rule settled 2026-07-05), `docs/vintage-story.md` sec 3 (Rooms and Heat: heat registry, `IHeatSource`).

### drift / drift detection / DriftReport
The R11 reconciliation backstop: detecting that Manatee's incrementally-maintained circuit topology has silently diverged from the truth, logging exactly what differs, and repairing it — turning silent corruption into an actionable bug report with coordinates.

Manatee maintains the compacted conductor graph *incrementally* (patching state as cables are added or removed) because a full rebuild every tick would be wasteful. But incremental state fed by a live game mutation stream will eventually have edge-case bugs. The safeguard is a two-level scheme so the common case costs almost nothing (api.md §14 law 5): every N ticks the engine computes `Fingerprint(island, Structural)` — a cheap hash of the incrementally-maintained topology. Only on a mismatch does it run the expensive path: a from-scratch rebuild of the same island (always cheap and available), serializing both the live state and the fresh rebuild via `SaveNormalized` (a key-sorted canonical form, so equal circuits compare byte-equal) and diffing them. The result is a `DriftReport` whose entries each carry a `DriftKind` (e.g. `ValueMismatch`, `MissingInLive`, `PartitionMismatch`), the `ExternalKey` of the offending element (the client's own name for it — a block position, a cable id), and the live vs. shadow values. A drift report is followed by a `Resync`, which replaces the corrupted incremental state with the rebuild. This runs continuously in CI and ships as a debug config in-game. For a programmer, this is standard cache reconciliation: the incremental state is a cache of the world geometry, the rebuild is the source of truth, and the diff catches stale entries. For an EE: fingerprint = a quick checksum on the netlist; diff = element-by-element comparison against a netlist re-extracted from the physical schematic; resync = throw away the annotated copy and start from the re-extraction.

**Standard term:** drift detection / reconciliation diff (an SRE and distributed-systems idiom, e.g. anti-entropy repair); the `DriftReport`/`DriftKind` API is project-coined. Closest concepts: state reconciliation, anti-entropy, checksum-then-diff validation.

**Where it appears:** `docs/compaction.md` (Incremental Maintenance, "Resync backstop"); `docs/design.md` (R11, Testing and Validation); realized as `ConductorGraph.Diff` → `DriftReport` in `docs/api.md` §19 (`DriftReport`, `DriftEntry`, `DriftKind`, `Resync`), §14 executable law 5.

### Drive
Manatee's tier-1 mutation verb: change what a source is pushing into the circuit (a voltage or current setpoint) — the cheapest possible change, safe to call every tick for every device.

`Drive(VSourceId, volts)`, `Drive(ISourceId, amps)`, and `Drive(VSourceId, in SineDrive)` change only the right-hand side of the already-factorized system of equations: the matrix (which encodes topology and component values) is untouched, so the solver reuses its cached LU factorization and only re-runs back-substitution. In cost-tier terms this is tier 1 of the four-verb hierarchy `Drive` (T1) < `Adjust` (T2) < `Reconfigure` < `Edit`; the tutorial's tick loop calls `Drive` under the steady-state guard, where tier-3 changes are deliberately impossible. Programming analogy: `Drive` is updating a value in a prepared query's parameters versus recompiling the query plan. EE mapping: it is exactly changing a source's value in the SPICE right-hand-side vector b of Ax=b — A and its factorization stay valid, so only the forward/back-substitution cost is paid.

**Project-coined term**, closest standard concept: source value update (RHS-only update reusing a cached matrix factorization).

**Where it appears:** `docs/api.md` §4 (verb taxonomy) and the `Netlist`/`DeviceTickContext` surfaces (§18); `docs/integration-tutorial.md` §1 (tier table), §4–5 (drive phase of the tick loop); `docs/solver.md` change-cost tiers.

**References:** the LU-factorize-once, back-substitute-per-solve structure is standard MNA practice — Ho, Ruehli & Brennan (1975), "The Modified Nodal Approach to Network Analysis", IEEE Trans. Circuits and Systems; Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods*.

### dual occupancy / shared blocks
- A Vintage Story arrangement where one world position contains both vanilla chiseled stone **and** our cable voxels, by making the cable material just another material inside the same chiseled-block entity.

Vintage Story allows only one block entity per world position, so two independent blocks cannot share a spot. The workaround: the vanilla microblock (chiseled block) entity already stores a list of materials and the voxel cuboids made of each, and it accepts *any* block id as a material. Cable voxels therefore live as microblock voxels of cable materials inside the vanilla `BlockEntityChisel`; a mod behavior attached by JSON patch rescans the voxel data whenever it changes and maintains the electrical graph from it. For an EE, think of it as running two circuits through one shared junction box rather than mounting two boxes on the same stud — the box's own parts list is extended to include our wire. This is representation #1 of three in the VS architecture (alongside pure-cable blocks and surface wiring), and it is specifically what enables the oven: wire laid in grooves chiseled into stone, in the same block. Cable voxels are deliberately *not* protected from the player's chisel — chiseling out a live conductor is a hazard by design.

**Project-coined term**, closest standard concept: none — this is a Vintage Story engine-specific arrangement (multiple logical block types cohabiting one microblock block entity).

**Where it appears:** `docs/vintage-story.md` §1 (Chosen architecture, item 1; microblock engine facts with file:line references).

### dump type
- The first token of an element line in a Falstad circuit file — a type code that identifies which kind of component the line describes and therefore how to parse the rest of it.

A dump type is either a single letter (`r` = resistor, `c` = capacitor, `v` = voltage source, …) or, when the first character is a digit, the whole token parsed as a decimal integer (`172`, `403`, …) — dispatch is on that first character. It acts as a discriminated-union tag: the loader looks it up to pick the constructor, and that constructor then reads the remaining tokens positionally (`TYPE x1 y1 x2 y2 flags [params...]`). For an EE: it is the reference-designator convention of a netlist (R for resistor, C for capacitor) hardened into a machine dispatch key. Upstream silently skips lines with unrecognized dump types; our importer's contract instead rejects loudly — with line number and reason — every unsupported dump type, except presentation-only ones (scope lines, hints, sliders), which it ignores with a counted, logged notice rather than skipping silently. Silent skipping is how dialect drift starts.

**Project/circuitjs1 term**, closest standard concept: element type tag / type discriminator (compare SPICE's first-letter element convention, e.g. `R`, `C`, `L` in a netlist card).

**Where it appears:** `docs/falstad-format.md` §1 (dispatch, error posture), §3 (element line general form), §4 (authoritative per-type table), §7 (accept/reject contract).

### duty / electrical duty
- How hard a device is actually being driven electrically, as a fraction of what it wants — its effective load level, used by game devices to scale their in-world effect.

In the Vintage Story plan, the freezer hooks the game's perish-rate delegate and returns a spoilage rate scaled by its electrical duty: fully powered means full perish-rate reduction, a browned-out freezer preserves food proportionally worse, an unpowered one not at all. The value comes from the solver side (the power the device is actually receiving versus its rating), so game behavior degrades continuously with electrical conditions instead of flipping between on and off. For programmers: think of it as a utilization metric, like CPU load — a normalized 0-to-1 signal that downstream systems consume. For EEs: it is closest to a capacity factor or utilization factor — the delivered-versus-rated power ratio — not the switching duty cycle of PWM.

**Standard term:** closest standard concepts are *capacity factor* / *utilization factor* (delivered power as a fraction of rated power); we use "duty" loosely for the instantaneous ratio, not a switching waveform's on-time fraction.
- **Where it appears:** `docs/vintage-story.md` §3 (Freezer: perish rate "scaled by its electrical duty").

### EA dialect
- Electrical Age's variant of the Falstad classic-text circuit format: the subset its four lessons use, plus a `#` comment-line extension, plus several element reinterpretations of its own.

The dialect is defined by observation of EA's parser (`FalstadNetlist.kt`): it accepts the standard classic-text elements EA's corpus needs (`v`, `w`, `r`, `c`, `l`, `s`, `g`, `O` and a `$` header) and adds `#`-prefixed comment lines carrying the teaching narrative — upstream Falstad has no comment syntax at all. But it also silently *reinterprets* elements: `p` becomes a push switch (upstream: voltmeter), SPDT `S` is substituted with two complementary switches, zener `z` degrades to a plain diode, and lowercasing element codes conflates scope-config `o` lines with `O` probes. Manatee's importer adopts exactly one piece of the dialect — the `#` comment lines, because lessons live in them — and rejects the reinterpretations loudly rather than inheriting them; a Manatee lesson file must also load unmodified in upstream falstad.com/circuitjs. For programmers: a dialect in the file-format sense, like a CSV variant with a nonstandard quoting rule — you accept the useful extension and refuse the semantic drift.

**Project-coined term**, closest standard concept: a file-format dialect/profile of the Falstad (circuitjs1) classic text format.
- **Where it appears:** `docs/falstad-format.md` §6 (dialect description) and §7 (Manatee's accept/reject contract); design requirement R20.

### EA / Electrical Age
- The Minecraft mod that pioneered running a real circuit simulator (MNA) inside a game; Manatee is its self-declared spiritual successor.

Electrical Age proved the core bet — that a game can be built on genuine nodal circuit analysis rather than scripted "power numbers," where voltage drop, resistance, and failure are real consequences and the game is hard *because* real electricity is (the property design.md names as what made EA unique and what Manatee preserves as its core requirement). The user is an ex-EA developer, so EA also serves as shared vocabulary. Manatee inherits several concrete practices from it: the `docs/examples` lesson-corpus format (a Falstad-format netlist plus a Markdown teaching narrative per directory) and its examples-README teaching structure, which shaped the lesson corpus and its narrative style; the "EA trick" of choosing component values so time constants sit at human-visible scales (0.1–10 s) where Backward Euler at a ~50 ms game tick stays accurate ("now documented in EA's own examples README", design.md:277-279); lightweight low-voltage signal components alongside the power simulation (EA's low-voltage components were skins over the same solver — the precedent behind the elevator-control progression arc's relay-logic controllers); and the precedent of a separate *idealized* transformer class for teaching alongside the physical one. A current checkout lives at `third_party/ElectricalAge/` as a reference; its importer source (`FalstadNetlist.kt`) is also our documentation of the EA dialect (see that entry). For EEs: EA is prior art, not a textbook — where its electrical interpretations diverge from standard practice, we document and reject the divergence rather than copy it.

### Edit()
- The tier-3 verb in manatee-core's four-verb cost grammar: open a structural batch that, on commit, atomically applies topology changes and marks the affected islands for rebuild.

The four verbs grade every mutation by cost — `Drive` (tier 1, back-substitution only), `Adjust` (tier 2, numeric refactor), `Reconfigure` (tier 3-lite, coupler membership), and `Edit()` (tier 3, full structural batch with island rebuilds). `Edit()` is meant for construction and for cuts (placing or destroying blocks/cables), never for the steady-state game tick: grepping a client for `.Edit(` finds every site that can trigger a rebuild, which is the intended audit mechanism. Guard rails are deliberate: a nested `Edit()` throws, and `Edit()` inside the steady-state guard throws in *all* build modes because a live edit scope cannot be coherently deferred — structural reactions belong in the topology or readback phase of the tick. For EEs: this is the "power down and rewire" operation, priced accordingly, versus adjusting a knob mid-run. For programmers: the expensive code path made syntactically loud — like a schema migration versus a row update.

**Project-coined term**, closest standard concept: batched netlist mutation triggering re-setup/refactorization of the system matrices.
- **Where it appears:** `docs/integration-tutorial.md` §1 (verb table), §4, §8, sharp-edges appendix items 7 and 15; `docs/api.md` §4 (declaration, `[CostTier(3)]`) and §6 (the returned scope).

### Edit / StructuralEdit
- The only way to change a circuit's topology in manatee-core: a batch scope in which you add or remove nodes and components, which then applies all-or-nothing.

`StructuralEdit` is a pooled sealed class rented from the netlist via `Edit()` and returned on `Dispose`; it exposes every `Add*` (resistor, sources, capacitor, inductor, switch, diode, sine source, ideal transformer, coupler, probes) and `Remove*` method. Commit is atomic: `Dispose` applies the whole staged batch or, after `Abort()`, none of it — a half-applied edit can never be observed. Commit returns an `EditReceipt` (journal window, counts, rebuild-size hints) so derived layers can sync or budget the coming island rebuild. It is deliberately "loud": nesting throws, calling `Edit()` inside the steady-state guard throws in all build modes, and a `RemoveNode` of a still-connected node aborts the whole batch. For EEs: it is the netlist editor — the moment you rewire the breadboard, as opposed to turning knobs on existing parts (`Drive`/`Adjust`). For programmers: a transaction object — begin, stage mutations, commit-or-rollback — implemented as an `IDisposable` scope so the type system marks every expensive topology-change site.

**Project-coined term**, closest standard concept: netlist construction/modification — SPICE-family tools rebuild from a netlist file; an incremental, transactional in-memory mutation batch is our addition.
- **Where it appears:** `docs/api.md` §4 (the `Edit()` verb) and §6 (the full `StructuralEdit` surface and `EditReceipt`); `docs/integration-tutorial.md` throughout.
- **References:** Ho, Ruehli & Brennan (1975), "The Modified Nodal Approach to Network Analysis", IEEE Trans. Circuits and Systems (the analysis whose matrices the edit ultimately rebuilds); Gamma et al., *Design Patterns* (the scope is a Builder-flavored unit-of-work; C#'s `IDisposable`/`using` supplies the commit-on-scope-exit mechanics).

### EditReceipt / EstimatedRebuildDim

A field on the commit receipt that predicts how big the upcoming matrix rebuild will be, so the caller can decide whether to pay for it now or defer it.

When a structural edit is committed, the solver may later need to rebuild an island's system matrix — the grid of equations it solves each step. `EstimatedRebuildDim` (with its sibling `EstimatedRebuildRows`) reports the expected dimension of that matrix *before* the rebuild happens, purely as a scheduler hint: a game tick has a fixed time budget, and a small rebuild can be paid immediately while a huge one (say, after loading a large chunk of the world) should be handed to a background task. For programmers: it is a cost estimate returned by a transaction commit, like a query planner's row estimate — advisory, cheap, and available before the expensive work runs. For the EE: matrix size is the dominant cost driver in a sparse solve, so "how many rows will the new system have" is the one number a real-time scheduler needs. It pairs with `CostOfAdjust`/`CostOfReconfigure`, which classify a mutation's cost tier without applying it — together they implement the tutorial's "ask before paying" rule.

**Project-coined term**, closest standard concept: a cost/cardinality estimate used for admission control or work scheduling.

**Where it appears:** `EditReceipt` struct, docs/api.md §6; scheduling guidance in api.md ("does a rebuild fit the tick budget") and docs/integration-tutorial.md §9 (the performance contract).

### EditReceipt / journal window / WindowLapped

The struct returned by committing a structural edit: it tells derived layers exactly which slice of the event journal describes this commit, how much changed, how big the coming rebuild is — and, via `WindowLapped`, whether that slice is already unusable.

`Commit()` on a `StructuralEdit` returns an `EditReceipt`, a plain read-only struct. Its **journal window** is the pair `JournalFrom`/`JournalTo`: the topology journal is a fixed-size ring buffer of change events, and replaying exactly that range brings a derived layer (e.g. the reduction/compaction layer, or a game client's bookkeeping) in sync with this one commit — the standard pattern is `OpenCursorAt(receipt.JournalFrom)` + `TryReadRange(.., receipt.JournalTo)`, no standing cursor needed. The receipt also carries counts (`NodesAdded`, `ComponentsAdded/Removed`, `IslandsTouched`) and the rebuild-size hints (`EstimatedRebuildDim/Rows`). **`WindowLapped`** covers the failure mode: a single oversized commit can emit more events than `JournalCapacity`, so the ring has already overwritten the start of its own window by the time `Commit` returns. Rather than letting a reader silently miss events, the receipt declares this up front, and the consuming layer must do a full `Resync` instead of a replay. For the EE: think of a chart recorder with a fixed length of paper — if one transaction writes more than the paper holds, the recorder tells you immediately that the trace is incomplete, instead of handing you a chart with an invisible gap. For programmers: it is the same contract as a lapped ring-buffer cursor (`Overflowed()`), but detected eagerly at commit time.

**Project-coined term**, closest standard concepts: a transaction commit record plus a write-ahead-log replay range, with explicit ring-buffer overrun signaling.

**Where it appears:** docs/api.md §6 (`EditReceipt` definition), §15 (journal, cursors, the two overflow cases, event-stream completeness), failure-mode table §17; docs/integration-tutorial.md §9.

### electrical graph
The connectivity model — which conductors touch which — that the Vintage Story mod maintains from the world's voxel data and hands to the solver as a netlist.

Cable voxels live inside the game's microblock system, so the mod cannot ask the world "what is connected to what"; it must derive that itself. A behavior attached to chiseled blocks scans the voxel cuboid lists whenever they are rebuilt and updates the electrical graph incrementally, so the graph is a cached, always-current answer to the connectivity question. For the programmer: it is literally a graph data structure (nodes and edges) kept in sync with world geometry, like an index maintained alongside a database table. For the EE: it is the schematic implied by the physical wiring — the extraction layer turns "these metal voxels are adjacent" into "these are one node," the same abstraction step as going from a PCB layout back to its netlist. The graph is the boundary artifact between world representation and solver: everything downstream (compaction, MNA) consumes it, never the voxels.

**Standard term:** essentially standard usage; in EDA this extraction step is called *netlist extraction* (from layout), and the result is the circuit's connectivity graph.

**Where it appears:** `docs/vintage-story.md` §Microblock Integration (Chosen architecture, representation 1); consumed by the compaction layer described in `docs/compaction.md`.

**References:** connectivity graphs of circuits are the starting point of nodal analysis — see Vlach & Singhal, *Computer Methods for Circuit Analysis and Design*, ch. 1–4.

### electrical tick (50 ms)
Manatee's own simulation heartbeat in Vintage Story: a tick listener the mod registers at a 50 ms interval, which is also the timestep (dt) for the DC-side transient analysis.

Vintage Story mods choose their own tick-listener intervals rather than inheriting a fixed engine rate, and Manatee chose 50 ms (20 Hz). Every electrical tick, the solver advances the circuit state by exactly dt = 50 ms using Backward Euler integration; component values are deliberately chosen so time constants sit at human-visible scales (0.1–10 s), where a 50 ms step is accurate — a trick inherited from Electrical Age. For the EE: this is a fixed-timestep transient simulation whose step size is pinned to the game's update cadence instead of being adaptively controlled. For the programmer: it is the classic fixed-dt game-loop pattern, with a performance budget attached (≤ 5 ms on-thread, ≤ 20 ms off-thread per tick). AC islands subcycle several solver substeps within each 50 ms tick. Note: an earlier 33 ms figure in older discussions was spurious and was corrected in the 2026-07-05 doc review; 50 ms is canonical. It is distinct from the engine's other clocks (the mechanical-power mod's 20 ms tick, Stationeers' 0.5 s power tick, and the tablet's 10 ms sim dt).

**Project-coined term**, closest standard concepts: fixed simulation timestep; SPICE's transient timestep, here non-adaptive.

**Where it appears:** `docs/design.md` (Simulation Model, Performance Targets, decision-log entry for the 2026-07-05 doc review), `docs/solver.md` (AC subcycling), `docs/api.md` (tick-loop examples).

### element line
In the Falstad/circuitjs1 text circuit format, a single line describing one circuit element: `TYPE x1 y1 x2 y2 flags [params...]`.

Each element line names an element type (`r` resistor, `c` capacitor, `w` wire, `v` voltage source, …), gives two integer canvas-pixel coordinates — the element's two posts for two-terminal parts (single-post parts use the second point only as a drawing direction; transformers and text spans use it as a bounding box) — then a per-type `flags` bitfield, then positional parameters in SI base units. Crucially, several types gate *the presence of later params* on flag bits, so a parser cannot skip flags; and versioning is purely positional-with-defaults (readers tolerate missing trailing params, ignore extra ones — that is the format's entire compatibility mechanism). For the EE: an element line is one row of a netlist, like a SPICE card (`R1 n1 n2 1k`), except connectivity is by canvas coordinates rather than named nodes — elements connect when their posts share a pixel position. Element lines are distinguished from non-element lines (`$` options header, `o` scope config, `x` text, `34` diode models, etc.). Manatee's importer treats these lines under a strict accept/reject contract: nothing is silently dropped.

**Standard term:** within the circuitjs1 ecosystem this is standard vocabulary; the closest general concept is a netlist entry / SPICE element card.

**Where it appears:** `docs/falstad-format.md` §3 (general form), §4 (authoritative per-element field lists), §7 (importer contract); source citations to `CircuitElm.java` and `CircuitLoader.java`.

**References:** Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, memo UCB/ERL-M520, for the netlist-card lineage; the format itself is defined only by the circuitjs1 source cited in the doc.

### energy accounting / conservation by construction
- The project-wide design rule (settled 2026-07-05) that the simulation can never create energy through its own bookkeeping: energy is conserved everywhere — solver, island boundaries, the Stationeers legacy-device adaptor — by how the code is *structured*, so every place accounting could produce a surplus makes that surplus explicit heat rather than usable work, not by after-the-fact correction.

Several parts of Manatee deliberately compute with slightly stale values — the Stationeers adaptor linearizes loads against last tick's voltage, and island boundaries exchange power with relaxation. Each such lag is a place where naive accounting would let energy appear from nowhere (e.g. an oscillating load "pumping" the one-tick delay). Two commitments follow. First, modeled inefficiency is *explicit*: every loss the model introduces is dumped as heat at the place where the loss physically lives (doubly interesting in Stationeers, where that heat lands in a real thermal simulation, so inefficiency is physically felt). Second, accounting surpluses that arise from mechanism — adaptor lag, boundary relaxation between islands — are debited against future transfers or converted to device heat, never silently stored as free work. Per-boundary and per-adapted-device **energy ledgers** (∫power out − ∫power in − modeled losses) enforce this at runtime; for programmers, the analogy is double-entry bookkeeping or a transactional invariant — designing a data structure so the invariant *cannot* be violated rather than checking it after the fact, with the ledgers as a runtime enforcement backstop. For the EE: the exact solver already conserves energy (Kirchhoff's laws, per Tellegen's theorem); this rule extends that guarantee across the approximation seams the solver's math doesn't cover.

The adaptor is the sharpest case. Because it models unconverted vanilla devices as constant-power elements linearized from the *previous* tick's voltage (G = P/V_prev²), a load oscillating in sync with the tick rate could be systematically over-served each tick — an exploit found in the 2026-07-05 adversarial playtest (finding C2). The fix is the across-tick rule **deliverable this tick = advertised − accumulated debt**: any over-delivery is recorded as debt and debited from the next tick's clamp, so the exploit degrades to a one-tick loan that repays itself (an anti-cheat on a cache-staleness race, implemented as a saturating counter; it bounds the model's energy error to zero-mean over any window). It ships in the stock `AdaptedLoad` alongside brownout hysteresis, staggered rejoin, and lockout. The whole rule is tested mechanically: the always-on debug energy audit (source energy = dissipated + Δstored within integration error), coupling-conservation ledger checks under oscillating loads, and both "conservation-under-oscillation" and "oscillating load vs adapted generator conserves energy over any window" as standing property tests.
- **Project-coined phrasing**, closest standard concepts: energy conservation as a structural invariant (Tellegen's theorem guarantees instantaneous power balance in any consistent circuit solution) plus passivity/dissipativity of the numerical scheme and anti-windup/saturation logic in a discrete-time controller; the ledger enforcement at island boundaries and adaptors is our own construction.
- **Where it appears:** `docs/design.md` (Simulation Model, "Energy accounting" design rule settled 2026-07-05; R18 adaptor energy-debt ledger); `docs/solver.md` (Islands: boundary ledgers, scheduling-and-conservation); `docs/stationeers.md` R18 (per-adapted-device ledgers); `docs/integration-tutorial.md` §4 (`AdaptedLoad`'s "anti-free-energy ledger"); `docs/api.md` §"Conservation is physical, not clerical" and `ConservationAuditTests`; `docs/testing-strategy.md` (Invariant Checks: energy audit, coupling conservation, coupling dissipativity); `docs/experiments/2026-07-05-adversarial-playtest.md` (finding C2).
- **References:** P. Penfield, R. Spence & S. Duinker, *Tellegen's Theorem and Electrical Networks*, MIT Press, 1970; Tellegen's theorem is also covered in Vlach & Singhal, *Computer Methods for Circuit Analysis and Design*.

### EnergyAudit / conservation audit
Manatee's true energy-conservation check: over any window of a transient simulation, energy put in by sources must equal energy dissipated plus the change in stored energy plus heat dumped at island boundaries — computed entirely from public readbacks.

`EnergyAudit` is a small read-only struct on `IslandHandle` (`Energy { get; }`) with fields `SourceJ, DissipatedJ, StoredJ, BoundaryNetJ, HeatDumpedJ, ResidualJ` — a running source/dissipated/Δstored accumulator. The *conservation audit* is the windowed test built on it: `SourceJ ≈ DissipatedJ + ΔStoredJ + HeatDumpedJ` summed across all coupled islands. It is one of Manatee's standing invariant checks (alongside the KCL residual and finiteness checks): active in debug builds and assertable in any test. Crucially, it is computed from the same public API a game client sees, so it can catch bugs anywhere in the pipeline — the solver, the boundary exchange, the adaptor — not just one internal bookkeeping path; and it deliberately does NOT trust `EnergyLedger.Residual`, which is true by construction and can never fail. The audit gate runs at gain-capable turns ratios (loop voltage gain ≥ 1), the regime where a non-conservative boundary exchange demonstrably "rings up" — a phase-5 review found a two-coupler loop growing a seeded 8.6 J to 85 J, which motivated making conservation physical (debt droop, DC-link cap) rather than clerical. For a programmer, it is an accounting reconciliation run continuously against the simulation (like reconciling a bank statement against actual transactions); if the books don't balance, some component wrote the wrong sign or corrupted its companion-model state (the per-step history capacitors and inductors carry between integration steps) — classes of bug that produce plausible-looking waveforms a human reviewer would miss. For an EE, it is a Tellegen-style power balance over the whole network, applied as a regression test.

**Project-coined term** (the name "energy audit"/`EnergyAudit`; outside the project "energy audit" usually means a building-efficiency inspection), closest standard concept: energy/power-balance verification (Tellegen's theorem applied as a simulator sanity check).

**Where it appears:** `docs/api.md` §7 (conservation-is-physical ruling, windowed audits), §11 (`IslandHandle.Energy`, `EnergyAudit` struct), §14 (audited state rides snapshots); `ConservationAuditTests` in the test suite; `docs/design.md` energy-accounting rule; `docs/testing-strategy.md` (Invariant Checks); listed as invariants-as-API in `docs/experiments/api-competition/proposal-testing.md`.

**References:** Tellegen's theorem in Vlach & Singhal, *Computer Methods for Circuit Analysis and Design* (transient analysis and companion models whose bugs this check targets); Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods*; conservation-based checking is standard simulator-validation practice (cf. the ngspice manual's convergence and accuracy checks).

### energy-debt feedback / debt droop
The conservation mechanism at transformer boundaries: when the boundary's ledger shows the B side has cumulatively received more than the A side supplied (minus modeled loss), the deficit is treated as debt that reduces ("droops") the amplitude injected into B until the boundary balances.

It is the boundary-coupling instance of the same rule the R18 adaptor uses — *deliverable = advertised − accumulated debt* — introduced after a 2026-07-06 review found a ledger-only clamp non-conservative (a gain-capable two-coupler loop grew a seeded 8.6 J to 85 J): the ledger records, the droop enforces. For a programmer, it is a feedback controller on an accounting error, like a rate limiter whose allowance shrinks by exactly the amount previously over-granted. For an EE, it resembles droop control, except the controlled variable is cumulative energy imbalance, not frequency or voltage. The hardened version survives a *driven* pump (an oscillating B-side load, e.g. a blinking switch, which defeated a naive deadband+leak — free energy leaked linearly at ~2.6% of throughput) via three scale-invariant pieces: a peak-hold input-throughput reference, a restoring leak gated on a settled-boundary detector (a sustained oscillation never settles, so it cannot launder its debt), and a wide deadband so isolated transients recover the exact turns ratio while sustained pumps accumulate past it and choke. Guarantee: driven over-delivery is a bounded one-time transient (imbalance slope → 0), and a sustained pump stays choked until the drive stops — erring conservative, never inventing energy. Note the droop is not a numerical-stability device: gain-capable-loop convergence is handled separately by clamping transformer relaxation α to ≤ 0.7.

**Project-coined term**, closest standard concepts: energy-balance enforcement at a co-simulation boundary; the "droop" naming nods to droop control in power systems but the mechanism is anti-windup-style debt feedback.

**Where it appears:** `docs/api.md` §7 ("Conservation is physical, not clerical", transformer boundary rules; `ConservationAuditTests`); `docs/design.md` (R18, Energy accounting); `docs/reviews/2026-07-06-overnight-build.md`.

### EnergyLedger / SurplusJ / HeatDumpedJ / ModeledLossJ / Residual
A running accumulator kept at every island boundary (and per adapted device) that tracks energy crossing a coupler — a transformer/converter/breaker boundary between islands — broken into named categories in joules, so any accounting surplus is detected and disposed of as heat rather than becoming free energy.

The struct is `EnergyLedger { InJ, OutJ, ModeledLossJ, SurplusJ, HeatDumpedJ, Residual }`, read via `IslandHandle.Ledger(CouplerId)`, accumulating ∫P_out − ∫P_in − modeled losses across substeps. `InJ`/`OutJ` are cumulative energy drawn from side A and delivered to side B. `ModeledLossJ` is the intended physical inefficiency, computed *independently* from the device's efficiency curve (e.g. `P_out·(1/η−1)` for the converter) — never synthesized as In−Out, so a bookkeeping error cannot masquerade as modeled loss. Because coupled islands are exchanged with a one-substep lag and relaxation, small accounting errors are inevitable; `SurplusJ` is that numerical surplus, and by the design rule it is **never stored as work** — it is converted to `HeatDumpedJ` on the coupling device (physically defensible: transformers do get hot, and in Stationeers that heat lands in a real thermal simulation). `Residual` is the definitional closure identity of these columns, ~0 *by construction* — it is NOT a violation signal; the real conservation test is the separate windowed `EnergyAudit`. For a programmer, it is double-entry bookkeeping where `Residual` is the check-digit of the ledger's own arithmetic, not an audit of the underlying reality — hence the 2026-07-06 ruling "the ledger records; it does not license": the ledger alone is bookkeeping, and actual conservation is enforced by the debt-droop and DC-link-capacitor mechanisms (see *energy-debt feedback*). For an EE, `SurplusJ`/`HeatDumpedJ` formalize the numerical energy error of a relaxed boundary exchange and force it to be dissipative rather than generative. It doubles as a CI invariant: property tests drive oscillating loads and assert the ledger stays within tolerance over any window.

**Project-coined term**, closest standard concept: per-interface energy/power accounting in co-simulation (partitioned/relaxation-coupled simulators), with the numerical coupling error made explicitly dissipative.

**Where it appears:** `docs/solver.md` (Islands, boundary couplings); `docs/design.md` (R18, Energy accounting rule, settled 2026-07-05); `docs/api.md` §7 (coupler specs, `EnergyLedger` struct, conservation ruling) and §11 (`Ledger(CouplerId)`); ledger integrators ride snapshots as `CouplerRuntime` scalars (§14); `docs/testing-strategy.md` (Coupling conservation invariant); origin in `docs/experiments/2026-07-05-adversarial-playtest.md`.

### EnterSteadyState / steady-state guard
A machine-checked scope placed around the per-tick device-drive phase that makes structural (tier-3) changes to the circuit impossible while devices are computing.

`Netlist.EnterSteadyState()` returns a `SteadyStateGuard` — a C# `ref struct`, meaning it can only live on the stack for the duration of one block and cannot be captured, boxed, or carried across an `await` (the type system itself guarantees the scope is bracketed). Inside the guard, `Edit()` throws in *all* build modes; a coupler `Reconfigure` asserts in debug builds and is deferred to guard-Dispose (same tick, before Solve, counted in `TickStats.DeferredStructuralOps`) in release. Constructing the guard also arms an `AllocationSentinel` — a memory-allocation tripwire that checks the thread allocated zero bytes inside the region — enforcing the zero-allocation-in-steady-state guarantee (R8). The sentinel is best-effort in-process: its counter is unreliable on Unity/Mono, so it self-disarms if inert, and the binding enforcement is delegated to BenchmarkDotNet in CI. The intent, verbatim from the tutorial: the guard is "the machine-checked version of 'nobody rebuilds the world inside the device loop'" — it converts a mid-loop structural mutation from a heisenbug into an immediate assert or counter. For EEs: think of it as an interlock (or a lockout-tag hung on a running breaker panel) — while the machine is running, the panel that reconfigures its wiring is physically locked; adjustments to knobs (drive values, setpoints) remain allowed. Costs nothing per tick (stack-only, zero heap allocation).

**Project-coined term**, closest standard concepts: a scoped guard / RAII critical section (the C++/C# `using`-scope idiom) combined with a mode invariant check; note "steady state" here means the *engine's* no-structural-changes phase, not the EE sense of a circuit's settled sinusoidal/DC operating point.

**Where it appears:** `docs/api.md` §8 (`SteadyStateGuard`, `AllocationSentinel`, full semantics), §4 (`Netlist.EnterSteadyState`), §16 (0B table); `docs/integration-tutorial.md` §5 (tick-loop phase 3), §9 (performance contract), sharp-edges appendix item 7; ratified decisions §22.

**References:** Stroustrup, *The C++ Programming Language* (the RAII acquire-in-constructor / release-in-destructor idiom the `using`-scope mirrors).

### envelope (per-limit-type)
The safety-limit configuration a collapsed series chain carries: for each kind of limit, the strictest (minimum) value over all the original segments — where different limit types may each be inherited from a *different* segment.

When the reduction layer collapses a run of cable segments into one equivalent resistor, the equivalent must still trip exactly when the raw chain would. Its envelope therefore takes, per limit type, the minimum over constituents: instantaneous ampacity, i²t thermal mass, and melting threshold can each come from a different segment in a mixed-material chain — the canonical example is a lead fuse in a copper run, where the fuse dominates the thermal limit while other properties come from the copper. Instantaneous limits keep a plain minimum (always exact); the i²t envelope is a **Pareto-minimal set** of `(rating, melt, tau)` pairs rather than a single hybrid pair, because a hybrid of X's rating and Y's melt threshold fires trips the raw graph never would (ruled 2026-07-06, api.md §19). Attribution runs the other way: when the solver raises a limit event on the equivalent, the envelope's `PairIndex` names which segment actually melts or pops. Recomputing the envelope on ambient change (`SetAmbient` → `Meta.SetLimits`) is tier-0 — metadata only, never a matrix change. For programmers: it is a precomputed aggregate over a collapsed range, like a segment tree node storing the min of its children, kept per-key rather than as one scalar so no information is fabricated. Note this is unrelated to the EE sense of "envelope" as a modulated waveform's outline.

**Project-coined term**, closest standard concepts: composite device rating / worst-case derating of a series string; the i²t part corresponds to fuse melting-integral (I²t) coordination, per IEC 60269 / manufacturer fuse curves.

**Where it appears:** `docs/api.md` §12 (thermal envelopes, `I2tPair`, `PairIndex`), §19 (Pareto-set ruling, dominance contract, `Attribute`); `docs/compaction.md` Responsibilities #4.

**References:** fuse I²t (melting integral) coordination is standard practice — see e.g. Wright & Newbery, *Electric Fuses* (IET); protection-coordination chapters of Horowitz & Hill, *The Art of Electronics* cover the intuition of series protective elements.

### environment-adjusted threshold
A limit threshold (ampacity, melting point, i²t rating) corrected per cable segment for the ambient temperature it sits in, so the same wire tolerates less current when it is already hot.

Stationeers worlds span roughly −150 °C (Europa) to +800 °C (Vulcan), so a fixed current rating would be wrong almost everywhere: a conductor near its melting point has far less thermal headroom than one in cryogenic vacuum. Each segment's thresholds are therefore derated (or uprated) from its local ambient before being folded into the collapsed chain's per-limit-type envelope. The key engineering property is the cost class: an environment change dirties *only the envelope* — a tier-0, metadata-only recompute via `ConductorGraph.SetAmbient(segment, kelvin)` → `Meta.SetLimits` — and **never** touches the solver matrix, because temperature here changes what counts as "too much" without changing the circuit being solved. An ambient re-derate with unchanged envelope membership re-registers in place and preserves the per-pair melting integrals (partial melt survives a weather change, matching raw semantics); only a membership change resets them. For programmers: it is a config-value refresh on a hot path's guard conditions, deliberately kept off the expensive rebuild path — like changing an alert threshold without redeploying the service. For EEs: this is standard ampacity derating for ambient temperature, applied per segment and then min-folded.

This usage is standard (temperature derating of conductor ratings); the project-specific part is the tier-0 "metadata only, never a matrix change" cost guarantee.

**Where it appears:** `docs/compaction.md` Responsibilities #4; `docs/api.md` §19 (`SetAmbient`, tagged `[T0]`) and §12 (envelope re-registration semantics).

**References:** ambient-temperature ampacity derating is codified in NEC (NFPA 70) Table 310.15 correction factors and IEC 60287 (current rating of cables); Horowitz & Hill, *The Art of Electronics*, discusses wire ratings and thermal limits informally.

### Error model: three channels
- Manatee's binding rule that every failure travels on exactly one of three distinct channels — exceptions, `Faulted` island state, or polled post-solve events — and the channels are never blended.

Channel 1 is **exceptions**: programmer/contract errors (stale handle misuse, an illegal edit, a nested `Edit()`) fail fast and indict client code — the bug is in the caller, so the program should stop and say so. Channel 2 is the **`Faulted` island state**: circuits that are legal to build but electrically degenerate (a singular matrix after gmin, contradictory sources, Newton failing to converge) become *data*, not exceptions — players build nonsense as a hobby and the game must not crash, so the island reads as de-energized and carries a `FaultDiagnostic` naming the offending nodes/components. Channel 3 is **polled post-solve events**: normal runtime signals (limit trips, island merges, ring overflows) sit in fixed-size rings the client drains after each solve — no callbacks ever. For an EE, the analogy is instrumentation discipline: channel 1 is a design-rule violation caught at the bench, channel 2 is a protective relay tripping into a defined safe state, channel 3 is the event log you read after the run — and you never route a relay trip through the design-rule checker.

**Project-coined term**, closest standard concepts: fail-fast exceptions for contract violations, error codes/status as data, and event polling versus callbacks (see "Observer" in Gamma et al. — deliberately *rejected* here in favor of polled rings).

**Where it appears:** `docs/api.md` §20 (the full condition-by-channel table); Faulted semantics also in `docs/solver.md` failure handling and design.md R9.

### Error posture
- A parser's policy for what happens when input is malformed or unrecognized — what gets rejected loudly, what gets skipped, and what the user is told.

In the Falstad-format work the term contrasts two postures. Upstream circuitjs1 is *silently lossy*: an unrecognized dump type or a line that throws during parsing is simply skipped with only a console log (`CircuitLoader.java:200-207`), so a file can load "successfully" while missing components. Manatee's importer is *strict and loud*: nothing may be silently dropped, because the imported lesson corpus is CI truth — accepted elements get full electrical semantics; presentation-only lines (scopes, hints, sliders) are ignored but counted and reported; everything else is rejected with line number, offending token, and a one-line reason. For an EE, the analogy is incoming-inspection policy: one shop quietly discards parts that fail inspection and builds with what's left; the other refuses the shipment and files a discrepancy report — only the second lets you trust that what was built matches the drawing.

The term itself is ordinary English usage (a stance on error handling), not project jargon; the closest named concepts are strict versus lenient/permissive parsing and Postel's law (which Manatee deliberately does not follow here).

**Where it appears:** `docs/falstad-format.md` §1 (upstream's silently-lossy behavior) and §7 ("Manatee importer subset and error posture" — the accept/ignore-with-notice/reject-loudly contract).

### Exchange scheme / exchanged values
- The protocol by which two islands joined by a power-transfer boundary (a transformer or converter two-port) communicate, since they are solved as separate matrices rather than one.

When a coupling device is a power-transfer boundary rather than a galvanic bridge, each side stays its own island with its own matrix, and the device passes summary quantities — the *exchanged values* — across the seam: AC boundaries exchange amplitude + phase once per **substep** (so the lag is 1/20 of a cycle, versus a quarter cycle if exchanged only per game tick), DC boundaries exchange DC values. The exchange is damped by explicit relaxation — exponential smoothing on the exchanged values with a per-device-type α — so the two sides converge instead of oscillating against each other. For a programmer, this is message passing between two independently-stepped simulations with a low-pass filter on the messages to keep the feedback loop stable; the coupled islands are scheduled as one work unit and substep in lockstep, so the exchange never crosses free-running threads and determinism holds. Conservation is enforced by an energy ledger per boundary: transfer is clamped to what the source island actually delivered, and any accounting surplus becomes heat in the coupling device, never stored work.

For an EE, this is a co-simulation / waveform-relaxation style partitioned solve rather than one monolithic MNA system; "exchange scheme" is our name for the agreed quantities, cadence, and damping. **Standard term:** relaxation-based co-simulation / waveform relaxation (Lelarasmee, Ruehli & Sangiovanni-Vincentelli's waveform relaxation is the classic ancestor).

**Where it appears:** `docs/solver.md` Islands (coupling devices, exchange scheme settled 2026-07-05, scheduling and conservation); coupler runtime state in `docs/api.md` §14.

**References:** Lelarasmee, Ruehli & Sangiovanni-Vincentelli (1982), "The Waveform Relaxation Method for Time-Domain Analysis of Large Scale Integrated Circuits", IEEE Trans. CAD.

### ExternalKey
- The mandatory 128-bit identity a client attaches to every node, component, coupler, and probe it adds, and the only name for a piece of the circuit that is guaranteed to survive the solver reorganizing itself.

Internally the solver renumbers freely: an island rebuild throws away all internal ids (`NodeId`, `IslandId`, component refs are "volatile"). The `ExternalKey` is the client's own stable name — VS packs dimension+position+material into it, Stationeers packs a RefId, the tablet packs an element id — and it is a *non-optional* parameter on every `Add*` call, so forgetting it is a compile error. It drives everything that must outlive a rebuild: `TryResolve` (re-find your component after a rebuild), drift diffs, canonical serialization, and deterministic island ordering. `Derive(ordinal)` mints deterministic sub-keys for the internal components of a compound device. For an EE: think of it as the schematic reference designator (R1, C3) that stays meaningful no matter how the simulator internally reorders its matrix rows. It names *topology* (a thing's place in the circuit); its sibling `StateKey` names serializable dynamic state — one device has one StateKey but many ExternalKeys.

**Project-coined term**, closest standard concept: a stable external/user key (as in database natural keys or a persistent handle/GUID scheme), versus the solver's internal surrogate ids.
- **Where it appears:** `docs/api.md` §3 (Identity), §16 (serialization), §18 (device kits); every `Add*` signature; `docs/integration-tutorial.md` §3 and appendix 17 (handle survival).

### extraction / extraction layer / intake
- "Extraction" is the game-side act of converting in-world objects (voxels, block entities, cable exports) into a netlist the solver understands; the "extraction layer" is the code that does it, and "intake" is the matching entry point it hands the result to on the compaction/solver side.

The solver core never sees a game — it sees nodes and components. Extraction is the translation step in each client that produces that graph. On the Vintage Story side (per R17) cables live in three coexisting representations: cable voxels inside vanilla chiseled (microblock) blocks, dedicated pure-cable block entities (the cheap fast path for long runs), and decor-layer surface wiring. Rather than three code paths producing three flavors of circuit, the VS extraction layer is the single funnel that consumes all three *uniformly* (sparse voxels → greedy-meshed prisms → regions, sparky's pipeline) and emits one electrical graph, which flows through compaction into manatee-core; the Stationeers adaptor instead reads `NetworkExport` adjacency, and the tablet/2D harness passes schematic wires through nearly as-is. "Intake" is compaction's word for its entry point — the client intake contract each game meets when handing raw circuit graphs to the reduction layer (`docs/compaction.md` names one per client). The design point is the funnel shape: however many storage formats a game invents, they converge before compaction, so series-collapse, probe interpolation, and limit attribution never need to know which representation a wire came from. For a programmer this is an adapter/facade over heterogeneous storage (three parsers, one AST); for an EE it is the schematic-capture stage (three construction techniques, one netlist), after which everything downstream is pure circuit topology. It sits strictly on the game side of the dependency arrows: the core knows nothing of voxels.

Because in-game testing is manual by nature, the testing strategy gives these layers thin automated coverage built on **fixtures**: recorded voxel layouts or game exports saved to disk, replayed through the extraction code in CI (**extraction unit tests**), with the resulting netlist asserted against a golden answer. For an EE, a fixture is like a saved measurement setup — the same known input replayed so any change in output is the code's fault, not the world's. The standing fixture rule: every in-game bug that *can* be captured as an extraction fixture or netlist becomes a permanent regression test; only genuinely engine-interactive bugs stay manual.

**Project-coined usage**, closest standard concepts: netlist generation / schematic capture / netlist builder / graph extractor (EE), the netlist-input front-end of a circuit preprocessor, plus fixture-based regression testing (software).
- **Where it appears:** `docs/design.md` (R17, System Overview diagram — "VS voxel extraction"), `docs/vintage-story.md` §1 (Chosen architecture: "three coexisting representations, consumed uniformly by the extraction layer"), `docs/compaction.md` (Client Intake Contracts), `docs/testing-strategy.md` (Game-Layer Testing, fixture rule).

### Falstad format / Falstad text / Falstad importer
- The circuit-file format all Manatee lessons and importable schematics are written in ("Falstad format" / "Falstad text" / a "Falstad-format netlist"), and the Manatee parser that reads it (the "Falstad importer").

A netlist is the EE machine-readable parts-and-connections list of a circuit (a SPICE `.cir` deck is the classic example); a Falstad-format netlist is that list in Paul Falstad's circuitjs1 line-per-element text dialect, which additionally carries 2D canvas coordinates so the same file is both a solvable circuit and a drawable schematic — human-readable and hand-editable. The format is line-oriented plain text: a `$` options header followed by one line per element with grid coordinates, flags, and values, tokenized on whitespace (and, for historical URL-encoding reasons, on `+`). It has no version field. circuitjs1's only "spec" is its own source code, so `docs/falstad-format.md` documents the format from that source, plus the extensions Electrical Age added (notably `#` comment lines). In each lesson directory the file is named `lesson.txt` (following Electrical Age's `docs/examples` layout) and is paired with a Markdown narrative carrying numeric expectations and a machine-readable front-matter block.

The Falstad importer, which lives in Manatee's schematic document model (`Manatee.Schematic`, harness layer 1), turns a Falstad-format text file into a manatee-core netlist via `Edit()` — applying the ideal-node-per-wire convention and a strict accept/reject contract: supported elements parse exactly, presentation-only lines are ignored-and-counted, and unsupported ones are rejected loudly (with line number and reason), deliberately unlike upstream circuitjs1, which silently skips unparseable lines. Under design.md R20 ("one corpus, three consumers"), the identical lesson file serves as tablet tutorial content, documentation example, and CI golden test — solved in manatee-core and checked against both ngspice and the expected values stated in its own prose, so "a lesson that stops being true fails the build." The same format doubles as the tablet's persistence layer: lesson progress and player sandbox circuits ride Vintage Story player attributes as Falstad text, so save data, lesson files, and CI test inputs are all one format handled by one parser. Because CI must parse lessons from day one, testing-strategy.md classifies the importer as core infrastructure written first (not late-stage UI polish), with the EA example corpus as its initial test data. Systems framing: the format is the wire protocol for the lesson corpus, and the importer is the shared protocol decoder three independent consumers rely on.

**Standard term:** circuitjs1 format (also called the Falstad export/text format; there is no official upstream name). Aliases: circuitjs1 export text; often just "Falstad text" in our docs.

**Where it appears:** `docs/design.md` (The Three Clients, R16 Falstad-netlist import, R20); `docs/testing-strategy.md` (Lesson Corpus as Goldens; "core infrastructure, not polish"); `docs/curriculum.md` (Format, Lesson Arc, `lesson.txt` per lesson); `docs/vintage-story.md` (Tablet Host: persistence as Falstad text); `docs/harness.md` (Layering, layer 1 — document model); full spec and accept/reject contract in `docs/falstad-format.md`.

**References:** L. W. Nagel (1975), "SPICE2: A Computer Program to Simulate Semiconductor Circuits", UCB/ERL-M520 — the netlist-card ancestor of all such formats; P. Falstad & I. Sharp, *circuitjs1* source (github.com/pfalstad/circuitjs1, GPLv2), specifically `CircuitLoader.java` — the de facto specification, since the format has no written spec upstream.

### fault / island faults
- In the Stationeers (Re-Volt) integration: an island whose solve hits a singularity or fails to converge, which triggers a per-tick fallback rather than any stall of the game.

Each island (electrically independent sub-network) solves on its own; when one faults, only that network is affected. For that tick the integration falls back to Re-Volt's pre-existing scalar ratio distribution — the simple "split available power proportionally among loads" arithmetic the mod used before manatee — and logs the diagnostic. Neighboring islands keep solving normally, and the faulted island retries once something about it changes. This sits inside a larger graceful-degradation ladder: manatee → Re-Volt scalar → vanilla game logic. The systems analogy is a service mesh with per-shard circuit breakers: one shard failing health checks gets its traffic routed to a degraded-but-correct fallback while the rest of the fleet is untouched, and an alert fires. For an EE: it is selective protection coordination — the fault is isolated to its own bus section and the rest of the plant rides through.

The *causes* are the standard numerical-analysis failures (singular matrix, Newton non-convergence); the fallback-per-island policy is project-specific.

**Where it appears:** `docs/stationeers.md` (Failure and Fallback); the mechanism it triggers on is `docs/api.md` §11/§20 (`Faulted`, `FaultDiagnostic`).

**References:** on the underlying failure modes, Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods* (Newton convergence, singular systems); Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520.

### Fault (verb/state)
- To fail *legibly*: when a circuit cannot be solved, the solver deliberately transitions into a diagnosable error state instead of crashing, hanging, or emitting NaN garbage.

Used as a verb throughout the docs ("must converge or Fault legibly — never NaN, never hang"). Faulting is a designed, first-class outcome of a solve attempt, on equal footing with success: the island's status becomes `Faulted`, a `FaultDiagnostic` records why and names the worst offender, and readback returns clean de-energized zeros rather than the failed solve's numbers. This is requirement R9 ("Legible failure"): players build pathological circuits as a hobby, so "why doesn't this work" must be answerable in-game. The programming analogy is returning a typed error value instead of throwing or corrupting state — a `Result` type, not an exception; the EE analogy is a breaker tripping and annunciating, versus letting the magic smoke out. Property/fuzz tests enforce the contract mechanically: randomized nonlinear circuits must either converge or Fault, with no third outcome.

**Project-coined term** (as a verb), closest standard concepts: fail-fast / graceful degradation (software); in SPICE-family simulators the underlying events are reported as convergence failure or a singular-matrix error, but they typically abort the analysis rather than quarantine it.

**Where it appears:** `docs/design.md` R9; `docs/testing-strategy.md` (Property and Fuzz Tests); `docs/api.md` §11, §20.

### FaultDiagnostic / FaultKind
- The API types that answer "why did this island Fault?": a small fixed-size struct of scalars (`FaultDiagnostic`) categorized by an enum (`FaultKind`), with the full culprit lists drained on demand.

`FaultKind` has four values naming the failure class: `Singular` (the equations have no unique solution, e.g. a floating sub-circuit even after gmin remedies), `ContradictorySources` (two ideal sources demand incompatible values — like two hard drivers fighting over one wire), `NewtonDiverged` (the nonlinear iteration never settled), and `NonFinite` (a NaN/Infinity appeared in the solution — the NaN/Infinity value itself is never published, since no NaN ever leaves the API; debug builds assert, release builds Fault the island with `Kind=NonFinite`). `FaultDiagnostic` carries the kind plus the single *worst* component and node and counts of how many are implicated; if the client wants the full lists it calls `island.DescribeFault(...)` with its own buffers, keeping the hot path allocation-free. The design pattern is a fixed-size error record plus a pull-based detail drain — the same shape as an errno plus a follow-up query, rather than a heap-allocated exception object. For an EE: the struct is the annunciator panel (what tripped, worst offender), `DescribeFault` is pulling the full event log.

**Project-coined types**; closest standard concepts: a SPICE convergence-failure / singular-matrix report, which names the last non-converging node or device (see the ngspice manual's convergence diagnostics).

**Where it appears:** `docs/api.md` §11 (`IslandHandle.Fault`, `DescribeFault`, struct/enum definitions) and §20 (error model table: which failures produce which `FaultKind`).

### Faulted
- The island status for a circuit that is *legal but unsolvable*: the player built something the rules allow but mathematics refuses (contradictory sources, a singular system), so the island parks in a diagnosable error state instead of publishing bad numbers.

`Faulted` is one of the five `IslandStatus` values (`Empty, Building, Ready, Dirty, Faulted`) in the island lifecycle state machine. An island enters it when a solve fails after all remedies (gmin ramping, the Newton retry ladder), and leaves it automatically: any structural or parameter change flips it back to `Dirty`, which schedules a retry — there is no manual "reset" verb. While Faulted, the island reads as de-energized: `IsLive == false` and every potential/current reads 0, enforced at the read path and scoped to the *status* — the failed solve's numbers are never published, and once the island leaves Faulted its previously published good vector becomes readable again as ordinary last-good values (or 0 if it never published). Neighboring islands keep solving throughout. The programming analogy is a circuit-breaker pattern with automatic half-open retry: the shard serves a safe default (zero) while broken, and any write to it re-arms the health check. Docs stress two consumer sharp edges: never ignore `IsLive`, and treat ~0 V buses as dead (shed load) rather than seeking power from them.

**Project-coined term**, closest standard concept: a non-convergent or singular solve — but where SPICE-family tools abort the whole analysis, here the failure is quarantined per island, held, and retried.

**Where it appears:** `docs/api.md` §11 (status enum, lifecycle comments), §17.4 and §20 (de-energized read gating, decision log #27); `docs/integration-tutorial.md` §1, §6, §8 (consumer-side handling and sharp edges).

### Faulted / de-energized
- The read-side contract for a failed island: while its status is `Faulted`, every voltage and current the API reports is exactly 0, as if someone had pulled the breaker.

When a solve fails (singular matrix, contradictory ideal sources, Newton divergence, or a non-finite result), the island's status becomes `Faulted` and `IsLive` reports false. `NodePotential`/`BranchCurrent` gate on that status and return de-energized zeros — the guarantee "no NaN ever leaves the API" is enforced at the read path, not by hoping the math stayed finite. The scoping matters and is status-scoped, not taint-scoped: the pre-fault solution vector is still held internally, and the moment any tier-2/3 change flips the island back to `Dirty`, that held vector reads again as an ordinary last-good value (decision log #27 settled this after an earlier draft kept pre-fault values readable during the fault). For a programmer: it is a property getter that checks an enum before touching the backing field. For an EE: a protective relay that opens on any fault and presents dead terminals until the circuit is touched again.

**Project-coined contract**; "de-energized" is the standard electrical-safety term for a conductor with no voltage on it, used here deliberately.
- **Where it appears:** `docs/api.md` §11 (status machine), §17.4 (merge with a Faulted side), §20 (read-path enforcement), §24.27 (decision log); `docs/solver.md` Failure Handling ("client-facing rule").

### Faulted (state)
- The per-island error state an island enters when the solver cannot produce a valid solution — the electrical equivalent of a caught exception that quarantines one circuit instead of crashing the simulation.

An island becomes `Faulted` when its matrix is still singular after the gmin fallback, when the Newton fallback ladder (reuse last operating point → source-stepping ramp → give up) is exhausted, or when a non-finite value shows up in a solution vector. Entering the state emits an event carrying a `FaultDiagnostic` naming the offending nodes/components — R9's principle that pathological player circuits are data, not crashes. The previous solution is held internally as a warm start, but while the status is `Faulted` all client reads return de-energized zero, so no NaN or stale value ever leaves the API. Recovery is automatic: any tier-2/3 change flips the island back to `Dirty` and the next solve retries. Think of it as a circuit breaker pattern per island: trip on failure, hold safe defaults, re-arm on the next state change.

**Project-coined term**, closest standard concept: a solver error/degraded state; SPICE-family tools instead abort the analysis with a "singular matrix" or "no convergence" error — Manatee's requirement to keep the rest of the world ticking makes it a first-class island status instead.
- **Where it appears:** `docs/solver.md` Failure Handling; `docs/api.md` `IslandStatus.Faulted`, the island status state machine (§11), failure table (§24), decision log #27.
- **References:** Nagel (1975), SPICE2 memo UCB/ERL-M520, for the convergence-failure behavior this replaces; Nygard, "Release It!" for the circuit-breaker stability pattern.

### fetched-not-vendored fixtures
- Test input files we download at test time instead of copying into our repository, so their license never attaches to our code.

Falstad's circuitjs1 ships hundreds of example circuit files (`public/circuits/`) that exercise every quirk of the text format — ideal fuzz and round-trip material for our importer. But circuitjs1 is GPLv2 and manatee-core is MIT: "vendoring" (checking a copy of third-party files into our repo, the code equivalent of stocking a shelf with someone else's parts) would mix GPL content into an MIT tree. So the test harness fetches the files from upstream when the tests run, uses them only as parser inputs, and never includes them in the shipped corpus; our own lesson files are hand-written instead. For an EE: like borrowing a calibration standard from the metrology lab for the duration of a measurement rather than keeping it in your bench drawer — you get the reference without owning (or being bound by) it.

**Project-coined term**, closest standard concept: test fixtures (canned inputs a test suite runs against), with a fetch-at-test-time acquisition policy.
- **Where it appears:** `docs/falstad-format.md` §5.7 (don't vendor GPL examples) and the "Free test material" note in §7.

### Fingerprint
- A cheap hash summarizing an island's entire state in one 64-bit number, used as the fast first check that the live simulation hasn't silently drifted from the truth.

`Fingerprint(island, scope)` is a zero-allocation call ("0B") that folds an island's contents into a `ulong`; two scopes exist — `Structural` (topology only) and `Full` (adds tier-0 envelopes and probe weights). It is level 1 of the two-level drift backstop (R11): every N ticks the cheap fingerprint is compared, and only on a mismatch does the system pay for level 2 — a full `SaveNormalized` canonical diff producing a typed `DriftReport` (named `ExternalKey`s, live vs shadow values: a bug report with coordinates, never mystery corruption) and then a `Resync`. This is the classic checksum-then-diff pattern from replication systems (compare hashes cheaply, transfer/diff only what disagrees). For an EE: a go/no-go continuity check you run periodically; only when it fails do you bring out the full instrumented characterization.

**Project-coined API name**; closest standard concepts: a checksum/hash digest used for integrity verification, as in Merkle-tree or rsync-style replication checks. (Distinct from `EnvFingerprint`, which identifies the runtime/arch/build for bit-exactness scoping.)
- **Where it appears:** `docs/api.md` §4 (Netlist API, `FingerprintScope`), §14 executable law 5, §19 (drift backstop / `DriftReport` / `Resync`); `docs/compaction.md` resync backstop.

### FLAG_COS
- A legacy flag bit in Falstad's voltage-source file format (`flags & 2`) that once meant "use cosine instead of sine"; modern loaders rewrite it into an equivalent phase shift of π/2 and clear the flag.

Falstad/circuitjs1 AC voltage sources compute `maxVoltage·sin(2πft + phaseShift) + bias`. Very old saved circuits could instead request a cosine waveform via bit 2 of the element's flags field. Since cos(x) = sin(x + π/2), the cosine option is redundant, and circuitjs1 normalizes it on load: if `flags & 2` is set, set `phaseShift = π/2` and drop the flag (`VoltageElm.java:60-64`). For a programmer, this is a data-migration shim: an obsolete on-disk representation upgraded to the current one at parse time, so the rest of the code only ever sees one canonical form. Manatee's importer follows the same contract so that ancient Falstad and Electrical Age circuit dumps load with correct phase.
- **Project-relevant legacy term** from the Falstad/circuitjs1 source (the constant's actual name there); the underlying math is the standard trigonometric identity cos θ = sin(θ + π/2), i.e. a 90° phase lead.
- **Where it appears:** `docs/falstad-format.md` §4, voltage-source semantics (with circuitjs1 file:line citations).

### Flag-gated params
- Parameters in a Falstad-format element line that are only present when a specific bit in that line's flags field is set — so the flags value changes how many fields follow and what they mean.

  Most of the format is "positional with optional tail": extra parameters are appended over the years and old readers ignore them. But some additions were instead gated on a new flag bit — the capacitor's series resistance appears only if `flags&4`, a switch's label only if `flags&4`, a diode's model name only if `flags&2`, a pulse source's duty cycle only if `flags&4`. For a programmer this is a variant record: you must decode the discriminant (flags) before you can parse the rest of the line. For an EE, think of it like an option code on a part number — the suffix fields only exist when the option digit says the option is fitted. The practical consequence, stated in our spec: **a parser cannot skip the flags field**, even though it is "just display options" for many element types.

- **Project-coined term**, closest standard concept: optional fields discriminated by a tag/bitmask (a tagged variant record).
- **Where it appears:** `docs/falstad-format.md` §3 (element-line grammar), §4 (per-element tables), §5.1 (compatibility mechanics); circuitjs1's `public/circuits/` corpus exercises them in our parser tests.

### FLAG_MODEL / FLAG_FWDROP / FLAG_ESCAPE / FLAG_REVERSE
- Named individual bits inside the per-element flags bitfield of the Falstad file format, each switching on an optional behavior or an optional extra parameter for a specific element type.

  These names come from circuitjs1's source and we reuse them so our importer spec can be checked against the original code. `FLAG_FWDROP` (`flags&1` on a diode `d` line) means "a forward-voltage-drop parameter follows"; `FLAG_MODEL` (`flags&2` on `d`/`z` lines) means "an escaped model *name* follows instead, referring to a `34` model-definition line"; `FLAG_ESCAPE` (`flags&4` on an `x` text annotation) selects the newer single-escaped-token text encoding over the old join-all-tokens style; `FLAG_REVERSE` (`flags&4` on a transformer `T` line) flips winding polarity — the dot convention, for an EE; a sign flip on the coupling, for a programmer. Note that the same bit *value* means different things on different element types (bit 4 is ESCAPE on text but REVERSE on transformers) — the flag namespace is per-element, like a tagged union whose field meanings depend on the tag.

- **Standard term:** none — these are circuitjs1 source constants (Falstad's simulator), adopted verbatim in our format spec.
- **Where it appears:** `docs/falstad-format.md` §4 (per-element param tables) and §5 (compatibility notes); governs the importer's accept/reject contract in §7.
- **References:** circuitjs1 source (`DiodeElm.java`, `TextElm.java`, `TransformerElm.java`), the authority our spec cites by file:line.

### from-scratch build
- Rebuilding an island's compacted netlist directly from the raw cable geometry, using none of the incrementally maintained state.

The compaction layer normally updates its reduced netlist incrementally as players add and remove cables, because that is cheaper than recomputing everything. The from-scratch build is the reference path: throw away the incremental bookkeeping and rederive the compacted netlist from the source-of-truth geometry alone. It is deliberately kept cheap at island scale (an island is one connected circuit), so it is always affordable to run. This makes it the basis of the resync backstop: a validation mode rebuilds from scratch in the background and diffs the result against the incrementally maintained state, turning inevitable incremental-update edge-case bugs into logged discrepancies rather than mystery corruption. EE analogy: it is the difference between keeping a running tally of a changing network's reduced equivalent versus re-deriving the equivalent from the full schematic — the re-derivation is slower per use but cannot accumulate drift, so you use it to audit the tally.

**Standard term:** essentially a full rebuild / recomputation-from-source; the pattern of checking incremental state against it is often called a shadow or oracle check in software testing.

**Where it appears:** `docs/compaction.md` Incremental Maintenance (Resync backstop) and `docs/testing-strategy.md` (incremental equivalence tests).

### front-matter block
- The machine-readable header in each lesson's README that states the lesson's expected measurements — probe, time, value, tolerance — so CI can verify the narrative against the solver.

Every lesson in the curriculum corpus pairs a Falstad-format netlist with a Markdown narrative that makes concrete numeric claims ("the capacitor should read about 2.7 V after 4.5 s"). The front-matter block restates those claims as data: which probe to read, at what simulated time, what value is expected, and how much tolerance is allowed. CI's narrative pass runs the netlist through manatee-core and checks each tuple; a mismatch fails the build. This is the mechanism behind the project rule *"a lesson that stops being true fails the build"* — the teaching text is a regression test, not just documentation. EE analogy: the block is the test-conditions-and-limits table for the lesson, and CI is the automated test bench that re-measures against it on every change.

**Standard term:** YAML front matter (the document convention); the use as executable expectations makes each lesson a golden test.

**Where it appears:** `docs/curriculum.md` (Format) and `docs/testing-strategy.md` (The Lesson Corpus as Goldens); requirement R20 in `docs/design.md` (one corpus, three consumers).

### fudge factor
- An arbitrary tuning constant inserted into a formula purely to make the output land where you want it, with no physical or architectural justification.

In Manatee the term appears only to be *refused*. `docs/vintage-story.md` derives generator frequency honestly from verified engine facts — the Vintage Story engine integrates shaft angle at ω = 5·speed rad/s, so electrical frequency is f ≈ 0.8 × speed × polePairs Hz — and then states "**No fudge factor.**" If designers want higher output frequency, the honest knob is the machine's pole-pair count (a real, costly design parameter: pole pieces are hand-smithed), not a hidden multiplier. This is the project's general stance: every number a player can observe must trace to a real mechanism, because educational realism is the point. For programmers, a fudge factor is the moral equivalent of a magic constant patched in to make a test pass; for EEs, it is a coefficient with no derivation. Both audiences should read its absence as a design invariant, not an oversight.

The term itself is common engineering slang, not project-coined.

**Where it appears:** `docs/vintage-story.md` §2 (frequency arithmetic, "No fudge factor"); the same honesty rule shows up in `docs/experiments/2026-07-05-adversarial-playtest.md` ("teachable mechanisms, not fudges").

### Galvanic bridge
- A closed breaker or bus-tie in the Stationeers integration that forms a real conductive (metal-to-metal) connection, so the two vanilla cable networks it straddles are solved as one Manatee island.

Vanilla Stationeers merges each connected cable run into a `CableNetwork` and never merges those identities across a breaker. Manatee instead tracks coupler state with a union-find over networks: while the breaker is Closed, both `CableNetwork`s belong to one solver island (one matrix), and current can flow through the bridge in either direction like any other piece of metal. The vanilla network objects stay distinct — only the solver's grouping spans them, the way two departments might keep separate org charts while sharing one budget. One standing caveat (api.md §23 item 9): wherever stationeers.md asserts this bridge, it must also carry the §7 derivation of vanilla's directional metrics (throughput, trip, analyzer readouts) from the signed bridge flow, plus the back-feed caveat — because real bidirectional conduction is a flagged divergence from vanilla behavior.
- **Project-coined term**, closest standard concept: a closed circuit breaker / bus-tie providing galvanic (conductive) coupling between two bus sections.
- **Where it appears:** `docs/stationeers.md` (Islands and Coupling Devices); `docs/api.md` §7 `CouplerSpec.Breaker()` and §23 open item 9.

### golden test / lesson corpus as goldens

The phrase names the design decision, not just the mechanism. In systems terms it is the same move as treating your user-facing documentation examples as executable tests (doctests): one artifact, three consumers — tutorial, documentation, regression gate. The corpus lives in `lessons/`, one directory per lesson; the Falstad importer that parses it is therefore core infrastructure written first, not tablet polish.
**Project-coined phrase**, closest standard concepts: golden/approval testing plus executable documentation (doctest-style).

### GPLv2 / MIT / LGPL
The three open-source licenses whose interaction constrains what code and test data Manatee may copy, link, or ship.

A software license is the legal envelope around code, and these three differ in how "sticky" they are. **MIT** is permissive: anyone may use, modify, and redistribute with attribution and no further obligations — Re-Volt is MIT, so manatee-core is MIT to be embeddable in it. **GPLv2** is copyleft: redistributing GPL code (or works derived from it) obliges you to release your whole work under GPL, which would be incompatible with shipping MIT. **LGPL** is the "lesser" middle ground, tolerable as a separate library but still avoided in shipped artifacts here. For an EE, think of it as datasheet-style usage restrictions: MIT is a public-domain-like reference design, GPLv2 is a design you may study but whose parts you cannot solder into your own product without adopting its terms, LGPL is a part you may socket but not embed. Concretely (falstad-format.md): circuitjs1 is GPLv2, so Manatee *reads its source to learn the file format* (facts are not copyrightable) but never vendors its files — its example circuits are used only as fetched-at-test-time fixtures, kept out of the shipped corpus, and the lesson corpus is hand-written. CSparse.NET is LGPL and is confined to a dev/test-only role (equivalence oracle), shipping in no mod, so no LGPL artifact reaches players (design.md, Repository Layout and Licensing).
**Standard terms:** these are the standard SPDX-identified licenses GPL-2.0, MIT, LGPL; "copyleft" vs "permissive" is the standard distinction.
- **Where it appears:** falstad-format.md §1 (circuitjs1 is GPLv2), §5 note 7 and §7 (fetched-not-vendored fixture policy); design.md "Repository Layout and Licensing"; CLAUDE.md working agreements.

### Graceful-degradation ladder
- The ordered chain of fallbacks in the Stationeers integration — manatee → Re-Volt's scalar distribution → vanilla `PowerTick` — where each level catches the failure of the level above so the game never stalls on the solver.

If a manatee island faults during a tick (a singular matrix or a non-convergent solve), that network falls back for the tick to Re-Volt's existing scalar ratio distribution and logs a diagnostic; if the Harmony code-injection into the game fails entirely, Re-Volt's prefix pattern already falls back to the game's vanilla power logic. For an EE, this is like a protection scheme with backup relays: the primary protection is precise, but if it fails to operate, a coarser backup still clears the fault rather than letting the system go dark. The invariant it enforces is availability-over-fidelity: a wrong-but-plausible power result for one tick beats a frozen game. Each rung is strictly less accurate but strictly more robust than the one above.

**Project-coined term**, closest standard concept: graceful degradation / fault-tolerant fallback chain (in reliability engineering, "defense in depth").

**Where it appears:** `docs/stationeers.md`, "Failure and Fallback".

### ground-return coda
- The closing segment of tablet lesson 14 ("Line loss and why we transmit high-voltage"), which asks: *"copper is expensive — when can you get away with one wire?"*
- After the main lesson establishes why long low-voltage runs waste power in the wire, the coda examines single-wire earth return (SWER): using the earth itself as the return conductor, paying tens of ohms of electrode contact resistance at each end instead of the price of a second copper wire. The lesson's predict-then-observe core is the electrode-loss arithmetic, which shows SWER works at 240 V distribution and for milliamp signalling (telegraph, doorbell, electric fence — historically accurate SWER territory) but fails for 12 V loads, where electrode resistance starves the load. This framing is deliberate: an earlier design made SWER the default wiring idiom, then reversed course when the 12 V arithmetic failed (design.md Grounding model, revised 2026-07-05) — so the curriculum teaches SWER as a cost-driven engineering choice, not as "how wiring works." For programmers: think of it as the lesson's epilogue that turns a failed default into a taught trade-off, like a postmortem section explaining when a rejected optimization is actually valid.
- **Project-coined term**, closest standard concept: single-wire earth return (SWER) as a curriculum topic. "Coda" is just the musical sense — a tail section.
- **Where it appears:** `docs/curriculum.md` Lesson Arc III, lesson 14; backed by `docs/design.md` § Grounding model.

### grounding model / two-wire-default
- The Vintage Story design decision (settled and revised 2026-07-05) that wiring is two-wire by default — the cable pathfinder routes conductor *pairs*, one gesture laying supply and return — with single-wire earth return (SWER) available as an explicit, user-visible choice rather than the norm.
- There is no inherent grounding in the model: earth is an ordinary solver node reached through per-electrode contact resistance (tens of ohms, better when wet), and each device negative is tied to earth by an implicit high-resistance leak (~1 MΩ, auto-stamped by the `TwoWireLeak` wiring policy) that gives every circuit a near-invisible earth reference — benchmarked at machine-epsilon conditioning cost, never affecting observable behavior. Note this leak is *not* the solver's singularity guard: that job belongs to a separate, six-orders-of-magnitude-smaller mechanism, the gmin = 1e-12 S shunts stamped on every non-ground node (the standard SPICE gmin idea), which keep deliberately floating subgraphs numerically solvable even with no leak at all. For programmers: gmin is the tie-breaker that keeps the linear system well-posed; the 1 MΩ leak is a modeling choice — the "grounding fiction" that makes earth-referenced behavior (and earth faults) real by default. Consequences of the model: deliberately floating systems (isolation transformer, no earth reference) are buildable — the tablet's `ExplicitOnly` policy stamps no leak, leaving gmin only — electrode quality is a real teachable parameter with its own overload envelope (soil dries and glassifies, raising contact resistance), and SWER earns its keep exactly where the real arithmetic works — high-voltage distribution and milliamp signalling — not at 12 V. Stationeers has no such choice: single-terminal devices force implicit ground-return with a near-ideal return conductance per network.
- **Project-coined term** (the design-decision name); the ingredients are standard: single-wire earth return (SWER), earthing/grounding electrode resistance, and SPICE-style gmin shunts for matrix nonsingularity.
- **Where it appears:** `docs/design.md` § Grounding model (and its Open Questions history); `docs/api.md` §Netlist wiring policy (`WiringPolicy.TwoWireLeak(1e6)` vs `ExplicitOnly`); gmin policy in `docs/solver.md` Numerics; curriculum tie-in in `docs/curriculum.md` lesson 14 (ground-return coda).
- **References:** ngspice manual (gmin, the 1e-12 S default shunt conductance); Horowitz & Hill, *The Art of Electronics* (grounding and earth-return practice).

### Harness grid
The rectilinear coordinate grid the 2D schematic harness lays components on — relevant because circuits imported from the Falstad format must fit it, so diagonally-placed two-terminal elements are rejected at import.

Falstad/circuitjs schematics place components on an 8- or 16-pixel grid but allow a two-terminal part (a resistor, say) to run diagonally between any two grid points. Our editor's geometry may require every element to be axis-aligned (horizontal or vertical), the way most professional schematic-capture tools do. The importer's contract therefore lists "diagonal 2-terminal elements, if the harness grid needs rectilinear" among the inputs it must **reject loudly** — with line number, offending token, and a one-line reason — rather than silently distorting the drawing. Think of it as input validation at a system boundary: anything the internal representation cannot faithfully hold is refused up front, not coerced.

**Project-coined term**, closest standard concept: the snap grid / rectilinear (Manhattan) routing constraint of a schematic capture editor.

**Where it appears:** `docs/falstad-format.md` §7 (accept/reject contract); the grid itself belongs to `docs/harness.md`'s document model.

### header line / options header
- The first-ish line of a Falstad circuit file, starting with `$`, that carries the simulator's global options rather than describing any circuit element.

Its format is `$ flags maxTimeStep simSpeed currentSpeed voltageRange [powerBarValue [minTimeStep]]` (e.g. `$ 1 0.000005 10.20027730826997 50 5 43 5e-11`). Almost all of it is UI configuration for the original Falstad applet — the `flags` bitfield toggles display options like current dots and grid size (with two inverted-sense bits: 4 = do *not* show voltage color, 16 = do *not* show values), and `simSpeed`, `currentSpeed`, `voltageRange`, `powerBarValue` are display pacing and color-scale settings. The only field with electrical meaning is `maxTimeStep`, the transient timestep in seconds — the analog of a sampling interval, and the one value our importer actually cares about; the trailing two fields are optional and older files omit them. Think of it as the file-format equivalent of an HTTP header: metadata about how to present and pace the simulation, separate from the payload (the element lines) that defines the circuit itself.
- **Project-adopted term** from the Falstad/circuitjs1 source, where it is written by `CirSim.dumpOptions()` and read by `CircuitLoader.readOptions`; there is no standard EE name for it.
- **Where it appears:** `docs/falstad-format.md` §2 (Header line), with the flag bitfield decoded; §1 lists `$` among non-element line types.
- **References:** Falstad/circuitjs1 source (`CirSim.java`, `CircuitLoader.java`), the authority the spec in `docs/falstad-format.md` was derived from.

### Heat registry
- The mod-owned data structure that tracks how much heat is being produced at each position in the Vintage Story world, so that electrical devices' waste heat can actually warm (or fail to cool) their surroundings.

Vintage Story's engine has no room temperature state at all: temperature is recomputed on demand from world climate, and no vanilla block ever warms a room (the engine's `IHeatSource` only warms *player bodies*). Since Manatee's design says "heat is universal" — every device that dissipates electrical power must dump that heat somewhere it matters — the mod defines its own convention: a position-indexed registry of heater strength per room, maintained by the block behaviors of heat-dissipating devices, each publishing its current dissipation (the I²R losses the solver already computes). Consumers then read the registry and fold it into their own math: freezers hook the vanilla perish-rate delegate and scale food spoilage by electrical duty, space heaters and other mod systems query it for ambient effects, and heaters may optionally also implement `IHeatSource` for firepit-style player warmth. To an EE, think of it as the thermal side of a power budget, book-kept in software because the game world has no physics for it; to a programmer, it is a shared spatial map (position → watts of heating) written by producers and polled by consumers.

**Project-coined term**, closest standard concept: a mod-defined spatial heat map / thermal accounting layer (no engine or EE standard term exists — VS vanilla rooms have no temperature).

**Where it appears:** `docs/vintage-story.md` §Rooms and Heat (the engine findings with file:line evidence and the plan); `docs/design.md` §"Heat is universal".

### Heat registry (position-indexed)
- The qualifier "position-indexed" names the registry's key: heat contributions are stored by world position (grouped per room), not per device type or per circuit.

This is the same structure as the **heat registry** — the phrase appears in full in `docs/design.md` when stating the convention. Indexing by position is what lets consumers ask a spatial question ("how warm is it *here*, in *this* room?") without knowing anything about the electrical network: a freezer's perish-rate hook or a comfort calculation looks up its own location, sums nearby registered heater strengths, and never touches the solver. Producers are the device block behaviors, which publish their dissipation (computed by the electrical solver) into the map at their block's position. Programmers can read it as a dictionary keyed by block position with room-scoped aggregation; an EE can read it as a lumped thermal source list with location metadata, standing in for the heat-transfer physics the game engine does not simulate.

**Project-coined term**, closest standard concept: spatial heat map keyed by world coordinates (no standard EE or engine term; VS vanilla has no room temperature state).

**Where it appears:** `docs/design.md` §"Heat is universal"; `docs/vintage-story.md` §Rooms and Heat.

### Hint (`h`)
- A non-element line type in the Falstad circuit text format, `h type item1 item2`, that carries a UI presentation hint rather than any electrical content.

Falstad/circuitjs1 export files are line-oriented: most lines describe circuit elements (resistors, sources, wires), but a handful of line types configure the simulator's *display* instead — scope panes (`o`), option headers (`$`), sliders (`38`), and hints (`h`). In Falstad's own loader, `h` lines are dispatched before element parsing. Because a hint has no electrical meaning, Manatee's importer neither stamps anything into the netlist nor rejects the file: the contract in `docs/falstad-format.md` is "ignore with a logged notice" — the line is dropped, but counted and reported so a lesson author sees exactly what was discarded. Hard-rejecting such lines would make the importer useless in practice, since presentation lines appear in virtually every real Falstad export. Think of it like a comment or editor metadata in a source file: the compiler skips it, but a good tool still tells you it did.

This is standard Falstad-format vocabulary (the `h` line type from circuitjs1's `CircuitLoader`), not a Manatee coinage; the ignore-with-notice policy is ours.

**Where it appears:** `docs/falstad-format.md` §1 (non-element line types, with the circuitjs1 source reference) and §7 (importer accept/reject contract: "ignore with a logged notice").

### hybrid pair
A rejected design shortcut: fabricating a single (rating, melt) protection pair for a collapsed cable chain by taking the rating from one physical segment and the melt threshold from a *different* one.

When compaction collapses a chain of cable segments into one equivalent resistor, the solver still needs to know when that chain would overheat. Each real segment has an i²t pair — a current rating and a melt (thermal) threshold that belong together because they describe one piece of material. A hybrid pair mixes segment X's rating with segment Y's melt into a fictitious "worst of both" element. It was ruled out (2026-07-06) because it is over-conservative in a way that produces false trips: a current profile can violate the hybrid while violating *no actual segment*, so the compacted circuit would melt a cable the raw, uncompacted circuit would not — breaking the invariant that compaction is behavior-preserving. The adopted alternative keeps the Pareto-minimal *set* of real (rating, melt) pairs; since all segments in a series chain carry the identical current, accumulating each pair independently reproduces the raw circuit's trip decisions exactly. Programming analogy: it is like merging two structs field-by-field into a chimera that satisfies neither original's invariants — the composite passes checks no real instance corresponds to (or here, fails checks every real instance would pass).

**Project-coined term**, closest standard concepts: a composite i²t (let-through energy) damage curve / time-current characteristic; the flaw is taking a pointwise-worst-case envelope over distinct devices' curves.

**Where it appears:** `docs/compaction.md` (Responsibilities #4, Limit attribution), `docs/api.md` §19.

### implicit ground-return (near-ideal return conductance)
- The Stationeers/Re-Volt wiring mode: every device's return terminal is tied to its network's reference node through a very large built-in conductance, giving the numerical effect of a return wire without ever simulating one.

Stationeers has no cable voxels and its vanilla devices expose a single power terminal, so a supply-vs-return wiring choice cannot exist there. Instead of doubling the matrix by modeling a return conductor per cable run, `WiringPolicy.ReferenceBound` (default 1e3 S, i.e. ~1 mΩ) binds each return terminal to the partition's reference node. Because the conductance is near-ideal, the solution is numerically identical to modeling an explicit near-zero-resistance return wire — but the "full double-wire solve is never paid." In programming terms it is interning: many logically distinct return paths collapse onto one shared node, trading representational detail you'd never observe for half the problem size. The fiction still describes an ordinary two-wire cable bundle; no earth leak is stamped in this mode (contrast the VS `TwoWireLeak` policy).

**Project-coined term**, closest standard concept: a single-node ideal return / common reference rail — the standard SPICE idiom of wiring all returns to node 0, here made an explicit per-network policy with a finite (near-ideal) conductance instead of a hard merge. Not to be confused with SWER (single-wire earth return), which in this project is the *user-visible* VS technique where earth itself carries current through electrode resistance.

**Where it appears:** `docs/design.md` Grounding model; `docs/api.md` §5 `WiringPolicy.ReferenceBound`; `docs/stationeers.md` (Re-Volt integration).

**References:** Ho, Ruehli & Brennan (1975), "The Modified Nodal Approach to Network Analysis," *IEEE Trans. Circuits and Systems* (the reference-node convention this collapses onto).

### implicit high-resistance leak (~1 MΩ)
- An automatically added large resistor from each device's return terminal to the earth node, present only so the solver's matrix has a defined voltage reference — carrying negligible current itself.

Vintage Story wiring is two-wire: both supply and return are real routed conductors, so a correctly built circuit forms a closed loop that touches earth nowhere — it *floats*. A floating circuit has no absolute voltage reference, which makes the MNA matrix singular: infinitely many solutions differ by a constant offset, the linear-algebra analogue of an uninitialized variable that happens to work until it doesn't. The fix (`WiringPolicy.TwoWireLeak`, default 1e6 Ω): at commit, the core auto-stamps a ~1 MΩ resistor from every Return-role node to the island's reference (the literal earth node). At megohm scale the leak current is microamps — gameplay-invisible — but the matrix is pinned. It is never hand-authored and has no client handle. Benchmarked at machine-epsilon conditioning cost, with a 1.5× hot-path / 2.3× refactor multiplier on realistic post-compaction circuits, absorbed by the budget. It also mirrors reality: real installations leak slightly to earth through insulation, which is why "verified floating" is a taught ritual rather than an assumption.

**Standard term:** GMIN / node-to-ground shunt conductance — SPICE's `GMIN` serves the same keep-it-nonsingular role, though SPICE applies a much smaller conductance (default 1e-12 S) to device junctions, while Manatee also uses gmin shunts internally and reserves the deliberately larger, physically-motivated 1 MΩ leak for Return-role nodes.

**Where it appears:** `docs/design.md` Grounding model; `docs/api.md` §5 `WiringPolicy.TwoWireLeak` and the VS walkthrough; `docs/solver.md` Numerics (gmin shunts).

**References:** Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520 (GMIN); the ngspice manual (`GMIN` option).

### Incremental equivalence
- The invariant — and the test that enforces it — that a circuit built up through many small edits must end in exactly the same state as one built from scratch.

Concretely: apply any sequence of tier-3 (topology) edits incrementally, then separately rebuild the final circuit from nothing; the two must produce identical islands and identical solutions. The test suite runs this over randomized, seeded edit sequences. For an EE, this is a path-independence check: the final state must depend only on the final circuit, never on the editing history — any divergence means the incremental bookkeeping has drifted. The same comparison ships *in the game* as the background resync backstop required by R11: incrementally-maintained topology is periodically diffed against a from-scratch rebuild, and drift is detected and corrected. So one mechanism does double duty as a CI test and a runtime reconciliation path — the SRE instinct that incremental state always needs a reconciliation loop.

**Project-coined name** for a standard testing idea, closest standard concepts: metamorphic/differential testing; state reconciliation (anti-entropy) in distributed systems.
- **Where it appears:** `docs/testing-strategy.md` (Equivalence Tests), `docs/design.md` R11.

### Insulation
- In Manatee's Vintage Story design, insulation is a physically distinct voxel material (the in-game fiction is tree resin) wrapped around conductor voxels, not a property flag on the wire.
- Because it is its own chisel material, it participates in the microblock system independently of the conductor it covers: it chisels away separately, and overload damage destroys it separately (sustained low-level overload visibly erodes surface voxels — degradation is the warning). A conductor with missing insulation is *bare*: bare conductors short against each other, ground when wet, and shock entities relative to global ground (design.md R12). This turns insulation from a data attribute into emergent physics: the state of the wire's covering is literally the geometry present at that position, the same way a fuse is just a lead segment in a copper run rather than a special-cased component. Salvage is lossy — overload-destroyed insulation is simply gone, with no strip-and-resmelt loop.
- The EE concept is entirely standard (dielectric insulation on conductors); what is project-specific is representing it as removable voxel geometry with its own material identity.
- **Where it appears:** `docs/design.md` R12 and Hazards; `docs/vintage-story.md` sec 1 (chisel semantics: insulation chisels separately).

### Integration seam
- The specific, named points where manatee-core plugs into a host game's existing update cycle — for Stationeers/Re-Volt, the game's per-network power-tick calls plus the one solve point.
- Re-Volt already replaces Stationeers' vanilla power tick with a three-phase contract per cable network (`Initialise` → `CalculateState` → `ApplyState`, via `RevoltTick`). Manatee attaches at exactly four places: `Initialize_New` (topology changed → rebuild/patch the island graph), `CalculateState_New` (devices state their wants → update stamps), the scalar `_powerRatio` computation point (`RevoltTick.cs:285` → replaced by the MNA solve), and `ApplyState_New` (write solved actuals back through the vanilla device calls). Confining integration to these seams is what makes the removability requirement possible: pull Manatee out and vanilla mechanics resume intact, because all device state keeps flowing through the vanilla calls. An EE analogy: seams are the terminal blocks where new equipment lands on an existing panel — everything else stays untouched wiring.
- **Standard term:** "seam" in this sense comes from software engineering — Michael Feathers' *Working Effectively with Legacy Code* defines a seam as a place where behavior can be altered without editing the surrounding code.
- **Where it appears:** `docs/stationeers.md` Integration Seams; echoed in `docs/integration-tutorial.md`.
- **References:** Feathers, *Working Effectively with Legacy Code* (Prentice Hall, 2004).

### island
- A maximal group of components that are electrically connected to each other and to nothing else; Manatee solves each island completely independently.

In Manatee an island is the fundamental unit of *everything*: one island = one connected component = one matrix = one LU factorization = one unit of parallel scheduling, failure isolation, snapshot/restore, and handle invalidation. Rebuilding or faulting one island never disturbs another, and islands solve in parallel on a client-supplied worker pool. For programmers: it is a connected component in the graph-theory sense, managed by a union-find — merges are incremental, removals trigger a rebuild of the affected island (no incremental split detection, per design.md R11). For EEs: it is an electrically isolated subnetwork, each anchoring its own ground reference. Coupling devices (decoupling transformers, converter two-ports, breakers) are the only bridges between islands: a galvanic bridge (closed breaker) merges islands into one matrix, while a power-transfer boundary keeps them as separate matrices exchanging values.
- **Standard term:** connected component (graph theory) / electrically isolated subnetwork. SPICE-family simulators typically treat the whole netlist as one system; per-component partitioning as a first-class scheduling and failure unit is our design choice.
- **Where it appears:** design.md R5 and Simulation Model; solver.md "Islands" (the canonical section); compaction.md Responsibilities #5; stationeers.md "Islands and Coupling Devices"; integration tutorial §1, §5; API surface throughout (per-island Circuit, Faulted state, snapshot).
- **References:** Cormen, Leiserson, Rivest & Stein, *Introduction to Algorithms* (connected components, union-find); Ho, Ruehli & Brennan (1975), "The Modified Nodal Approach to Network Analysis", IEEE Trans. Circuits and Systems (the per-island matrix formulation).

### island bookkeeping
- The compaction layer's job of tracking which nodes and components belong to which island as the circuit is edited.

Concretely it is a connectivity union-find (disjoint-set data structure) maintained over the netlist, feeding the solver's island management. Adding a component that bridges two islands performs an incremental merge — cheap, just uniting two sets. Removing a component might split an island, but detecting splits incrementally is not worth the complexity at our circuit sizes, so any removal instead invalidates and rebuilds the whole affected island; removals in a tick are coalesced so the island rebuilds at most once per tick. For EEs: this is the analogue of noticing, as you clip wires in and out of a breadboard, which parts of the bench are still one circuit and which have become separate ones — done by machine, incrementally, on every edit.
- **Project-coined term**, closest standard concept: dynamic connectivity maintenance via union-find (disjoint-set forest).
- **Where it appears:** compaction.md Responsibilities #5 and Incremental Maintenance; solver.md "Islands" (the consuming side).
- **References:** Cormen, Leiserson, Rivest & Stein, *Introduction to Algorithms* (disjoint-set data structures).

### island boundary
- The interface where two independently-solved islands meet at a power-transfer coupling device, exchanging summary values instead of sharing a matrix.

At an island boundary the two sides never appear in the same system of equations. Instead, each AC substep the boundary exchanges amplitude+phase of the transferred waveform, damped by explicit relaxation (exponential smoothing with a per-device-type α) so the exchange settles rather than rings. This keeps each island linear in its own domain — a rectifier boundary lets the AC side and DC side each stay a cheap linear solve. For programmers: it is like two services communicating through a rate-limited message channel rather than sharing memory — one substep of lag, bounded and explicit. Conservation is enforced, not assumed: each boundary carries a running energy ledger (∫P_out − ∫P_in − modeled losses); the coupling device clamps transfer to what the source island actually delivered, and any accounting surplus is dumped as device heat, never stored work. Islands joined by boundaries are scheduled as one work unit and substep in lockstep, so the exchange never crosses free-running threads.
- **Project-coined term**, closest standard concepts: waveform relaxation / co-simulation coupling interface between subcircuit solvers.
- **Where it appears:** design.md Simulation Model ("Island coupling (transformers)", energy accounting rule); solver.md "Islands" (exchange scheme, scheduling and conservation); testing-strategy.md coupling-conservation and boundary-stability property tests.

### island coupling / decoupling transformer / converter two-port
- The family of devices that link separate islands across an island boundary: decoupling transformers on the AC side, converter two-ports (rectifier-flavored devices, chargers) between AC and DC or DC and DC.

These devices join islands *logically* while keeping them separate matrices. Per substep they exchange amplitude+phase with explicit relaxation damping (solver.md, Islands); a converter two-port additionally applies a behavioral efficiency curve, enforces its own limits, and — on the DC side — gets its smoothing storage by construction from an honestly-sized DC-link capacitor, whose job in a real converter is exactly that. No semiconductor-level simulation happens in world circuits; the device is a behavioral power-transfer model whose whole purpose is to keep both coupled islands linear. Because the coupling scheme could in principle oscillate or manufacture energy, both properties are property-tested (testing-strategy.md): randomized load steps across a boundary must settle without unbounded ringing, and the boundary energy ledger must stay bounded and non-positive-surplus — no free energy at boundaries. For programmers: think of an anti-corruption layer between two subsystems, with an invariant checker auditing every transaction.
- **Project-coined terms**, closest standard concepts: two-port network / behavioral macromodel; the boundary exchange is a relaxation-based co-simulation coupling.
- **Where it appears:** design.md Simulation Model ("Island coupling (transformers)"); solver.md "Islands" and the devices-layer transformer/converter entries; testing-strategy.md Property and Fuzz Tests (coupling conservation, boundary stability).
- **References:** Vlach & Singhal, *Computer Methods for Circuit Analysis and Design* (two-port formulations).

### island rebuild
Reconstructing one island (an independent connected sub-circuit) from scratch — its node numbering, matrix structure, and factorization — instead of patching the existing data structures incrementally.

In Manatee's change-cost tiers this is tier 3, the most expensive kind of edit: additions and merges are handled incrementally, but *any removal* invalidates the affected island and triggers a rebuild, because detecting whether a removal split the island into pieces is not worth the complexity at our island sizes (design.md R11). Rebuilds are **coalesced**: a removal only marks the island dirty, and the actual rebuild runs at most once per tick, at solve time — so a player bulldozing forty cable segments in one tick pays for one rebuild, not forty. For an EE, think of it as re-deriving the netlist and redoing the symbolic + numeric LU factorization for that sub-circuit; for a programmer, it is a cache-invalidation strategy — throw away the derived state and regenerate it from the source-of-truth geometry, which doubles as the resync backstop against incremental-maintenance drift. A rebuild invalidates that island's component and node handles (reissued at the next Solve) and emits a `Rebuilt` island change.

**Where it appears:** `docs/solver.md` (change-cost tier 3, Islands), `docs/design.md` R4/R11, `docs/compaction.md` (Responsibilities #5, Incremental Maintenance); surfaced in the API as `TickStats.IslandRebuilds` and `IslandChangeKind.Rebuilt` (`docs/api.md`).

**References:** the incremental-vs-rebuild trade-off around matrix structure follows standard sparse-solver practice, e.g. Vlach & Singhal, *Computer Methods for Circuit Analysis and Design*; Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods*.

### IslandChange / DrainChanges
The typed post-solve event feed telling the game client which islands changed identity this tick, and the method that empties it into a caller-supplied buffer.

An `IslandChange` is a small struct — a kind plus up to two `IslandId`s — with six kinds: `Created`, `Merged` (A = survivor, B = absorbed), `Rebuilt` (A = old, B = new), `Removed`, `Faulted`, `Recovered`. `IslandTable.DrainChanges(span, out lost)` copies pending changes into the caller's span and clears them; a too-small span drains partially, so callers loop until it returns 0. This feed is *the* game client's re-pin trigger: after draining, the client re-resolves any handles the changes invalidated (by external key) — never by replaying the journal. The events live in a fixed-size ring allocated at shape time (no per-tick allocation); `lost == true` means a burst or a skipped drain overflowed the ring, and the client is then **obligated** to a full re-pin of every held device handle — an explicit-overflow discipline, never a silent drop. In hardware terms it is exactly a status/interrupt FIFO with an overflow flag: cheap level-triggered notification, with a defined recovery protocol when the FIFO overruns. The stream is complete by contract: every `IslandId` that dies is reported, even when the matrix work was coalesced.

**Project-coined term**, closest standard concepts: change notification / event queue with drain semantics; the overflow-forces-full-resync rule mirrors "generation counter overflow ⇒ full resynchronization" protocols in replication systems.

**Where it appears:** `docs/api.md` §11 (definition), §18 (re-pin obligation), §20 (validity table), §24.20; `docs/integration-tutorial.md` §5, §7, sharp-edges appendix items 2 and 16.

### IslandId
The generational identity handle of one island — one independently-solved connected sub-circuit.

It is a 12-byte value (`Slot`, `Gen`, `Net`): a slot index plus a generation counter, so a recycled slot is detectable as stale rather than silently pointing at a different island — the same trick as a version-stamped pointer or a weak reference. Its key property is what a merge does *not* do: when two islands merge (say a breaker closes), component and node handles all survive; the **only** thing renumbered is the absorbed side's `IslandId`, which becomes stale while the survivor's stays valid. Rebuilds and resyncs, by contrast, issue a fresh `IslandId` (old one invalid, reported as `Rebuilt`). Because `IslandId`s are volatile across rebuilds, deterministic iteration (`IslandTable.Ids`) is ordered by minimum node external key, not by id. For an EE: it names "this galvanically-connected network" in a world where players constantly splice and cut networks, so the name must carry an edition number.

**Where it appears:** `docs/api.md` (declared near §5's id types; lifecycle table §16/§20), `docs/integration-tutorial.md` §6 (handle survival rules).

### IslandRebuilds
A per-tick counter in `TickStats` recording how many tier-3 structural island rebuilds the solve performed.

It is a performance tripwire, not just telemetry: the steady-state contract says quiet ticks cost right-hand-side solves only, so the standing CI assertion (and the tutorial's example) is `Refactorizations == 0 && IslandRebuilds == 0` when nothing structural happened. A nonzero value means a coupler (breaker/transformer) opened or a cut/removal fired — legitimate during construction or switching, a bug if it shows up every tick. Programmers can read it as a cache-miss counter for the most expensive cache in the system (the island's symbolic + numeric factorization); an EE can read it as "how many sub-circuits had their netlist re-derived and re-factorized this tick."

**Where it appears:** `docs/api.md` §"TickStats" (field with the `steady state ⇒ 0` comment, plus the CI-assertion guidance), `docs/integration-tutorial.md` §9 (the performance contract).

### IslandStatus
- The enum that tells a game client what condition a circuit island is in — most importantly, whether its published values can be trusted right now.

Every island (an electrically connected group of components solved as one unit) carries a `Status` field on its `IslandHandle`. The values are `Empty`, `Building`, `Ready`, `Dirty`, and `Faulted` (see the state-machine entry below for the full transition rules). Clients check it before acting on readback: the canonical tick-loop pattern is `if (isl.Status == IslandStatus.Faulted) { run scalar fallback, log isl.Fault; continue; }` — that is, when the solver could not solve a circuit, the game falls back to a simple non-matrix power model rather than crashing or displaying garbage. Think of it like an HTTP status code on a cached response: you look at it before you use the payload. For an EE: it is the front-panel lamp on each sub-circuit's solver saying "converged", "stale", or "no solution exists".

**Where it appears:** `docs/api.md` §11 (enum definition and state machine), §17.4 and §20 (read-path enforcement); `docs/integration-tutorial.md` §8 (the fallback pattern).

### IslandStatus (Empty/Building/Ready/Dirty/Faulted)
- The client-visible state machine each island moves through, defining exactly what a read returns in every state.

The transitions are a contract, not an implementation detail: an edit commit takes an island from `Empty` to `Building` (background construction; reads return the last-good values; `Step` is a no-op) or straight to `Dirty`. `Dirty` means something changed and a solve is needed; a successful solve flips it to `Ready`; any change that touches the island (Drive, Adjust, Reconfigure, Edit) flips it back to `Dirty`. If the solve fails — the matrix stays singular after gmin remedies, or Newton iteration exhausts its ladder — the island becomes `Faulted`: reads are de-energized (all voltages and currents zero, enforced at the read path), and any tier-2/3 change flips it back to `Dirty` for an automatic retry. In EE terms, `Faulted` marks a legal-but-degenerate circuit a player built (e.g. contradictory voltage sources) — the simulator refuses to publish nonsense but the game keeps running and neighboring islands keep solving. In programming terms, this is a classic explicit finite state machine with documented invariants per state, and `IsLive` is derived from it (`false` for Building, Dirty-with-stale-values, and Faulted).

**Project-coined term**, closest standard concept: a per-partition solver lifecycle / dirty-flag state machine; SPICE-family simulators have internal analogues (needs-reanalysis, convergence failure) but do not expose them as a public per-subcircuit enum.

- **Where it appears:** `docs/api.md` §11 (the commented state machine next to `enum IslandStatus`), §17.4 and §20 (Faulted read gating), `docs/integration-tutorial.md` §8.

### IslandTable / IslandHandle
- The API surface for working with individual islands: `IslandTable` is the directory of all islands, `IslandHandle` is the remote control for one of them.

`IslandTable` answers scheduling questions across the whole netlist: how many islands exist, which island a given node belongs to (`Of(node)`), which islands need solving this tick (`CollectDirty`, which returns one handle per *scheduling unit*, deduplicating boundary-coupled islands to their lead), and what membership changes happened since last tick (`DrainChanges` — the game's trigger to re-resolve any device handles it holds). Its `Ids` span defines the deterministic, rebuild-stable iteration order used by replay and golden tests. `IslandHandle` then operates on one island: `Step` solves it (callable from any thread, single-writer-per-island debug-asserted), plus `Status`, the substep `Plan`, `Fault` diagnostics, snapshot/restore, and invariant checks (`CheckInvariants`, `Energy`). For a programmer: table = collection, handle = element accessor. For an EE: the table is the one-line diagram listing every electrically isolated sub-network; a handle is the test bench hooked up to one of them.

**Project-coined term**, closest standard concepts: connected-component partitioning of the circuit graph, exposed through a handle/table API (the handle-table idiom is standard systems programming).

- **Where it appears:** `docs/api.md` §11 (full definitions), §15 (DrainChanges vs. journal), §21 (threading/validity rules); used throughout `docs/integration-tutorial.md`.

### IsLive
- A per-island boolean on a `Solution` saying whether the values you are about to read were freshly computed, as opposed to stale, last-good, or forced to zero.

`Solution.IsLive(islandId)` returns `false` when the island is `Building` or `Dirty` (reads return the last successfully published values — correct as of some earlier tick, like a cache serving stale-while-revalidate) or `Faulted` (reads return de-energized zeros, because the solver has no valid answer at all). Game code that takes meaningful action on electrical values — billing energy, tripping breakers, showing a meter as authoritative — should gate on `IsLive`; ignoring it is listed as sharp edge #3 in the tutorial appendix. Fault output itself is never published, so a non-live read is never garbage: it is either an old good answer or an explicit zero. For an EE: it is the "reading valid" annunciator next to the meter — the needle may show something, but this flag says whether it reflects the circuit as it stands right now.

**Project-coined term**, closest standard concept: a data-freshness / cache-validity flag on a published snapshot.

- **Where it appears:** `docs/api.md` §10 (`Solution.IsLive`), §11 and §17.4 (which statuses read non-live), §22.a; `docs/integration-tutorial.md` §6, §8, and sharp-edges appendix item 3.

### journal
- Manatee's internal, ordered stream of topology events (`TopologyJournal`) that feeds the reduction (compaction) layer — explicitly *not* the channel game clients listen to.

Every structural change the solver commits — island merges, splits, rebuilds, component removals — is appended to a fixed-capacity ring buffer of small fixed-size records, each with a monotonically increasing sequence number. The reduction layer replays this stream to keep its collapsed view of the circuit incrementally up to date, with an explicit-overflow discipline: if the ring wraps before a reader catches up, the reader is told it lost events and must resynchronize from scratch rather than proceed on stale data. Game clients never read the journal; their one notification channel is `DrainChanges`, which carries the client-relevant subset with the same overflow discipline. For EEs: think of it as the solver's flight recorder — a strictly ordered log of "what changed in the wiring" that a downstream bookkeeping subsystem replays, analogous to a lab notebook that another engineer reconstructs your rework from.

**Project-coined term**, closest standard concepts: event log / write-ahead log, ring buffer (bounded circular queue) with lost-data signaling.

**Where it appears:** `docs/api.md` §15 (TopologyJournal) and §17; `docs/integration-tutorial.md` §5 ("never the journal — that serves the reduction layer"); `docs/compaction.md` (incremental maintenance).

**References:** the log/replay pattern is standard systems fare — see e.g. Gray & Reuter, *Transaction Processing: Concepts and Techniques* (1993) on write-ahead logging.

### key-allocation contract / Derive(ordinal)
- The binding rule for how a multi-part device names its internal parts: each sub-component's identity is minted from the device's own identity plus a fixed, documented slot number.

A game device (say, a battery) is one object to the client but several primitive components inside the solver (an EMF source, an internal resistance, ...). Each primitive needs its own stable `ExternalKey` — the 128-bit identity that survives island rebuilds and drives restore-by-key, drift diffs, and serialization. The contract (declared binding in `Device.Build`) says: the device mints each component key as `baseKey.Derive(ordinal)`, where `baseKey` is the device's client-assigned key and the ordinal is a stable, documented constant per component *role* of that device class — e.g. Battery: 0 = EMF source, 1 = Rint. Because the derivation is a pure function of (base key, role), rebuilding the device from scratch reproduces the exact same keys, so saved state finds its components again and a drift diff compares like with like. Programming analogy: deterministic namespaced IDs (like UUIDv5 or a path `battery-17/0`) rather than auto-increment counters, whose values would depend on construction order. For an EE: it is the schematic reference-designator scheme (BT1's internals are always BT1.V and BT1.R) enforced by the type system instead of by drawing convention.

**Project-coined term**, closest standard concepts: deterministic/namespaced identifier derivation (cf. name-based UUIDs, RFC 4122 §4.3); hierarchical instance naming in SPICE subcircuits (`X1.R1`).
- **Where it appears:** `docs/api.md` §3 (`ExternalKey.Derive(ushort ordinal)`) and §18 (`Device.Build`, where the contract is stated as binding).

### Last-good read
- The API rule that when an island cannot currently publish a fresh solution, reads return the last values it *successfully* published — never a half-built matrix's garbage and never a fabricated zero.

Every island's solution is only visible to callers after a solve *publishes* it; between a topology change and the next solve (island status `Building` or `Dirty`), `Solution.*` reads keep returning the previous published vector, flagged by `IsLive == false` so callers know it is a hold, not fresh physics. Think of it as a cache that serves the last known-good value while the backend recomputes, with a staleness bit. Two carefully-adjudicated edges: (1) a **`Faulted`** island reads as de-energized 0 — but this is scoped to the status, enforced at the read path, not a taint on the data: the moment a repairing change or a merge flips it back to `Dirty`, its last successfully-published values read again (fault output itself can never leak, because failed solves never publish); (2) the **absorbed side of a merge** keeps its pre-merge last-good potentials — and its voltage sources their branch currents and powers — until the merged island first publishes (see "last-good read / merge window"). The rule exists because game-side adaptor code does arithmetic on these reads; a manufactured 0 V looks like a dead short to a `G = P/V²` power adaptor. An EE can read `IsLive == false` as "meter frozen at the last valid measurement while the circuit is being rewired."

**Project-coined term**, closest standard concepts: SPICE's reuse of the previous operating point, combined with the systems idea of stale-while-revalidate cache semantics.

**Where it appears:** `docs/api.md` §10 (Solution and readback, `IsLive`), §17.4 (what reads observe between mutation and Solve), §24 decision log #27; `docs/solver.md` (Failure Handling).

### Last-good read / merge window
- The guarantee that on the tick two islands are joined (e.g. a breaker closes), *both* sides keep reading their pre-merge published values until the combined island first publishes — the merge never manufactures a zero.

When a switch closes between two separately-solved islands, the solver merges them: one island survives, the other is *absorbed* and its nodes relabeled into the survivor. The "merge window" is the gap between that relabeling and the merged island's first successful solve. Through that window the survivor keeps its last-good published vector, and the absorbed side's nodes carry their pre-merge last-good potentials — including voltage-source branch currents and powers, captured per-component at merge commit — all readable via `Solution`/`ctx.Previous` with `IsLive == false`. This holds symmetrically on both union orientations, even when one side was previously `Faulted` (the merge flips the union to `Dirty`, so the faulted side's reads revert from de-energized 0 to its last *published* values — 0 only if it never published). The rule was hard-won: an earlier build let absorbed-side handles read 0.0 through the window, and the integration tutorial's `G = P/V²` power adaptor trusted that zero, stamped a fictional dead short, and blew a 6 A fuse with a phantom 597 A surge on the merge tick — fixed 2026-07-07. Programming analogy: it is a read-your-last-committed-write guarantee across a data migration; no reader ever observes the uninitialized intermediate state.

**Project-coined term**; there is no standard circuit-simulation equivalent because batch simulators do not expose mid-reconfiguration reads. Closest systems concept: atomic visibility across a schema migration.

**Where it appears:** `docs/integration-tutorial.md` §6 ("Merge-tick reads hold last-good on BOTH sides") and sharp-edges appendix edge 4; `docs/api.md` §17.4 and decision log #27.

### launders reactive power
- Project shorthand for what a decoupling-transformer island boundary does to phase information: real power gets across, but the reactive/phase relationships that power-factor lessons depend on do not survive the crossing faithfully.

Manatee has two transformer classes (design.md, Simulation Model): *idealized* transformers live inside one solver matrix and preserve full circuit behavior, while *decoupling* transformers are island boundaries that exchange only an amplitude+phase summary per substep, damped by explicit relaxation. That summary is enough for energy transfer but it is a lossy interface — like two services communicating through a narrow API that serializes only aggregate numbers: the detailed phase relationships between individual currents and voltages on opposite sides are reconstructed, not propagated, so they are "laundered" the way money loses its provenance through a shell company. The curriculum authoring rule that uses the phrase draws the consequence: any lesson demonstrating power factor or impedance behavior must keep the whole demonstration in one island, so the tablet uses idealized transformers throughout.

**Project-coined term**, closest standard concept: loss of phase coherence across a co-simulation / waveform-relaxation partition boundary — there is no single standard EE term; in real power systems a transformer does *not* hide reactive power, so this is purely an artifact of the island-decoupling simulation scheme.

**Where it appears:** `docs/curriculum.md`, Authoring Rules (transformer-lesson rule); underlying mechanism in `docs/design.md` ("Island coupling (transformers)") and `docs/solver.md` (Islands).

### lead-fuse-in-a-copper-run
- The project's canonical test case for emergent physics: place one lead segment in an otherwise-copper conductor run, and it *is* a fuse — with zero fuse-specific code anywhere.

A real fuse is nothing but a deliberately weak link: a low-melting-point metal that fails first under overcurrent. Because the compaction layer computes inter-region resistances from real material resistivity and contact cross-section (Responsibility #1), and because each collapsed conductor carries a per-limit-type envelope taking the minimum over its constituent segments (Responsibility #4 — instantaneous ampacity, i²t thermal accumulation, and melting threshold can each pick a *different* weakest segment), the lead segment automatically becomes the run's failure point. For programmers, this is the design payoff of modeling the domain instead of special-casing features: "fuse" falls out as an invariant of the material data, the way a well-factored type system makes illegal states unrepresentable rather than checking for them. The case motivates design requirement R12 (voxel cables with physical semantics) and is the acceptance test for the limit-attribution machinery, including the ruling that the i²t envelope is a Pareto-minimal *set* of (rating, melt) pairs rather than a single hybrid pair.

**Project-coined term** (a scenario name, not a device); closest standard concept: a fusible link — a sacrificial low-melting-point conductor section used as overcurrent protection.

**Where it appears:** `docs/compaction.md`, Responsibilities #1 and #4; `docs/design.md`, R12; envelope semantics in `docs/api.md` §19.

### Legacy-device adaptor

The compatibility layer that lets unconverted Stationeers devices participate in the real circuit simulation by wrapping the vanilla game's power API (requirement R18).

Stationeers devices speak a four-call joules-per-half-second protocol (available/needed, provide/draw) and know nothing about voltage or current. Rather than convert every device up front, the adaptor treats each unconverted device as a **constant-power element**: a load that always tries to draw its advertised wattage, or a generator that offers its advertised wattage at nominal voltage. Constant power is nonlinear (I = P/V), so the adaptor linearizes it *across game ticks* using the previous tick's voltage, with guard rails — a brownout clamp with hysteresis and lockout so overloaded nets settle or collapse legibly rather than strobing, an across-tick energy ledger so the linearization lag can't be pumped for free energy, and small series resistance on every adapted source so paralleled generators stay solvable. Post-solve it reports actual delivered energy back through the vanilla calls, so vanilla bookkeeping stays intact and removing the mod reverts cleanly. Systems analogy: it's an adapter/shim in the Design Patterns sense, plus an eventual-consistency loop. Devices are converted to native models one at a time; the two populations coexist indefinitely.

**Project-coined term**, closest standard concepts: the Adapter pattern (software) combined with a constant-power load linearized by successive substitution / a SPICE-style companion model (EE).

**Where it appears:** `docs/design.md` R18; `docs/stationeers.md`, "The Legacy-Device Adaptor".

**References:** Gamma, Helm, Johnson & Vlissides, *Design Patterns* (Adapter); constant-power load models are standard in power-flow literature (e.g. ZIP load models).

### Legacy-device adaptor / "adaptor" (shorthand)

In Stationeers-side docs, "the adaptor" unqualified always means the legacy-device adaptor — the mechanism that turns the vanilla 4-call power API into circuit stamps each tick and distributes real results back afterward.

Mechanically, each tick the adaptor: (1) stamps each legacy load as a conductance G = P_wanted / V_prev² — the companion-model pattern, where a nonlinear element is replaced by a linear stand-in built from the last known operating point, converging over a few ticks in normal operation; (2) stamps legacy generators as voltage sources at tier-nominal voltage with a current limit derived from their advertised power, enforced by an across-tick clamp rather than in-solve mode switching; (3) after the solve, calls the vanilla provide/draw with the *actuals* — what the circuit really delivered — so the game's own energy accounting never diverges from the simulation. For programmers: it's a bidirectional translation layer with one tick of staleness by design. For the EE: it's fixed-point iteration on a constant-power load, spread across game ticks instead of Newton iterations, with a brownout dropout below a voltage threshold — the voltage-collapse death spiral above that threshold is real physics and shipped as a feature.

**Where it appears:** `docs/stationeers.md`, "The Legacy-Device Adaptor" (mechanics); `docs/design.md` R18 (requirement and guard rails); the adaptor-stability property test in `docs/testing-strategy.md`.

**References:** companion models: Nagel (1975), *SPICE2: A Computer Program to Simulate Semiconductor Circuits*, UCB/ERL-M520; Pillage, Rohrer & Visweswariah, *Electronic Circuit and System Simulation Methods*.

### Legible failure

The design principle (requirement R9) that when a circuit is broken, degenerate, or unsolvable, the simulator must produce a diagnosable, explainable error — never silently corrupt results.

Players will build pathological circuits as a hobby: floating nodes, two ideal sources fighting each other, networks that refuse to converge. R9 demands that every such case yields a specific error or event a player can act on — "why doesn't this work" must be answerable in-game — and forbids the classic numerical failure mode where a NaN (Not-a-Number, the floating-point value that poisons every calculation it touches) propagates through the solution and everything quietly reads as garbage. For the EE: it's the discipline of a good instrument — it reads OVERLOAD, it never displays a plausible wrong number. For programmers: fail-fast with structured diagnostics instead of undefined behavior. The docs extend the same word to performance and overload behavior ("degrade legibly"): an overloaded net settles or collapses in an understandable way (brownout hysteresis, lockout) rather than strobing at tick rate, and event floods are rate-limited rather than melting the log.

**Project-coined term**, closest standard concepts: fail-fast design, graceful degradation, and SPICE-style convergence-failure diagnostics.

**Where it appears:** `docs/design.md` R9 and the Performance/pathological-builds table; `docs/api.md` "Legibility rails" and the Faulted state; `docs/testing-strategy.md` ("must converge or Fault legibly — never NaN, never hang").
